Restore the stock Alpine OpenRC networking service.

Problem:

- Exploration 026 allows progress by generating a Bunix-specific
  `/etc/init.d/networking` service in the Alpine rootfs.
- That is acceptable as a temporary bootstrap path, but the intended Alpine
  environment should run the real Alpine/ifupdown-ng networking service.
- The current generated service is fine for now because it keeps `make run`
  useful while networking is still being brought up, but it is explicitly a
  compatibility crutch to remove in a future slice.
- The current replacement service is a stopgap only.  It should not become the
  Bunix network-management interface or a permanent OpenRC compatibility
  contract.
- The stock service currently pulls in OpenRC dependencies such as
  `localmount` and helper programs like `mkmntdirs`, `mountinfo`, and
  `fstabinfo` that expose gaps in Bunix's VFS, mount-table, and Linux
  compatibility behavior.

Intended outcome:

- The generated Alpine rootfs keeps Alpine's stock `/etc/init.d/networking`.
- The packaged service is not rewritten, shadowed, or replaced by rootfs
  generation.  Any Bunix-specific behavior lives in standard configuration
  files, hook scripts, or lower-level compatibility fixes.
- `make run` boots through BusyBox init, runs `openrc sysinit`, `openrc boot`,
  and `openrc default`, then BusyBox init manages the `ttyS0` login respawn.
- The stock OpenRC `networking` service starts `eth0` through ifupdown-ng and
  `udhcpc`, installs the DHCP lease and default route, and does not require a
  Bunix-specific replacement service.

Constraints:

- BusyBox init, not OpenRC, manages login respawn processes.
- Do not reintroduce a blocking `bunix-login` OpenRC service.
- Do not bypass OpenRC networking with bootstrap DHCP or test-only hooks.
- Do not regress the current working default Alpine boot while this exploration
  is pending; keep the temporary generated service enabled until the real
  Alpine service has a deterministic test and a replacement path.
- Keep policy in userspace: Alpine OpenRC, ifupdown-ng, and BusyBox/Alpine
  `udhcpc` drive network configuration.
- Any Bunix-specific compatibility should be in general VFS, mount, procfs,
  sysfs, or Linux syscall behavior, not a forked networking script.
- Preserve the existing working temporary service until this exploration has a
  deterministic failing test and the stock service path is ready to replace it.

Investigation targets:

- Identify the exact helper or syscall path that causes stock OpenRC boot to
  stop after `fsck` and before the normal login respawn.
- Audit stock `localmount`, `mkmntdirs`, `mountinfo`, `fstabinfo`, and
  `/proc/self/mountinfo` expectations against Bunix VFS/procfs behavior.
- Verify whether `/etc/fstab` entries, mount table reporting, `statfs`,
  `getmntent`, `openat`, `fstat`, `readlink`, or directory mutation semantics
  are still incomplete for OpenRC.

Completion criteria:

- A fresh `make run` Alpine environment uses Alpine's stock
  `/etc/init.d/networking`.
- Networking is up after login without manual intervention.
- BusyBox init owns login respawn after OpenRC returns.
- The temporary Bunix networking-service replacement is gone.

Implementation Notes:

- Added `ALPINE_NETWORKING_SERVICE=generated|stock` to
  `tools/build-alpine-rootfs.sh`.  The default remains `generated`, preserving
  the current working `make run` Alpine path while the stock service path is
  investigated.
- In `stock` mode, the rootfs builder leaves the packaged Alpine
  `/etc/init.d/networking` untouched and records
  `alpine_networking_service=stock` in the rootfs manifest.
- `tools/test-alpine-rootfs.sh` now checks the selected service mode.  In
  generated mode it requires the temporary `/bin/sh` helper body.  In stock
  mode it requires the packaged `#!/sbin/openrc-run` service and rejects the
  generated helper body.
- Added `make test-alpine-rootfs-stock-networking` as the deterministic
  packaging test for the stock service body.
- Added `make test-boot-alpine-stock-networking`, which builds
  `build/modules/alpine-stock-networking-disk0.sqfs` with the stock service,
  packages it into a separate VirtIO-net ESP, and runs the Alpine smoke boot
  markers without disturbing the default generated-service image.
- Added a stock-mode-only `/etc/init.d/bunix-dev` OpenRC provider that declares
  the already-mounted Bunix devfs as `dev` and `dev-mount`.  This keeps the
  generated networking-service rootfs unchanged while allowing stock OpenRC
  dependency resolution for services such as `hwdrivers`, `machine-id`, and
  `fsck` to progress without missing-provider warnings.
- A trial `/etc/fstab` root entry moved the timeout from after `fsck` into the
  visible `Checking local filesystems` step, so it was removed.  The remaining
  blocker is still in the stock OpenRC fsck/root/localmount band and should be
  solved as a general mount-table or Linux/VFS compatibility issue, not by
  adding a synthetic root fstab row.

Validation:

- `make test-alpine-rootfs-stock-networking`
- `make test-alpine-rootfs`
- `make test-boot-alpine-stock-networking` currently fails deterministically
  before login, with failure artifacts from the first run at
  `build/failures/20260710T085814Z-6344/20260710T090114Z-6344`.
- After adding the stock-mode dev provider, the narrow packaging checks pass:
  - `make test-alpine-rootfs`
  - `make test-alpine-rootfs-stock-networking`
- The stock boot probe still times out before `login:`.  The provider removes
  the previous `Service 'hwdrivers' needs non existent service 'dev'` and
  `Service 'machine-id' needs non existent service 'dev'` warnings.  The latest
  clean failure artifact is
  `build/failures/20260710T093004Z-20138/20260710T093304Z-20138`; it reaches
  OpenRC startup, mounts `/proc`, creates `/run/openrc` and `/run/lock`, caches
  dependencies, reports clock-skew warnings, and then times out before login.
- Added targeted debug controls while investigating the post-`fsck` stall:
  - `debug-serial-userspace` keeps server/userspace logs on serial after the
    Linux init handoff.
  - `debug-linux-rpc`, `debug-linux-wait`, `debug-linux-exec`,
    `debug-linux-path-taskNNN`, and `debug-linux-syscall-task=NNN` allow
    focused Linux-server and Linux-syscall tracing without enabling the full
    noisy strace path.
  - `BUNIX_BOOT_LOGIN_WAIT` and `BUNIX_BOOT_LOGIN_CAPTURE` let the boot harness
    run longer stock-OpenRC probes without changing the default test timeout.
- Added Linux-server open-file descriptions for VFS file and directory FDs.
  `openat()` now creates a per-open description, while `dup()` and `fork()`
  share that description.  This gives directory streams and regular files
  Linux-style shared offsets only for duplicated/inherited FDs, instead of
  accidentally sharing offsets through a global VFS handle table.
- A quiet stock boot probe with `BUNIX_BOOT_LOGIN_WAIT=360` still exceeded the
  old 180s login window and was interrupted after it remained at the stock
  OpenRC fsck/localmount band, confirming that this is not only serial logging
  overhead.
- Targeted `debug-linux-wait` and `debug-linux-exec` runs show BusyBox init
  waits on `/sbin/openrc boot`; OpenRC boot waits on the `/etc/init.d/fsck`
  wrapper; the wrapper execs through `/sbin/openrc-run` into the shell runner.
- Targeted syscall/path diagnostics for task 143 show the fsck wrapper is not
  asleep in `wait4`; it is actively walking OpenRC runlevel/provider state with
  repeated `faccessat`, `openat`, `getdents64`, and `newfstatat` calls.  The
  repeated paths include `bunix-dev`, `devfs`, `hwclock`, `osclock`, `swclock`,
  and `/etc/runlevels/boot/...`.
- The rootfs runlevel layout is intentionally minimal in stock-networking
  mode: `/etc/runlevels/sysinit/bunix-dev` exists to provide `dev` and
  `dev-mount`, while `/etc/runlevels/boot/networking` is the only boot
  runlevel symlink.  The repeated `boot` runlevel checks are OpenRC dependency
  resolution looking for providers/services, not a Bunix service respawn loop.
- The current blocker is therefore narrower than the first failure: stock
  OpenRC progresses through sysinit and into `/sbin/openrc boot`, but the
  OpenRC fsck/service wrapper path is still too slow or is repeatedly probing
  missing provider state through VFS.  The next fix should improve the general
  VFS/Linux path behavior that makes those probes expensive or incorrect,
  rather than adding another generated networking-service workaround.
- A later no-debug stock boot measurement was allowed to run past 390 seconds
  of host time.  It eventually printed the visible fsck `[ ok ]` line but still
  did not advance to networking, default runlevel, or login.  This rules out a
  small timeout adjustment; the service wrapper is effectively stuck or doing
  unbounded OpenRC state/path probing after fsck succeeds.
- Added and tested a conservative VFS symlink-resolution cache keyed by
  subject task and absolute path.  The cache only stores fully-followed
  resolution results and is cleared on VFS mutations and mount changes.  This
  reduces repeated prefix symlink walks, but it is not sufficient to get the
  stock OpenRC boot to `login:` inside the current 180 second stock-networking
  boot test.
- A targeted userspace-serial debug boot with `debug-linux-wait`,
  `debug-linux-exec`, and late path tracing shows the post-`fsck` slowdown is a
  fork/exec storm of tiny OpenRC helpers, especially
  `/usr/libexec/rc/bin/shell_var`.  Each helper itself exits quickly, but the
  parent shell/OpenRC wrapper exits with status 1 and OpenRC continues to the
  next dependency/provider query.  This is normal OpenRC behavior made
  pathological by Bunix's current per-process/per-exec cost.
- A trial SquashFS regular-file data-block cache was rejected.  Caching full
  128 KiB data blocks on first read made the stock boot slower before the fsck
  phase, because OpenRC dependency generation touches many tiny service/helper
  files.  Do not reintroduce a first-touch full-block SquashFS cache without a
  narrower hot-block policy or measured evidence that it helps.
- A trial proc-server ELF segment batching change was also rejected.  Raising
  the proc loader buffer to 64 KiB and calling `task_map` on larger aligned
  chunks reduced the theoretical syscall/RPC count, but it regressed both the
  synthetic shell boot and the stock OpenRC probe.  The synthetic image stopped
  after `/bin/first` printed `first: exit 0`, and the stock Alpine image timed
  out before the visible fsck phase.  Keep the page-at-a-time loader until the
  lower-level task-map/process-exit interaction is understood.
- Added `debug-proc-exec-timing` to measure proc-server loader phases.  A
  stock OpenRC capture with this flag shows `shell_var` and `fstabinfo` exec
  image loading completes in 0-2 timer ticks; the visible 1-2 second gaps occur
  before the helper reaches `execve`, while the parent shell/OpenRC wrapper is
  running script logic and Linux syscalls.  This rules out proc-server ELF
  mapping as the dominant post-`fsck` cost.
- The next likely fix is therefore in Linux syscall/VFS behavior exercised by
  the parent shell/OpenRC scripts, or in a narrower path/stat cache.  Reduce
  repeated shell-side service/dependency probing cost without replacing the
  stock scripts or turning cold reads of many small OpenRC files into full-block
  reads.
- A trial Linux signal-pending optimization was rejected.  Narrowing the
  kernel's post-syscall signal-pending RPC to selected syscall classes reduced
  stock OpenRC `signal_pending` volume in debug captures, but it broke the
  synthetic shell path: `BUNIX_SHELL_PART=linux-signaltest make
  test-shell-part` stopped after `/bin/first` exited and never reached the
  login prompt.  The current Linux/SIGCHLD contract still depends on broad
  post-syscall signal polling, so this should not be reduced without first
  adding an explicit wakeup/pending-signal mechanism between linux.server and
  blocked Linux tasks.
- While investigating the slowdown, a clean-HEAD rebuild of `proc.server` and
  `linux.server` still reproduced the synthetic shell stop-after-first-exit
  failure with `BUNIX_SHELL_PART=linux-signaltest make test-shell-part`; latest
  artifact from this turn:
  `build/failures/20260710T204030Z-17038/20260710T204150Z-17038`.  Resolve or
  re-baseline that failure before relying on the shell shard as a guard for
  stock OpenRC performance changes.
- The synthetic shell guard regression was resolved without changing the
  intended login ownership model.  `proc.server` now accepts
  `BUNIX_PROC_EXIT` from a native task for its own process record, while still
  requiring the management capability for killing some other task.  This lets
  native one-shot boot spawns report completion instead of leaving bootstrap
  stuck waiting.
- A trial bootstrap `::respawn:/bin/login` entry was rejected.  BusyBox init
  already starts the synthetic `/etc/inittab` login while `/etc/spawns` is
  still running one-shot smoke programs, so bootstrap-level login supervision
  creates a second login that races for serial input.  Keep login respawn owned
  by BusyBox init, matching the Alpine constraint.
- The login prompt storm was caused by `/bin/login` treating empty reads as an
  immediate reason to print another prompt.  Login now waits and backs off
  through the time service when `read(0, ...)` returns no input, so it prints a
  prompt once and waits for console input instead of consuming later shell
  commands as a password.
- The synthetic shell guard is usable again:
  `BUNIX_SHELL_PART=linux-signaltest make test-shell-part` passed in 5
  seconds after the proc/login fixes, and passed again in 65 seconds after a
  full harness rebuild with the login fallback-yield path.
- A fresh post-reboot stock OpenRC debug probe reproduced the slow path with
  current artifacts at
  `build/failures/20260710T211208Z-30499/20260710T211508Z-30499`.  The late
  trace is still one OpenRC process repeatedly issuing `faccessat()` and
  `openat()` against service/provider names such as `devfs`, `bunix-dev`,
  `hwclock`, `osclock`, `swclock`, `dmesg`, and `modloop`.
- Added a conservative Linux-server `faccessat()` result cache keyed by process,
  VFS base handle, path, mode, and flags.  The cache is invalidated by a global
  VFS mutation epoch bumped on create, remove, link, rename, chmod, chown,
  truncate, mount, and unmount operations.  This reduces repeated Linux-to-VFS
  access RPCs without caching across filesystem mutations.
- The access cache is not sufficient by itself.  A normal stock-networking boot
  still timed out before `login:` with artifacts at
  `build/failures/20260710T212042Z-974/20260710T212342Z-974`.
- The stock networking service declares `need localmount hostname` and orders
  after `bootmisc`, `hwdrivers`, and `modules`, while `localmount` pulls in
  `root`, `fsck`, and a `clock` provider.  The previous stock-networking policy
  enabled only `networking`, which forced OpenRC to resolve a graph full of
  absent providers.  The policy now enables the stock provider chain in boot
  using `osclock` as the lightweight clock provider and exposes `bunix-dev` to
  boot dependencies as well as sysinit.
- The runlevel-policy change packages correctly:
  `make test-alpine-rootfs-stock-networking` passed.  A boot probe after adding
  the provider chain still timed out before `login:` at
  `build/failures/20260710T212521Z-3051/20260710T212821Z-3051`, and a debug
  probe still shows repeated provider probing at
  `build/failures/20260710T212844Z-4237/20260710T213144Z-4237`.
- Narrow guards after the access-cache change:
  `BUNIX_SHELL_PART=path-limits-statfs make test-shell-part` passed in 78
  seconds, and `BUNIX_SHELL_PART=linux-signaltest make test-shell-part` passed
  in 5 seconds.
