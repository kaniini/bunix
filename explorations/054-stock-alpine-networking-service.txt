Restore the stock Alpine OpenRC networking service.

Problem:

- Exploration 026 allows progress by generating a Bunix-specific
  `/etc/init.d/networking` service in the Alpine rootfs.
- That is acceptable as a temporary bootstrap path, but the intended Alpine
  environment should run the real Alpine/ifupdown-ng networking service.
- The current generated service is fine for now because it keeps `make run`
  useful while networking is still being brought up, but it is explicitly a
  compatibility crutch to remove in a future slice.
- The current replacement service is a stopgap only.  It should not become the
  Bunix network-management interface or a permanent OpenRC compatibility
  contract.
- The stock service currently pulls in OpenRC dependencies such as
  `localmount` and helper programs like `mkmntdirs`, `mountinfo`, and
  `fstabinfo` that expose gaps in Bunix's VFS, mount-table, and Linux
  compatibility behavior.

Intended outcome:

- The generated Alpine rootfs keeps Alpine's stock `/etc/init.d/networking`.
- The packaged service is not rewritten, shadowed, or replaced by rootfs
  generation.  Any Bunix-specific behavior lives in standard configuration
  files, hook scripts, or lower-level compatibility fixes.
- `make run` boots through BusyBox init, runs `openrc sysinit`, `openrc boot`,
  and `openrc default`, then BusyBox init manages the `ttyS0` login respawn.
- The stock OpenRC `networking` service starts `eth0` through ifupdown-ng and
  `udhcpc`, installs the DHCP lease and default route, and does not require a
  Bunix-specific replacement service.

Constraints:

- BusyBox init, not OpenRC, manages login respawn processes.
- Do not reintroduce a blocking `bunix-login` OpenRC service.
- Do not bypass OpenRC networking with bootstrap DHCP or test-only hooks.
- Do not regress the current working default Alpine boot while this exploration
  is pending; keep the temporary generated service enabled until the real
  Alpine service has a deterministic test and a replacement path.
- Keep policy in userspace: Alpine OpenRC, ifupdown-ng, and BusyBox/Alpine
  `udhcpc` drive network configuration.
- Any Bunix-specific compatibility should be in general VFS, mount, procfs,
  sysfs, or Linux syscall behavior, not a forked networking script.
- Preserve the existing working temporary service until this exploration has a
  deterministic failing test and the stock service path is ready to replace it.

Investigation targets:

- Identify the exact helper or syscall path that causes stock OpenRC boot to
  stop after `fsck` and before the normal login respawn.
- Audit stock `localmount`, `mkmntdirs`, `mountinfo`, `fstabinfo`, and
  `/proc/self/mountinfo` expectations against Bunix VFS/procfs behavior.
- Verify whether `/etc/fstab` entries, mount table reporting, `statfs`,
  `getmntent`, `openat`, `fstat`, `readlink`, or directory mutation semantics
  are still incomplete for OpenRC.

- The temporary Bunix networking-service replacement is gone.

Implementation Notes:

- Added `ALPINE_NETWORKING_SERVICE=generated|stock` to
  `tools/build-alpine-rootfs.sh`.  The default remains `generated`, preserving
  the current working `make run` Alpine path while the stock service path is
  investigated.
- In `stock` mode, the rootfs builder leaves the packaged Alpine
  `/etc/init.d/networking` untouched and records
  `alpine_networking_service=stock` in the rootfs manifest.
- `tools/test-alpine-rootfs.sh` now checks the selected service mode.  In
  generated mode it requires the temporary `/bin/sh` helper body.  In stock
  mode it requires the packaged `#!/sbin/openrc-run` service and rejects the
  generated helper body.
- Added `make test-alpine-rootfs-stock-networking` as the deterministic
  packaging test for the stock service body.
- Added `make test-boot-alpine-stock-networking`, which builds
  `build/modules/alpine-stock-networking-disk0.sqfs` with the stock service,
  packages it into a separate VirtIO-net ESP, and runs the Alpine smoke boot
  markers without disturbing the default generated-service image.
