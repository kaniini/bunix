Isolate riscv64 boot-test serial logs so riscv64 gates can run in parallel.

Status:

- Pending.
- Start after the existing riscv64 bringup and board-port queue unless test
  throughput becomes the active bottleneck sooner.

Problem:

- The riscv64 boot targets currently share `$(RISCV64_SERIAL_LOG)`, which is
  `build/riscv64-serial.log`.
- Running `make test-boot-riscv64-alpine` and
  `make test-boot-riscv64-uart-console` concurrently can make one target read
  the other target's log and fail marker checks even when both guests behaved
  correctly.
- This blocks safe parallelization of riscv64 boot shards as the riscv64 test
  matrix grows.

Tasks:

- [x] Give each riscv64 boot target its own default serial log path.
- [x] Allow callers to override each log path without affecting other riscv64
      targets.
- [x] Add a narrow host-side or make-level check that proves two riscv64 boot
      targets do not write the same log by default.
- [x] Update test-efficiency notes with the riscv64 log-isolation contract.

Deferred:

- BPI-F3 helper-script log isolation belongs with the postponed BPI-F3 block.
  Do not do BPI-F3-specific script work in this exploration.

Needs:

- RISC-V boot verification is blocked by pre-existing userspace/server parity
  issues.  The first blocker, unknown native syscall `-72`
  (`BUNIX_SYSCALL_IPC_TRY_RECV`), has a local fix.  After that fix,
  `test-boot-riscv64-early` reaches `bootstrap-riscv64: time launched`, then
  `time` exits with status 1 because the riscv64 bootstrap path still launches
  services without the names registration claim needed by the current server
  authority model.
- Follow-up exploration `079-riscv64-early-userspace-server-parity.txt`
  captures the broader riscv64 userspace/server bringup work exposed while
  attempting the boot verification.  Keep exploration 058 scoped to serial-log
  isolation.
