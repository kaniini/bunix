Bring riscv64 early userspace server boot parity up to the point where the
isolated riscv64 boot tests can run useful guest markers again.

Status:

- Pending.
- This is follow-up work from exploration 058 verification.  Do not fold this
  work back into the serial-log isolation slice.

Problem:

- Exploration 058 isolated the riscv64 boot-test serial logs, but the guest
  boot verification exposed pre-existing riscv64 userspace/server parity gaps.
- The first clean failure is an unknown native syscall from riscv64 userspace:
  `BUNIX_SYSCALL_IPC_TRY_RECV` (`-72`).
- Scratch debugging also found likely next blockers around native task
  introspection, timer wakeups for sleeping server loops, claimed service
  registration, and proc/VFS exec authority once the first syscall blocker is
  removed.

Findings:

- With `ipc_try_recv` implemented, `make test-boot-riscv64-early` no longer
  reports unknown syscall `-72`.
- RISC-V now has native `task_info` syscall parity, so bootstrap can resolve
  real task IDs after launching modules that return service handles.
- The early RISC-V timer now treats the platform timebase and scheduler tick
  rate separately, re-arms the timer on each interrupt, and wakes sleepers.
- The RISC-V bootstrap now claims names admin authority, creates service
  registration claims, passes user/proc/linux management ports, launches core
  servers with registration claims, and uses the proc management port for
  exec/spawn control.
- Proc now opens executable images and interpreters with the target task ID as
  the VFS subject.  Linux-personality tasks are registered with the Linux server
  before image load so user/VFS permission checks can resolve credentials.
- `make test-boot-riscv64-early` now reaches `bootstrap-riscv64: linux ready`.
  The remaining blocker is a timeout while waiting for the first proc-spawned
  Linux smoke binary to complete.
