Fix VFS permission checks that trust caller-supplied task IDs.

Problem:

- VFS and filesystem translators frequently decide permissions using task IDs
  supplied in RPC payload words rather than deriving caller identity from the
  sender or from an authenticated delegation.
- Several permission helpers treat task ID `0` as privileged.
- A caller with direct VFS or translator send authority can potentially forge
  task identity, including task `0`, to bypass file mode checks or metadata
  ownership checks.

Threat model:

- Primary attacker: an unprivileged Linux process if Linux-server syscall paths
  can be confused into forwarding forged task IDs or if hybrid programs gain
  direct VFS RPC authority.
- Secondary attacker: a compromised userspace server with VFS or translator
  send authority.
- Impact candidates: unauthorized file read/write, chmod/chown, node creation,
  tmpfs mutation, mount-layer mutation, and protected metadata disclosure.

Evidence:

- `user/vfs/main.c` uses `message.words[3] & 0xffffffff` as the task passed to
  `can_read_entry()`, `can_search_entry()`, `vfs_resolve_symlinks()`, and
  downstream translator requests.
- `user/tmpfs/main.c` uses task IDs from `message.words[3]` in
  `task_can_access()`, `task_can_chmod()`, and `task_can_chown()`, and those
  helpers treat `task == 0` as privileged.
- `user/rootfs/main.c` has the same pattern in `task_can_access()`, including
  special handling for `task == 0`.
- `user/procfs/main.c` uses the task ID embedded in path/stat/readlink message
  words to resolve `/proc/self` and related process views.

Root cause:

- The VFS protocol conflates request metadata with authority.  Task identity is
  passed as untrusted message data rather than being bound to the authenticated
  sender or to a delegated credential object.

Fix plan:

- Define a VFS credential/subject primitive that can represent Linux process
  credentials, Bunix task identity, and explicitly delegated server authority.
- Bind path and open operations to authenticated subjects instead of accepting
  raw task IDs from arbitrary callers.
- Preserve legitimate server-initiated filesystem operations by giving servers
  explicit scoped credentials or service authority.
- Remove `task == 0` as an ambient privilege shortcut from translators, or
  restrict it to requests carrying a bootstrap/root filesystem authority.
- Add regression tests for forged task IDs on read, write, chmod, chown,
  mkdir, mknod, symlink, truncate, procfs `/proc/self`, and tmpfs mutation.

Tasks:

- [x] Build a local reproducer that forges VFS task identity from a weaker

- [x] Define the VFS credential/subject model for Linux processes, native
      Bunix tasks, and privileged servers.
- [ ] Update VFS and translators to derive/check authenticated subjects.
- [ ] Remove or strictly gate privileged `task == 0` behavior.
- [ ] Add regression tests for forged task IDs across rootfs, tmpfs, procfs,
      unionfs, and ext2 paths.
- [ ] Run focused permission tests, shell tests, and full tests before
      completion.

Notes:

- Added `test-vfs-subject-authority`, which launches `mgmt-test` with ordinary
  VFS send authority and proves it cannot forge task 0 to create
  `/root/forged-root`.  Before the VFS subject gate, this test failed with
  `vfs-subject-auth-test: forged root succeeded`.
- VFS now has an explicit subject-issuer grant,
  `BUNIX_VFS_GRANT_SUBJECT_TASK`.  Senders without that grant have the low
  32-bit task subject in VFS request metadata rewritten to their authenticated
  sender task before path or remote-handle forwarding.  Trusted brokers such as
  Linux may still assert process subjects after bootstrap delegation.
- Bootstrap claims the initial VFS subject grantor role, delegates subject
  authority to Linux and trusted bootstrap servers that need service/root
  filesystem credentials, and grants it to the existing VFS handle-authority
  fixture only for that fixture.
- Focused verification passed: `make test-vfs-subject-authority`,
  `make test-vfs-authority`, and `make test-vfs-mount-authority`.
- Added `test-tmpfs-subject-authority`, which launches `mgmt-test` with only
  names and console authority and proves that a direct VFS-protocol message to
  tmpfs cannot forge task 0 to create `/tmp/direct-forged-root`.
- Tmpfs now gates VFS-protocol callers through the same
  `BUNIX_VFS_GRANT_SUBJECT_TASK` grant primitive, with bootstrap as the first
  grantor.  Bootstrap delegates direct tmpfs VFS caller authority to VFS,
  unionfs, and Linux for legitimate mount/union/writeback paths.
- Tightening VFS subject authority exposed two legitimate service identities:
  proc opens executable images during exec, and user reads `/etc/passwd`,
  `/etc/shadow`, and `/etc/group` for login.  Bootstrap now captures their task
  IDs during launch and delegates VFS subject authority to those server tasks.
- Regression debugging confirmed that omitting proc authority broke
  `/sbin/init` exec with `proc: exec open failed`, and omitting user authority
  stalled hybrid login after password entry while the user server read account
  files.
- Focused verification for this slice passed: `make
  test-tmpfs-subject-authority`, `make test-vfs-subject-authority`, `make
  test-vfs-authority`, `make test-vfs-mount-authority`, and
  `BUNIX_TEST_SET=tmpfs-basic-linux-tests make test-shell-parallel`.
- Added `test-squashfs-subject-authority`, which resolves the squashfs
  translator directly and verifies that a caller with only names/console
  authority cannot forge task 0 to open `/etc/shadow` through the translator.
- Squashfs now gates direct VFS-protocol callers with
  `BUNIX_VFS_GRANT_SUBJECT_TASK`, matching tmpfs.  Bootstrap delegates only VFS
  and unionfs as legitimate direct squashfs VFS callers for the mounted lower
  root path.
- Focused verification for the squashfs slice passed: `make
  test-squashfs-subject-authority`, `make test-tmpfs-subject-authority`, `make
  test-vfs-subject-authority`, `make test-vfs-authority`, `make
  test-vfs-mount-authority`, and `BUNIX_TEST_SET=tmpfs-basic-linux-tests make
  test-shell-parallel`.
