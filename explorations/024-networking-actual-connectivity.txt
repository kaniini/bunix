Make networking actually work from the Alpine-style userspace environment.

Problem:

- Bunix has a core NET0 server, loopback plumbing, VirtIO-net device bringup,
  and initial route state, but the system does not yet behave like a normal
  userspace networked system.
- Alpine/OpenRC expects persistent network configuration under
  `/etc/network/interfaces`, ifupdown-ng to apply that configuration, the
  `/etc/init.d/networking` service to run during boot, and ordinary tools such
  as `ping` to work without Bunix-specific manual setup.
- The next useful milestone is end-to-end connectivity from the booted system:
  loopback IPv4, loopback IPv6, DHCP-provisioned external IPv4, and repeatable
  smoke tests against known external addresses.

Constraints:

- Keep networking policy in userspace.  Kernel participation should be limited
  to existing interrupt, scheduling, and privileged hardware-operation
  primitives.
- Preserve the server split: NET0 owns protocol state, routes, interfaces,
  socket behavior, and packet queues; VirtIO-net owns the NIC device; Linux
  compatibility translates Linux syscalls and ioctls into server RPCs.
- Use ifupdown-ng plus `/etc/network/interfaces` as the first network
  configuration source.  Do not hide network setup in bootstrap-only hardcoded
  defaults except for minimum bringup needed to start OpenRC and
  `/etc/init.d/networking`.
- Use BusyBox/Alpine `udhcpc` for external IPv4 DHCP configuration.  Static
  QEMU-user defaults are acceptable only as a fallback or test fixture when
  DHCP is explicitly absent.  Do not implement a Bunix DHCP client in this
  exploration; missing work should be socket/ioctl/script/VFS/NET0 plumbing
  needed to let `udhcpc` run normally.
- Keep IPv6 loopback working independently of external IPv6 support.  External
  IPv6 may be planned separately unless it is needed to satisfy the smoke tests.
- Tests must be deterministic enough for `make test-boot` and networking
  shards: every test must end in poweroff, avoid depending on the ambient host
  network unless the shard explicitly opts into it, and record enough logs to
  diagnose packet, route, or DHCP failures.

Inventory:

- `017-core-networking-stack.txt` completed localhost, IPv4/IPv6 route basics,
  socket compatibility, and `/proc/net` observability.
- `018-virtio-net.txt` completed the initial VirtIO-net driver attachment,
  packet-interface registration, TX/RX queues, and a deterministic QEMU-user
  route stub.
- `025-virtio-net-external-smoke.txt` now follows this exploration and should
  harden ARP/ND, packet egress, and host-side external-network smoke testing.
- `026-hot-route-tables.txt` follows the external smoke work and should replace
  route scans with dynamically growing IPv4/IPv6 HOT route tables.

Target behavior:

- The root filesystem contains ifupdown-ng and a normal
  `/etc/network/interfaces` suitable for Bunix's current Alpine bringup.
  Initial target contents should configure:
  - `lo` as loopback with `127.0.0.1/8` and `::1/128`.
  - The first VirtIO-net interface for DHCP IPv4 when present.
- OpenRC starts `/etc/init.d/networking` during the normal boot path.  That
  service uses ifupdown-ng to parse `/etc/network/interfaces`, configure
  loopback, start `udhcpc` for the external interface, install address and
  default-route state through Bunix-compatible hooks, and expose clear
  status/logging.
- BusyBox or Alpine `ping` succeeds for:
  - `127.0.0.1`
  - `::1`
  - `4.2.2.1`
  - `8.8.8.8`
- Failure modes produce bounded diagnostics that distinguish service startup,
  config parsing, DHCP lease acquisition, route installation, ARP resolution,
  packet transmit, packet receive, ICMP, and Linux syscall/personality issues.

Design notes:

- Interface policy should be driven by ifupdown-ng and OpenRC's
  `/etc/init.d/networking`, not by a parallel Bunix-specific configuration
  manager.  Existing native helpers such as `netcfg.server` may provide Bunix
  RPC hooks or temporary deterministic test setup, but the normal Alpine path
  should be ifupdown-ng invoking standard interface methods and hook scripts.
- DHCP should be driven by BusyBox/Alpine `udhcpc`, not by a new Bunix DHCP
  client.  The Bunix Linux compatibility layer, NET0 RPC surface, and hook
  scripts should provide the socket, script, route, address, and resolver
  plumbing needed for `udhcpc` and ifupdown-ng to configure NET0 through
  explicit RPCs rather than mutating NET0 internals.
- `/etc/init.d/networking` should be able to run from OpenRC.  The Bunix test
  harness may invoke a narrow `ifup` or `udhcpc` command first if that shortens
  debugging, but acceptance should come from the normal OpenRC service path.
- `ping` should exercise the Linux socket path where practical.  Native Bunix
  observability tools may be added for debugging, but the acceptance target is
  ordinary userspace commands.
- If QEMU user networking does not provide enough deterministic behavior for
  DHCP and ICMP, define a dedicated tap/namespace or socket-backed test shard
  and keep it separate from default fast boot tests until stable.

Inventory findings:

- NET0 lives in `user/net/main.c` and registers `BUNIX_SERVICE_NET`/`NET0`.
  It currently owns loopback interface state, packet-interface attachment,
  dynamic route add/enumeration, packet TX/RX queues, UDP sockets, TCP sockets,
  and `/proc/net` observation records.
- Existing NET0 configuration-style RPCs are
  `BUNIX_NET_PACKET_INTERFACE_ATTACH`, `BUNIX_NET_PACKET_INTERFACE_LINK`,
  `BUNIX_NET_ROUTE_ADD`, `BUNIX_NET_ROUTE_COUNT`, and `BUNIX_NET_ROUTE_AT`.
  There is no address add/delete RPC yet; IPv4/IPv6 addresses are embedded in
  the loopback interface record, and packet interfaces only expose flags, MTU,
  MAC, and counters.
- Loopback routes for `127.0.0.0/8` and `::1/128` are static inside NET0.
  UDP/TCP delivery is localhost-only today because socket connect/send paths
  require `net_addr_is_local()`.
- VirtIO-net lives in `user/virtio-net/main.c`; it attaches a packet interface,
  polls RX/TX, and currently installs a hardcoded QEMU-user IPv4 route
  (`10.0.2.0/24`) from inside the driver.  That is a manual policy leak that
  should move to ifupdown-ng/OpenRC bringup or a narrow Bunix hook invoked by
  that path.
- Bootstrap starts `virtio-net` only when `virtio-net-test` is present on the
  kernel command line.  Normal Alpine/OpenRC boots start NET0 but still need
  the NIC driver and OpenRC `/etc/init.d/networking` path wired cleanly.
- Linux socket compatibility in `user/linux/main.c` supports AF_INET/AF_INET6
  UDP/TCP sockets over NET0, including bind/connect/listen/accept/send/recv,
  `getsockname`, `getpeername`, `setsockopt`, `getsockopt`, and poll/select
  integration.  There is no ICMP socket path for BusyBox `ping` yet.
- Procfs networking observability is already backed by NET0 for
  `/proc/net/dev`, `/proc/net/route`, `/proc/net/sockstat`,
  `/proc/net/{udp,udp6,tcp,tcp6}` and is covered by `nettest`.
- Root image generation did not install `/etc/network/interfaces` before this
  exploration.  Synthetic and Alpine images now get a first config with `lo`
  loopback and `eth0` DHCP intent.
- Existing tests cover boot-time NET0 self-tests, Linux localhost TCP/UDP in
  `nettest`, procfs network files, and VirtIO-net device attach/queue bringup.
  They do not exercise ifupdown-ng, OpenRC `/etc/init.d/networking` startup,
  DHCP, ICMP, ARP, or external packet round trips.

NET0 configuration RPC contract:

- Keep packet-driver registration separate from network policy.  Drivers may
  attach packet interfaces and update physical link flags, but address
  assignment, default routes, DHCP leases, and service status belong to
  ifupdown-ng/OpenRC policy or Bunix hooks invoked by that policy.
- Add `BUNIX_NET_INTERFACE_SET_FLAGS` for administrative up/down state.  It
  takes interface id, set mask, clear mask, and flags; NET0 preserves physical
  `RUNNING` state from the driver and exposes the merged result through
  interface info/procfs.
- Add address records independent from `struct net_interface`:
  family, address hi/lo, prefix length, interface id, flags, preferred lifetime,
  valid lifetime.  Loopback starts with static `127.0.0.1/8` and `::1/128`
  records.  New RPCs: `BUNIX_NET_ADDR_ADD`, `BUNIX_NET_ADDR_DELETE`,
  `BUNIX_NET_ADDR_COUNT`, and `BUNIX_NET_ADDR_AT`.
- Extend route management with `BUNIX_NET_ROUTE_DELETE` and define
  `BUNIX_NET_ROUTE_ADD` as idempotent for identical route records.  Route
  records retain family, prefix, prefix length, interface id, gateway,
  flags, and metric.
- Add neighbor observation/control records for ARP/ND:
  `BUNIX_NET_NEIGHBOR_COUNT`, `BUNIX_NET_NEIGHBOR_AT`,
  `BUNIX_NET_NEIGHBOR_ADD`, and `BUNIX_NET_NEIGHBOR_DELETE`.  The first
  DHCP/IPv4 slice can expose an empty table plus manual add/delete until ARP
  resolution is implemented in the external-smoke exploration.
- Add `BUNIX_NET_DHCP4_LEASE_INSTALL` as a convenience RPC for the networking
  path.  It carries interface id, IPv4 address, prefix length or netmask,
  gateway, DNS server list metadata, lease lifetime, renewal time, and server
  identifier.  NET0 validates the interface and installs the address plus
  default route atomically for the `udhcpc`/ifupdown-ng hook.
- Add `BUNIX_NET_CONFIG_STATUS` for bounded diagnostics.  NET0 reports whether
  loopback is configured, whether each packet interface has an address,
  whether a default route exists, and the last configuration error code.  The
  ifupdown-ng/OpenRC bringup path should log config parsing, DHCP, route, and
  neighbor progress separately.

Implementation progress:

- Added the initial NET0 configuration RPC surface in `user/include/bunix/syscall.h`
  and `user/net/main.c`: administrative interface flags, address
  add/delete/enumeration, idempotent route add, route delete, DHCPv4 lease
  install, neighbor count stub, and config status.
- NET0 now treats assigned address records as the source of local address
  truth.  The static loopback addresses remain present at boot, while future
  DHCP/service configuration can add packet-interface addresses dynamically.
- Bootstrap's NET0 self-test now verifies address add/enumeration and config
  status in addition to the existing packet interface and route paths.
- DHCPv4 lease install currently installs the IPv4 address and default route
  atomically.  DNS metadata is accepted in the lease record but still needs a
  persistence/observation path before the DHCP task can be considered complete.
- Added an initial `netcfg.server` userspace service.  Bootstrap launches it
  after unionfs mounts `/`, waits for the `NETC` service registration, and then
  continues into Linux init.  `netcfg` reads `/etc/network/interfaces`,
  idempotently configures loopback through NET0, reports status through logs,
  and treats absent `eth0` as a normal no-NIC boot.  This is now considered a
  Bunix helper/test fixture; the target Alpine boot path should move policy to
  ifupdown-ng and OpenRC `/etc/init.d/networking`.
- VirtIO-net no longer installs the QEMU-user route itself.  In
  `virtio-net-test` boots, `netcfg` finds the running VirtIO packet interface
  and installs the current deterministic QEMU-user IPv4 lease through
  `BUNIX_NET_DHCP4_LEASE_INSTALL`.  This is an interim fallback, not the final
  DHCP client.
- Bootstrap now takes its packet-interface self-test interface back down so
  netcfg does not confuse the synthetic test interface with real `eth0`.
- Procfs now exposes `/proc/net/config`, backed by `BUNIX_NET_CONFIG_STATUS`,
  with loopback/default-route flags, interface/address/route counts, and the
  last NET0 configuration error.  `nettest` covers this alongside the existing
  `/proc/net` files.
- Added NET0 raw ICMP socket primitives and Linux `SOCK_RAW` compatibility for
  ICMP/ICMPv6.  BusyBox `ping -c 1 127.0.0.1` and
  `busybox ping -6 -c 1 ::1` now exercise Linux `socket`, `sendto`,
  `recvfrom`, and poll/select paths through NET0 loopback echo replies.
- Added a deterministic `network-loopback-ping` shell shard covering BusyBox
  IPv4 and IPv6 loopback ping with clean guest poweroff.
- Exposed the BusyBox networking applets in the synthetic rootfs:
  `/sbin/udhcpc`, `/sbin/ifup`, `/sbin/ifdown`, `/sbin/ip`, `/sbin/route`,
  and `/bin/ping`.
- The current target for interface bringup is ifupdown-ng's `/sbin/ifup` and
  `/sbin/ifdown`, started by OpenRC's `/etc/init.d/networking` service.  BusyBox
  applets may remain useful in the synthetic rootfs, but Alpine compatibility
  should not depend on BusyBox `ifup` semantics when ifupdown-ng is present.
- A direct VirtIO-net probe of `udhcpc -n -q -i eth0 -s /bin/echo -t 1 -T 1`
  now reaches the real DHCP client and fails at the first missing Linux socket
  primitive: `socket(AF_INET, SOCK_RAW, IPPROTO_RAW)`.
- Added Linux compatibility for `AF_INET/SOCK_RAW/IPPROTO_RAW` socket creation
  and forwarded read/write `ifreq` netdevice ioctls for `SIOCGIFINDEX`,
  `SIOCGIFFLAGS`, `SIOCGIFMTU`, and `SIOCGIFHWADDR`.  NET0 now exposes an
  interface-details RPC so Linux can answer those queries from live interface
  state.
- The same VirtIO-net `udhcpc` probe now reaches `udhcpc: started` and the
  `deconfig` script event.  The next missing primitive is packet socket
  support: `socket(AF_PACKET, SOCK_DGRAM, ETH_P_IP)`.
- Added initial Linux `AF_PACKET`/`SOCK_DGRAM` support for `ETH_P_IP`,
  including packet bind/send handling for `sockaddr_ll`, NET0 packet RX
  queueing, and packet poll/dequeue RPCs for cooked IPv4 packet receive.
- The VirtIO-net `udhcpc -n -q -i eth0 -s /bin/echo -t 1 -T 1` probe now
  reaches `udhcpc: broadcasting discover` without the previous socket or
  `sendto` errors, but it still does not complete or report a lease.  The next
  slice needs to debug the post-discover hang and prove packet TX completion,
  RX delivery, and DHCP offer handling end-to-end.
- Verified the physical VirtIO-net link state through NET0/procfs.  The real
  NIC reports `flags 000D` (`UP`, `BROADCAST`, `RUNNING`) in
  `/proc/net/config`, while the bootstrap self-test interface is down.
- Aligned procfs and Linux netdevice naming so the first non-loopback NET0
  packet interface is consistently `eth0`, and Linux ioctls now resolve
  `ethN` names instead of recognizing only `eth0`.
- Verified the QEMU DHCP path on the wire with a packet dump: Bunix transmits
  the DHCP Discover from `52:54:00:18:00:01`, and QEMU replies with a DHCP
  Offer for `10.0.2.15`.  NET0 receives and queues that Offer on the running
  `eth0` packet interface (`rxq 1` in `/proc/net/config`), but `udhcpc` still
  does not dequeue or process the packet.  The next missing piece is in the
  Linux packet-socket receive/wait path used by `udhcpc`, not in physical link
  bringup or QEMU packet delivery.
- Linux `read()` now routes UDP, TCP, ICMP, and packet sockets through the
  socket receive path instead of only accepting TCP sockets.  This matches
  normal Linux socket semantics and passed `make test-boot`, but was not
  sufficient by itself to make `udhcpc` consume the queued DHCP Offer.
- During the `udhcpc` hang, `/proc/net/udp` is empty, so the DHCP client is
  not waiting on a NET0 UDP socket despite transmitting UDP packets on the
  wire.  NET0 now performs initial inbound IPv4/UDP demux when packet drivers
  submit Ethernet frames, delivering matching broadcast or addressed UDP
  datagrams to bound IPv4 UDP sockets while still preserving the frame for
  packet-socket consumers.
- Linux raw IPv4 compatibility now accepts `SOCK_RAW`/`IPPROTO_UDP`, supports
  `read`/`recvfrom` by draining the default packet interface and returning the
  IPv4 packet without the Ethernet header, and polls raw sockets against the
  packet RX queue instead of reporting them as always readable.  This is
  necessary for raw-socket DHCP implementations, but the current BusyBox
  `udhcpc` probe still does not advance past `broadcasting discover`.
- Fixed the post-discover `udhcpc` wait path by preserving `O_NONBLOCK` through
  `fcntl(F_SETFL)`, honoring nonblocking empty-pipe reads with `EAGAIN`, and
  covering that behavior in `nettest`.  BusyBox uses this signal pipe around
  its DHCP poll loop; dropping the flag caused it to block before processing
  packet-socket readiness.
- Fixed receive-side `recvmsg` compatibility for ancillary-control buffers.
  BusyBox enables `PACKET_AUXDATA` and passes a control buffer to
  packet-socket `recvmsg`; Bunix now accepts that receive shape and reports
  zero control data instead of returning `EINVAL`.  `nettest` now covers
  `recvmsg` with a control buffer.
- Verified direct QEMU-user DHCP acquisition with
  `udhcpc -n -q -i eth0 -s /bin/echo -t 1 -T 1`: Bunix sends Discover and
  Select/Request packets, QEMU replies, `udhcpc` obtains lease `10.0.2.15`
  from server `10.0.2.2`, and `/proc/net/config` shows `eth0` with drained
  RX/TX queues (`rxq 0`, `txq 0`).  The remaining DHCP work is integration:
  ifupdown-ng/OpenRC script execution, lease installation through Bunix hooks,
  and deterministic test-shard poweroff.
- Added a native Bunix `bunix-udhcpc-script` hook and install it as
  `/usr/share/udhcpc/default.script` in both synthetic and Alpine rootfs
  images.  BusyBox/Alpine `udhcpc` now runs its normal script path; on
  `bound` or `renew`, the hook reads lease environment variables, maps
  `ethN` to the matching NET0 packet interface, and installs the IPv4 lease
  through `BUNIX_NET_DHCP4_LEASE_INSTALL`.
- Alpine rootfs generation now includes `ifupdown-ng`, exposing the packaged
  `/sbin/ifup`, `/sbin/ifdown`, and `/etc/init.d/networking` service.  The
  builder links `networking` into the OpenRC boot runlevel.  A plain Alpine
  boot still passes; the Alpine+VirtIO boot shard currently reaches login but
  stalls in the existing `busybox ps` smoke command, so OpenRC networking
  service acceptance remains pending until that shard can prove service
  execution directly.
- Added `make test-boot-virtio-net-dhcp`, a deterministic QEMU-user DHCP
  command shard.  It boots the VirtIO-net image, logs in as root, runs
  `udhcpc -n -q -i eth0 -t 1 -T 1` through the default script, checks
  `/proc/net/config` for `eth0` and drained queues, prints a status marker,
  and powers off through the command harness.
- Added a minimal Linux `AF_NETLINK`/`NETLINK_ROUTE` compatibility shim for
  route sockets.  It accepts raw route sockets, bind, `getsockname`,
  `setsockopt`, `SO_TYPE`, `sendmsg`/`write` request submission, ACK-style
  receive, and poll/read behavior sufficient for BusyBox `ip link set up` in
  ifupdown-ng's link executor.  This is deliberately not full rtnetlink
  enumeration yet; route/link dumps still need a real NET0-backed
  implementation in a later networking slice.
- Added `make test-boot-virtio-net-ifup`, a deterministic Alpine+VirtIO-net
  command shard.  It forces the Alpine rootfs, runs `/sbin/ifdown eth0` and
  `/sbin/ifup eth0`, verifies that `udhcpc` obtains the QEMU-user lease, checks
  `/proc/net/config` for `eth0`, `default_ipv4 1`, and drained RX/TX queues,
  and powers off through the command harness.
- Direct `/etc/init.d/networking restart` still returns nonzero even though
  NET0 is configured.  The current evidence points at OpenRC wrapper/runtime
  behavior (`readpath` noise and service return handling), so the OpenRC
  service acceptance task remains open until the service itself returns success
  and proves startup from the normal boot runlevel.
