Make networking actually work from the Alpine-style userspace environment.

Problem:

- Bunix has a core NET0 server, loopback plumbing, VirtIO-net device bringup,
  and initial route state, but the system does not yet behave like a normal
  userspace networked system.
- Alpine/OpenRC expects persistent network configuration under
  `/etc/network/interfaces`, a networking service that configures interfaces
  during boot, and ordinary tools such as `ping` to work without Bunix-specific
  manual setup.
- The next useful milestone is end-to-end connectivity from the booted system:
  loopback IPv4, loopback IPv6, DHCP-provisioned external IPv4, and repeatable
  smoke tests against known external addresses.

Constraints:

- Keep networking policy in userspace.  Kernel participation should be limited
  to existing interrupt, scheduling, and privileged hardware-operation
  primitives.
- Preserve the server split: NET0 owns protocol state, routes, interfaces,
  socket behavior, and packet queues; VirtIO-net owns the NIC device; Linux
  compatibility translates Linux syscalls and ioctls into server RPCs.
- Use `/etc/network/interfaces` as the first network configuration source.
  Do not hide network setup in bootstrap-only hardcoded defaults except for
  minimum bringup needed to start the networking service.
- Use DHCP for external IPv4 configuration.  Static QEMU-user defaults are
  acceptable only as a fallback or test fixture when DHCP is explicitly absent.
- Keep IPv6 loopback working independently of external IPv6 support.  External
  IPv6 may be planned separately unless it is needed to satisfy the smoke tests.
- Tests must be deterministic enough for `make test-boot` and networking
  shards: every test must end in poweroff, avoid depending on the ambient host
  network unless the shard explicitly opts into it, and record enough logs to
  diagnose packet, route, or DHCP failures.

Inventory:

- `017-core-networking-stack.txt` completed localhost, IPv4/IPv6 route basics,
  socket compatibility, and `/proc/net` observability.
- `018-virtio-net.txt` completed the initial VirtIO-net driver attachment,
  packet-interface registration, TX/RX queues, and a deterministic QEMU-user
  route stub.
- `025-virtio-net-external-smoke.txt` now follows this exploration and should
  harden ARP/ND, packet egress, and host-side external-network smoke testing.
- `026-hot-route-tables.txt` follows the external smoke work and should replace
  route scans with dynamically growing IPv4/IPv6 HOT route tables.

Target behavior:

- The root filesystem contains a normal `/etc/network/interfaces` suitable for
  Bunix's current Alpine bringup.  Initial target contents should configure:
  - `lo` as loopback with `127.0.0.1/8` and `::1/128`.
  - The first VirtIO-net interface for DHCP IPv4 when present.
- A networking service starts during the normal boot path, parses
  `/etc/network/interfaces`, configures loopback, starts DHCP for the external
  interface, installs address and default-route state through NET0 RPCs, and
  exposes clear status/logging.
- BusyBox or Alpine `ping` succeeds for:
  - `127.0.0.1`
  - `::1`
  - `4.2.2.1`
  - `8.8.8.8`
- Failure modes produce bounded diagnostics that distinguish service startup,
  config parsing, DHCP lease acquisition, route installation, ARP resolution,
  packet transmit, packet receive, ICMP, and Linux syscall/personality issues.

Design notes:

- DHCP can start with an MVP DHCPv4 client in a Bunix networking helper program
  or server, but it should configure NET0 through explicit RPCs rather than
  mutating NET0 internals.
- The networking service should be able to run from OpenRC, but the Bunix test
  harness may invoke a narrow bringup command first if that shortens debugging.
- `ping` should exercise the Linux socket path where practical.  Native Bunix
  observability tools may be added for debugging, but the acceptance target is
  ordinary userspace commands.
- If QEMU user networking does not provide enough deterministic behavior for
  DHCP and ICMP, define a dedicated tap/namespace or socket-backed test shard
  and keep it separate from default fast boot tests until stable.

Inventory findings:

- NET0 lives in `user/net/main.c` and registers `BUNIX_SERVICE_NET`/`NET0`.
  It currently owns loopback interface state, packet-interface attachment,
  dynamic route add/enumeration, packet TX/RX queues, UDP sockets, TCP sockets,
  and `/proc/net` observation records.
- Existing NET0 configuration-style RPCs are
  `BUNIX_NET_PACKET_INTERFACE_ATTACH`, `BUNIX_NET_PACKET_INTERFACE_LINK`,
  `BUNIX_NET_ROUTE_ADD`, `BUNIX_NET_ROUTE_COUNT`, and `BUNIX_NET_ROUTE_AT`.
  There is no address add/delete RPC yet; IPv4/IPv6 addresses are embedded in
  the loopback interface record, and packet interfaces only expose flags, MTU,
  MAC, and counters.
- Loopback routes for `127.0.0.0/8` and `::1/128` are static inside NET0.
  UDP/TCP delivery is localhost-only today because socket connect/send paths
  require `net_addr_is_local()`.
- VirtIO-net lives in `user/virtio-net/main.c`; it attaches a packet interface,
  polls RX/TX, and currently installs a hardcoded QEMU-user IPv4 route
  (`10.0.2.0/24`) from inside the driver.  That is a manual policy leak that
  should move to the networking service.
- Bootstrap starts `virtio-net` only when `virtio-net-test` is present on the
  kernel command line.  Normal Alpine/OpenRC boots start NET0 but do not start a
  network configuration service or NIC driver.
- Linux socket compatibility in `user/linux/main.c` supports AF_INET/AF_INET6
  UDP/TCP sockets over NET0, including bind/connect/listen/accept/send/recv,
  `getsockname`, `getpeername`, `setsockopt`, `getsockopt`, and poll/select
  integration.  There is no ICMP socket path for BusyBox `ping` yet.
- Procfs networking observability is already backed by NET0 for
  `/proc/net/dev`, `/proc/net/route`, `/proc/net/sockstat`,
  `/proc/net/{udp,udp6,tcp,tcp6}` and is covered by `nettest`.
- Root image generation did not install `/etc/network/interfaces` before this
  exploration.  Synthetic and Alpine images now get a first config with `lo`
  loopback and `eth0` DHCP intent.
- Existing tests cover boot-time NET0 self-tests, Linux localhost TCP/UDP in
  `nettest`, procfs network files, and VirtIO-net device attach/queue bringup.
  They do not exercise OpenRC networking service startup, DHCP, ICMP, ARP, or
  external packet round trips.

NET0 configuration RPC contract:

- Keep packet-driver registration separate from network policy.  Drivers may
  attach packet interfaces and update physical link flags, but address
  assignment, default routes, DHCP leases, and service status belong to the
  networking service.
- Add `BUNIX_NET_INTERFACE_SET_FLAGS` for administrative up/down state.  It
  takes interface id, set mask, clear mask, and flags; NET0 preserves physical
  `RUNNING` state from the driver and exposes the merged result through
  interface info/procfs.
- Add address records independent from `struct net_interface`:
  family, address hi/lo, prefix length, interface id, flags, preferred lifetime,
  valid lifetime.  Loopback starts with static `127.0.0.1/8` and `::1/128`
  records.  New RPCs: `BUNIX_NET_ADDR_ADD`, `BUNIX_NET_ADDR_DELETE`,
  `BUNIX_NET_ADDR_COUNT`, and `BUNIX_NET_ADDR_AT`.
- Extend route management with `BUNIX_NET_ROUTE_DELETE` and define
  `BUNIX_NET_ROUTE_ADD` as idempotent for identical route records.  Route
  records retain family, prefix, prefix length, interface id, gateway,
  flags, and metric.
- Add neighbor observation/control records for ARP/ND:
  `BUNIX_NET_NEIGHBOR_COUNT`, `BUNIX_NET_NEIGHBOR_AT`,
  `BUNIX_NET_NEIGHBOR_ADD`, and `BUNIX_NET_NEIGHBOR_DELETE`.  The first
  DHCP/IPv4 slice can expose an empty table plus manual add/delete until ARP
  resolution is implemented in the external-smoke exploration.
- Add `BUNIX_NET_DHCP4_LEASE_INSTALL` as a convenience RPC for the networking
  service.  It carries interface id, IPv4 address, prefix length or netmask,
  gateway, DNS server list metadata, lease lifetime, renewal time, and server
  identifier.  NET0 validates the interface and installs the address plus
  default route atomically.
- Add `BUNIX_NET_CONFIG_STATUS` for bounded diagnostics.  NET0 reports whether
  loopback is configured, whether each packet interface has an address,
  whether a default route exists, and the last configuration error code.  The
  networking service should log config parsing, DHCP, route, and neighbor
  progress separately.

Implementation progress:

- Added the initial NET0 configuration RPC surface in `user/include/bunix/syscall.h`
  and `user/net/main.c`: administrative interface flags, address
  add/delete/enumeration, idempotent route add, route delete, DHCPv4 lease
  install, neighbor count stub, and config status.
- NET0 now treats assigned address records as the source of local address
  truth.  The static loopback addresses remain present at boot, while future
  DHCP/service configuration can add packet-interface addresses dynamically.
- Bootstrap's NET0 self-test now verifies address add/enumeration and config
  status in addition to the existing packet interface and route paths.
- DHCPv4 lease install currently installs the IPv4 address and default route
  atomically.  DNS metadata is accepted in the lease record but still needs a
  persistence/observation path before the DHCP task can be considered complete.
- Added an initial `netcfg.server` userspace service.  Bootstrap launches it
  after unionfs mounts `/`, waits for the `NETC` service registration, and then
  continues into Linux init.  `netcfg` reads `/etc/network/interfaces`,
  idempotently configures loopback through NET0, reports status through logs,
  and treats absent `eth0` as a normal no-NIC boot.
- VirtIO-net no longer installs the QEMU-user route itself.  In
  `virtio-net-test` boots, `netcfg` finds the running VirtIO packet interface
  and installs the current deterministic QEMU-user IPv4 lease through
  `BUNIX_NET_DHCP4_LEASE_INSTALL`.  This is an interim fallback, not the final
  DHCP client.
- Bootstrap now takes its packet-interface self-test interface back down so
  netcfg does not confuse the synthetic test interface with real `eth0`.
