Bring up the BPI-F3 dual Gigabit Ethernet ports.

Status:

- Postponed.  Do not start BPI-F3-specific Ethernet work until the generic
  riscv64 port boots an Alpine environment in QEMU and exploration 070 has
  been explicitly resumed.
- Start after the first BPI-F3 boot milestone and after the generic Bunix
  networking stack has enough physical-NIC hooks to avoid hardcoding board
  policy in the kernel.

Problem:

- BPI-F3 has two GbE ports.  The generic networking explorations focus on
  protocol behavior and virtio-net, but real hardware needs MAC, PHY, MDIO,
  DMA, IRQ, and link-state support driven from firmware DT.

Tasks:

- [ ] Capture Ethernet MAC, PHY, MDIO, clock, reset, pin-control, and IRQ DT
      nodes from real hardware and the pinned Linux DT reference.
- [ ] Identify the K1 Ethernet MAC binding and whether the PHYs need board
      quirks.
- [ ] Add a userspace NIC driver server for the MAC or a hardware-specific
      backend under the planned network driver framework.
- [ ] Prove link-up detection for each port.
- [ ] Prove RX/TX packet movement with local ARP/IPv4 and IPv6 neighbor
      discovery.
- [ ] Prove DHCP through the normal Alpine networking service path.
- [ ] Prove external ping tests on each port independently.
- [ ] Expose useful link counters and errors through Bunix observability.
