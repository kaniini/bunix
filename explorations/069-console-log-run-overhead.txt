Reduce console/log overhead on interactive runs.

Parent exploration: `068-make-run-latency-baseline.txt`

Problem:

- `make run` defaults to `log=info`, and `-serial stdio` makes every emitted
  line part of the interactive critical path.
- The logger already demotes many scheduler, IPC, buffer, and task lifecycle
  lines to trace, but some expensive diagnostic paths still emit at info/error
  during normal boot.
- `arch/x86_64/user.c` negative Linux syscall dumps print multiple stack,
  byte, and register lines through `console_printf()`.  These are useful when
  debugging syscall personality bugs, but they are not acceptable as default
  run-path noise.
- Server logs should remain available through dmesg/ring-buffer diagnostics
  without slowing the controlling console once real userspace is up.

Goal:

- Make normal `make run` quiet and fast while preserving explicit debug and
  strace-style observability when requested.

Tasks:

- [ ] Use the baseline from exploration 068 to identify all high-volume log
      prefixes emitted before and after Linux userspace starts.
- [ ] Move `user-syscall:` negative syscall backtrace dumps behind an explicit
      debug/trace flag or `log=debug`, and keep a compact single-line warning
      only when needed.
- [ ] Audit info-level server logs and demote routine readiness/chatter to
      trace while keeping boot-critical milestones and errors visible.
- [ ] Ensure scheduler, IPC, buffer, VM, and VFS diagnostic prefixes are
      consistently classified by `kernel/console.c`.
- [ ] Keep `log=debug` and Linux server strace behavior available for focused
      debugging without changing default `make run`.
- [ ] Add a regression or smoke check that default run logs do not contain
      repeated high-volume diagnostic prefixes before login.
- [ ] Re-profile `make run` and record the latency/output delta.

Baseline from exploration 068:

- Default Alpine networking profile
  `build/run-profile/20260712T232619Z-17014`:
  - Serial output before poweroff: 357 lines, 17620 bytes.
  - Top prefixes: `kernel` 84, `names` 82, `vm-server` 35, `bootstrap` 14,
    `vfs` 13, `pci` 8, `netcfg` 8, `virtio-net` 7, `virtio-bus` 6.
  - Login prompt appeared at 40126 ms, while `bootstrap: linux init exec`
    appeared at 7071 ms.
- Synthetic comparison profile
  `build/run-profile/20260712T232231Z-14525`:
  - Serial output before poweroff: 420 lines, 22293 bytes.
  - Top prefixes: `names` 93, `kernel` 83, `user-syscall` 52, `vm-server` 33,
    `sched` 20.
- First read:
  - Console/log volume is real and should be reduced, especially routine
    `kernel`, `names`, `vm-server`, and `user-syscall` chatter.
  - The largest default `make run` latency gap is not explained by serial byte
    volume alone: the repeated profile showed a roughly 33s gap between Linux
    init handoff and login prompt with only 17 KB of serial output.  This
    exploration should still clean log overhead, while exploration 070 should
    treat OpenRC/login/networking startup as the primary boot critical-path
    suspect.
- Classified default Alpine networking profile
  `build/run-profile/20260712T233726Z-21452`:
  - Total serial output before poweroff: 360 lines, 17693 bytes.
  - Kernel/server bringup emitted 319 lines and 15671 bytes.  Top categories:
    `kernel` 89 lines, `names` 85, `vm` 35, `vfs-fs` 32, `networking` 17,
    `drivers-bus` 16, `sched-irq` 15, `linux-compat` 6, `ipc-buffer` 2.
  - The Linux-init-to-login slow path emitted only 19 lines and 640 bytes:
    `openrc` 14 lines, `vm` 1, `login-shell` 1, plus 3 blank lines.
  - Slow-path class totals were `guest-stdout` 15 lines / 573 bytes,
    Bunix `info` 1 line / 64 bytes, and blank `other` 3 bytes.
  - Default Alpine `log=info` did not emit `user-syscall:` in this run.
    Synthetic shell boot remains the measured `user-syscall:` offender, with
    52 lines before login in `build/run-profile/20260712T232231Z-14525`.
- Refined read:
  - This exploration should still quiet kernel/server bringup and
    `user-syscall:` dumps, because they dominate the serial volume.
  - It should not be expected to remove the entire 31-34s default login delay
    by itself: the measured slow path is mostly silent.
