Bring up an initial riscv64 port.

Problem:

- Bunix is intentionally structured with architecture directories, but the
  current build, boot, userspace ABI, syscall bridge, SMP, timers, interrupt
  handling, and QEMU test loop are all concretely x86_64.
- A riscv64 port will stress whether kernel/server boundaries are truly
  architecture-neutral and will expose x86_64 assumptions in task state,
  virtual memory, Linux compatibility, ELF loading, and device discovery.
- The initial target should be a small, testable bringup path rather than a
  full Linux-personality parity port on day one.

Initial target:

- Build a riscv64 kernel image and enough userspace server modules to boot
  under `qemu-system-riscv64`.
- Reach a minimal console log milestone first, then a native Bunix server
  milestone, then a Linux-personality/userspace milestone once syscall and ELF
  ABI work is ready.
- Prefer virt machine conventions: SBI for early console/power/timer where
  appropriate, device tree discovery, PLIC/CLINT or AIA-aware interrupt
  planning, and VirtIO MMIO before PCI.

Constraints:

- Keep architecture-specific code under `arch/riscv64/` and avoid spreading
  riscv64 conditionals through generic kernel/server code.
- Do not regress x86_64.  Existing x86_64 tests remain the baseline while the
  riscv64 target grows its own gates.
- Keep future ARM portability in mind.  Any abstraction introduced for riscv64
  should be useful for non-x86 ports generally, not just riscv64.
- Preserve the microkernel shape: userspace servers stay userspace servers;
  the port should not pull filesystem, network, or Linux policy into the
  kernel to simplify bringup.

Tasks:

- [x] Inventory all x86_64 assumptions in the build system, linker scripts,
      boot path, arch headers, task/thread state, VM, interrupt/timer code,
      syscall entry, Linux syscall bridge, userspace crt0, and test harness.
- [x] Choose the initial riscv64 boot path and machine model:

      format, memory map, and firmware expectations.
- [x] Add riscv64 toolchain variables, build rules, linker script, and
      `ARCH=riscv64` targets without disrupting the x86_64 defaults.
- [x] Add `arch/riscv64/` skeleton headers and implementations for early
      console, poweroff/reboot, interrupt enable/disable, timer source,
      thread context, page-table primitives, and user entry stubs.
- [ ] Define the riscv64 kernel virtual-memory layout, page-table flags,
      physical memory discovery, direct map policy, and userspace address
      limits.
- [ ] Implement enough boot and early memory initialization to print a kernel
      banner and power off cleanly under QEMU.
- [ ] Bring up trap/interrupt handling and a timer tick with SBI/CLINT/PLIC

- [ ] Define riscv64 Bunix syscall entry for native Bunix programs, including

- [ ] Decide the riscv64 Linux-personality ABI slice: static riscv64 musl ELF

- [ ] Add riscv64 rootfs/module packaging rules and QEMU smoke tests that end

Progress notes:

- Inventory found the first hard x86_64 dependencies:
  - The Makefile defaults `ARCH := x86_64`, `QEMU ?= qemu-system-x86_64`,
    x86_64 OVMF/EFI paths, `BOOTX64.EFI`, `MUSL_LDSO` for
    `ld-musl-x86_64.so.1`, `-m64`, `-mno-red-zone`, x86_64 linker emulation,
    and fixed x86_64 kernel/user source lists.
  - Kernel entry is Multiboot2/GRUB-specific: `kernel_main()` receives
    Multiboot2 magic/info, `pmm_init()` consumes Multiboot2 memory maps and
    modules, and boot module discovery depends on Multiboot2 module tags.
  - Generic kernel code still exposes x86_64 concepts through arch interfaces:
    LAPIC mapping in VM space creation, APIC timer/EOI/IPI hooks, ACPI RSDP
    discovery through Multiboot2, `hlt`/`cli` halt loops, CR3 logging, and
    syscall-stack setup.
  - Thread and trap state are x86_64-shaped: kernel stacks are aligned for the
    current assembly path, `arch_thread_context` stores `rsp` plus
    xsave/fxsave state, trap/syscall frames name `rip`, `rsp`, `rflags`, and
    x86 callee-saved registers.
  - Userspace startup and syscall ABI are x86_64-only: `user/crt0.S` reads
    argc/argv/envp from `%rsp`, user syscall wrappers emit `syscall`, assembly
    regression tests are x86_64, and Linux syscall numbers currently follow
    the x86_64 ABI.
  - Test harnesses assume x86_64 QEMU/OVMF/EFI/GRUB and x86_64 serial boot
    markers.  Riscv64 needs separate QEMU, firmware, marker, and packaging
    paths so x86_64 defaults remain unchanged.
- Initial riscv64 boot path is `qemu-system-riscv64 -machine virt` using the
  default OpenSBI firmware handoff into an ELF supervisor payload.  The first
  target is a direct kernel image loaded with QEMU `-kernel`, not EFI/GRUB.
  Early console and poweroff should use SBI calls first, with device-tree
  parsing introduced for memory and device discovery before VirtIO MMIO.
- Local tool availability check found no `qemu-system-riscv64`,
  `riscv64-linux-gnu-gcc`, `riscv64-alpine-linux-musl-gcc`, or
  `riscv64-unknown-elf-gcc` in `PATH`.  The next build-system slice should add
  `ARCH=riscv64` variables and clear skip/fail behavior when the riscv64 tools
  are absent.
- Added initial `ARCH=riscv64` build plumbing:
  - x86_64 remains the default `ARCH`, but kernel output is now scoped under
    `build/$(ARCH)/bunixos.kernel`.
  - x86_64 keeps its Multiboot2 source list, linker script, compiler flags,
    GRUB/EFI packaging, and test defaults.
  - riscv64 has explicit `riscv64-linux-gnu-` tool variables, `rv64gc/lp64`
    freestanding compile flags, `elf64lriscv` linker emulation, and a direct
    early kernel source list.
  - Added `arch/riscv64/linker.ld`, `arch/riscv64/boot/start.S`, and
    `arch/riscv64/early.c` as a tiny OpenSBI payload that prints an early
    banner and requests poweroff through SBI.
  - Added `run-riscv64-early` and `test-boot-riscv64-early` targets for the
    future QEMU bringup gate.
- `make ARCH=riscv64 all` now selects the riscv64 source/link path correctly
  and fails at the expected host prerequisite boundary because
  `riscv64-linux-gnu-gcc` is not installed.
- Added riscv64 arch skeleton headers and implementations:
  - `arch/riscv64/include/arch/sbi.h` centralizes the early SBI call ABI and
    console putchar helper used by the early payload and power path.
  - `interrupts.h`/`interrupts.c` define the riscv64 trap frame shape,
    interrupt enable/disable hooks, timer counters, and placeholder IRQ
    binding/mask/ack operations for later PLIC/AIA work.
  - `power.h`/`power.c` provide SBI-backed poweroff/reboot stubs.
  - `smp.h`/`smp.c` provide a single-hart implementation so scheduler-facing
    calls have a riscv64 provider before SMP bringup.
  - `thread.h`, `thread.c`, and `thread.S` define callee-saved riscv64 thread
    context, context initialization, a trampoline, and save/restore switching
    over `sp`, `ra`, and `s0`-`s11`.
  - `user.h`/`user.c` define a riscv64 syscall/trap frame and placeholder user
    entry/resume/syscall hooks for the later `ecall` ABI slice.
  - `vm.h`/`vm.c` define initial user/direct-map constants, a page-table root
    field, placeholder map/protect/unmap/translate/activate operations, and
    riscv64 PTE flag construction.
- The riscv64 skeleton intentionally keeps some x86-era generic function names
  such as `arch_smp_lapic_*` alive as compatibility shims.  A later portability
  cleanup should rename those generic contracts around CPU interrupt
  controller behavior instead of APIC-specific terminology.

Open design notes:

- The boot-info interface needs to be abstracted before real riscv64 server
  bringup.  Multiboot2 can remain the x86_64 provider, while riscv64 should
  provide equivalent command line, memory map, module list, and firmware data
  from OpenSBI/device tree or a small boot package.
- The native Bunix syscall ABI should stay negative-numbered and semantic, but
  the inline syscall mechanism must become arch-specific so riscv64 can use
  `ecall` while x86_64 keeps `syscall`.
- Linux personality support for riscv64 must not reuse x86_64 syscall numbers
  or signal frame layouts.  The first Linux slice should be a riscv64 static
  musl hello once kernel threads, user entry, and native server execution are
  proven.
