Bring up an initial riscv64 port.

Problem:

- Bunix is intentionally structured with architecture directories, but the
  current build, boot, userspace ABI, syscall bridge, SMP, timers, interrupt
  handling, and QEMU test loop are all concretely x86_64.
- A riscv64 port will stress whether kernel/server boundaries are truly
  architecture-neutral and will expose x86_64 assumptions in task state,
  virtual memory, Linux compatibility, ELF loading, and device discovery.
- The initial target should be a small, testable bringup path rather than a
  full Linux-personality parity port on day one.

Initial target:

- Build a riscv64 kernel image and enough userspace server modules to boot
  under `qemu-system-riscv64`.
- Reach a minimal console log milestone first, then a native Bunix server
  milestone, then a Linux-personality/userspace milestone once syscall and ELF
  ABI work is ready.
- Prefer virt machine conventions: SBI for early console/power/timer where
  appropriate, device tree discovery, PLIC/CLINT or AIA-aware interrupt
  planning, and VirtIO MMIO before PCI.

Constraints:

- Keep architecture-specific code under `arch/riscv64/` and avoid spreading
  riscv64 conditionals through generic kernel/server code.
- Do not regress x86_64.  Existing x86_64 tests remain the baseline while the
  riscv64 target grows its own gates.
- Keep future ARM portability in mind.  Any abstraction introduced for riscv64
  should be useful for non-x86 ports generally, not just riscv64.
- Preserve the microkernel shape: userspace servers stay userspace servers;
  the port should not pull filesystem, network, or Linux policy into the
  kernel to simplify bringup.

Progress notes:

- Inventory found the first hard x86_64 dependencies:
  - The Makefile defaults `ARCH := x86_64`, `QEMU ?= qemu-system-x86_64`,
    x86_64 OVMF/EFI paths, `BOOTX64.EFI`, `MUSL_LDSO` for
    `ld-musl-x86_64.so.1`, `-m64`, `-mno-red-zone`, x86_64 linker emulation,
    and fixed x86_64 kernel/user source lists.
  - Kernel entry is Multiboot2/GRUB-specific: `kernel_main()` receives
    Multiboot2 magic/info, `pmm_init()` consumes Multiboot2 memory maps and
    modules, and boot module discovery depends on Multiboot2 module tags.
  - Generic kernel code still exposes x86_64 concepts through arch interfaces:
    LAPIC mapping in VM space creation, APIC timer/EOI/IPI hooks, ACPI RSDP
    discovery through Multiboot2, `hlt`/`cli` halt loops, CR3 logging, and
    syscall-stack setup.
  - Thread and trap state are x86_64-shaped: kernel stacks are aligned for the
    current assembly path, `arch_thread_context` stores `rsp` plus
    xsave/fxsave state, trap/syscall frames name `rip`, `rsp`, `rflags`, and
    x86 callee-saved registers.
  - Userspace startup and syscall ABI are x86_64-only: `user/crt0.S` reads
    argc/argv/envp from `%rsp`, user syscall wrappers emit `syscall`, assembly
    regression tests are x86_64, and Linux syscall numbers currently follow
    the x86_64 ABI.
  - Test harnesses assume x86_64 QEMU/OVMF/EFI/GRUB and x86_64 serial boot
    markers.  Riscv64 needs separate QEMU, firmware, marker, and packaging
    paths so x86_64 defaults remain unchanged.
- Initial riscv64 boot path is `qemu-system-riscv64 -machine virt` using the
  default OpenSBI firmware handoff into an ELF supervisor payload.  The first
  target is a direct kernel image loaded with QEMU `-kernel`, not EFI/GRUB.
  Early console and poweroff should use SBI calls first, with device-tree
  parsing introduced for memory and device discovery before VirtIO MMIO.

  `ARCH=riscv64` variables and clear skip/fail behavior when the riscv64 tools

- After installing riscv64 system QEMU and the Alpine riscv64 linker, the
  early riscv64 kernel links and boots under QEMU.  `test-boot-riscv64-early`
  now verifies the OpenSBI direct kernel path prints
  `bunixos: riscv64 early bootstrap`, prints `machine: poweroff`, and exits
  via guest poweroff.
- Tightened `arch/riscv64/linker.ld` with explicit RX/R/RW program headers so
  the riscv64 kernel no longer links with a RWX LOAD segment.
- Added the first riscv64 supervisor trap/timer path:
  - `arch/riscv64/trap.S` installs a supervisor trap entry that saves general
    registers, snapshots `scause`, `stval`, `sepc`, `sstatus`, and interrupted
    `sp`, calls `arch_interrupt_dispatch()`, restores registers, and returns
    through `sret`.
  - `arch_interrupts_init()` programs `stvec` and enables supervisor timer
    interrupts in `sie`; `arch_interrupts_enable()`/`disable()` control
    `sstatus.SIE`.
  - `riscv64_timer_set_relative()` programs the next timer through SBI
    `set_timer`, and the dispatcher handles supervisor timer interrupts by
    incrementing the timer tick counter and disarming the timer.
  - The early boot path now schedules a short timer interrupt, waits in `wfi`
    until `arch_timer_ticks()` advances, prints `timer: riscv64 tick`, and
    then powers off.
  - The first QEMU attempt exposed a real low-level bug: `riscv64_trap_entry`
    was emitted at a 2-byte-aligned address, which corrupted the low `stvec`
    mode bits.  Aligning the trap entry to a 4-byte boundary fixed timer
    delivery.
- Proved the riscv64 kernel-thread context switch primitive in early boot:
  - The early path now initializes a boot context and worker context with
    `arch_thread_context_init_current()` and `arch_thread_context_init()`.
  - It switches to a worker stack through `arch_thread_switch()`, the worker
    records progress, and it switches back to the boot context.
  - `test-boot-riscv64-early` now requires the `thread: riscv64 switch`
    marker before poweroff.
  - This covers the kernel-thread context-switch half of the context/userspace
    task; native riscv64 userspace server execution remains open.
- Added a compile/link smoke for the riscv64 native Bunix userspace ABI:
  - `user/include/bunix/syscall.h` now emits rv64 `ecall` wrappers for native
    Bunix syscalls while preserving the x86_64 `syscall` path.
  - The initial rv64 ABI passes the negative Bunix syscall number in `a7`,
    arguments in `a0` through `a3`, and returns the result in `a0`.
  - `user/crt0-riscv64.S` derives `argc`, `argv`, and `envp` from the initial
    user stack, calls `main`, and exits through `BUNIX_SYSCALL_EXIT`.
  - `test-riscv64-user-abi` builds a freestanding riscv64 user ELF and checks
    that the resulting module is a RISC-V binary.
  - This only proves the userspace object ABI at build time; supervisor trap
    dispatch for user `ecall`, user page-table activation, and runtime native
    server execution remain open.
- Defined the riscv64 native syscall entry contract:
  - `arch/riscv64/include/arch/user.h` documents the rv64 native ABI:
    negative Bunix syscall number in `a7`, arguments in `a0` through `a3`,
    return value in `a0`, and `sepc` advanced past the 4-byte `ecall`.
  - `arch/riscv64/trap.S` now reloads `sepc` and `sstatus` after C trap
    dispatch so exception handlers can safely advance or adjust the return
    frame.
  - `arch/riscv64/include/arch/interrupts.h` now exposes the full saved
    register frame to C, matching the assembly save area.
  - `arch_interrupt_dispatch()` recognizes user and supervisor `ecall` causes,
    builds an `arch_syscall_frame`, dispatches through
    `arch_syscall_dispatch()`, writes the return value to saved `a0`, and
    advances `sepc`.
  - The riscv64 early boot smoke now runs a synthetic user-`ecall` frame test
    and prints `syscall: riscv64 ecall` when frame decoding, return placement,
    and `sepc` advancement behave as expected.
  - This completes the syscall-entry contract, but native userspace task
    launch, page-table activation, and full native syscall implementation are
    still part of the open native server execution task.
- Documented the riscv64 Linux-personality slice in `docs/riscv64-port.md`:
  - The first Linux target is a static rv64 musl ELF for
    `user/musl-hello/main.c`, loaded through the proc/VFS/server path once
    native riscv64 user task launch exists.
  - The Linux personality must use riscv64 Linux syscall numbers for
    non-negative syscall IDs, not the existing x86_64 Linux syscall table.
  - The initial syscall surface should be only what static musl hello needs:
    process exit, write, memory setup such as `brk`/`mmap` if required by
    musl startup, aux vector setup, and the minimal file/proc reads already
    exercised by the hello test.
  - Dynamic linking, BusyBox, Alpine rootfs parity, and broader Linux syscall
    coverage are explicitly later riscv64 slices.
  - The host currently has riscv64 binutils and QEMU but no riscv64 musl
    compiler wrapper or libc headers/libraries in the cross sysroot, so the
    tree should not grow a fake riscv64 musl target until those inputs can
    produce a real libc-linked ELF.
- Documented unsupported riscv64 features in `docs/riscv64-port.md`,
  including SMP, FPU/vector state, signal context, native U-mode task entry
  with kernel-stack trap transitions, user page tables, Linux signal/syscall
  parity, dynamic linking, VirtIO MMIO, networking, Alpine rootfs boot, PCI,
  USB, ACPI, board enumeration, and external interrupt routing.
- Added riscv64 arch skeleton headers and implementations:
  - `arch/riscv64/include/arch/sbi.h` centralizes the early SBI call ABI and
    console putchar helper used by the early payload and power path.
  - `interrupts.h`/`interrupts.c` define the riscv64 trap frame shape,
    interrupt enable/disable hooks, timer counters, and placeholder IRQ
    binding/mask/ack operations for later PLIC/AIA work.
  - `power.h`/`power.c` provide SBI-backed poweroff/reboot stubs.
  - `smp.h`/`smp.c` provide a single-hart implementation so scheduler-facing
    calls have a riscv64 provider before SMP bringup.
  - `thread.h`, `thread.c`, and `thread.S` define callee-saved riscv64 thread
    context, context initialization, a trampoline, and save/restore switching
    over `sp`, `ra`, and `s0`-`s11`.
  - `user.h`/`user.c` define a riscv64 syscall/trap frame and placeholder user
    entry/resume/syscall hooks for the later `ecall` ABI slice.
  - `vm.h`/`vm.c` define initial user/direct-map constants, a page-table root
    field, placeholder map/protect/unmap/translate/activate operations, and
    riscv64 PTE flag construction.
- The riscv64 skeleton intentionally keeps some x86-era generic function names
  such as `arch_smp_lapic_*` alive as compatibility shims.  A later portability
  cleanup should rename those generic contracts around CPU interrupt
  controller behavior instead of APIC-specific terminology.
- Defined the first riscv64 virtual-memory and boot-info contract:
  - `arch/riscv64/include/arch/layout.h` names the initial Sv39-era layout:
    QEMU virt physical memory at `0x80000000`, direct kernel load at
    `0x80200000`, user range `0x400000` through `0x4000000000`, a high
    direct-map window, and a separate high MMIO window.
  - The layout header centralizes riscv64 PTE flags and helper predicates for
    direct-map and user-address policy.
  - `arch/riscv64/linker.ld` documents that its load address must match
    `RISCV64_KERNEL_LOAD_BASE`.
  - `arch/riscv64/include/arch/boot.h` defines the initial boot-info record
    sourced from OpenSBI handoff state: hart ID in `a0`, FDT pointer in `a1`,
    physical base, kernel load base, direct-map policy, and user limits.
  - `arch/riscv64/early.c` preserves that handoff information for later FDT
    memory/device discovery.
  - Real physical memory enumeration is intentionally not implemented yet; the
    next boot/memory slice should parse the FDT memory node instead of assuming
    a fixed RAM size.
- Added a minimal FDT memory scanner for the next boot/memory slice:
  - `arch/riscv64/include/arch/fdt.h` exposes a bounded memory-range scan API.
  - `arch/riscv64/fdt.c` walks the FDT structure block, decodes big-endian
    cells, tracks root `#address-cells`/`#size-cells`, and extracts `reg`
    ranges from `memory` nodes.
  - `arch/riscv64/early.c` now records the first FDT memory range in boot
    info, falling back to the QEMU virt physical base if the FDT is absent or
    invalid.
  - This is enough structure for early PMM bringup to consume real firmware
    memory data later, but the QEMU boot/poweroff milestone remains unverified
    until riscv64 tools are available.

Open design notes:

- The boot-info interface needs to be abstracted before real riscv64 server
  bringup.  Multiboot2 can remain the x86_64 provider, while riscv64 should
  provide equivalent command line, memory map, module list, and firmware data
  from OpenSBI/device tree or a small boot package.
- The native Bunix syscall ABI should stay negative-numbered and semantic, but
  the inline syscall mechanism must become arch-specific so riscv64 can use
  `ecall` while x86_64 keeps `syscall`.
- Linux personality support for riscv64 must not reuse x86_64 syscall numbers
  or signal frame layouts.  The first Linux slice should be a riscv64 static
  musl hello once kernel threads, user entry, and native server execution are
  proven.
