Refactor the Linux personality so riscv64 uses the shared userspace server.

Status:

- Pending.
- Start after `035-riscv64-cleanup-alpine-environment.txt` has completed the
  initial riscv64 cleanup/audit work needed to keep the port testable.

Problem:

- `user/riscv64-linux/main.c` is a bringup-only server that handles only the
  static musl hello syscalls and powers off after `exit_group`.
- The real Linux personality lives in `user/linux/main.c`, but it currently
  assumes the mature x86_64 syscall frontend and x86_64 server build path.
- The real Linux personality also registers through the names service, while
  the current riscv64 smoke path launches only `abi-smoke.user`, `linux`, and
  `/bin/musl-hello`.  The shared-server slice therefore needs a minimal
  riscv64 service graph or an explicit temporary registration bridge, not just
  a source-file swap.
- Alpine-on-riscv64 needs the real Linux process, VFS, TTY, signal, wait,
  exec, mmap, and credential semantics.  Reimplementing those in a second
  riscv64 server would create the wrong architecture boundary.

Constraints:

- Keep the static riscv64 musl hello smoke deterministic while the full
  riscv64 Alpine boot graph remains future work.
- Do not add more semantics to `user/riscv64-linux/main.c`; retire it once the
  shared server can service the smoke path.
- Do not fold the broad x86/riscv64 DRYness pass into this exploration.  Any
  cleanup that is not required for the shared Linux personality belongs in
  `039-x86-riscv64-dryness-refactor.txt` or a later focused exploration.

Progress notes:

- Created from the audit in
  `035-riscv64-cleanup-alpine-environment.txt`.
- Follow-up scan during exploration 035 found the main replacement blocker:
  `user/linux/main.c` calls `register_service(BUNIX_SERVICE_LINUX)`, which
  talks to `names`, but `user/riscv64-bootstrap/main.c` currently launches no
  names server.  The old riscv64 stub works because the kernel finds the
  module task's private port by the name `linux`.  The shared-server slice
  must make service registration real on riscv64 before the action protocol
  can be deleted.
- Exploration 035 has since staged the first service-registration dependency:
  the main riscv64 boot package launches shared `names` and shared `user`
  servers before the temporary riscv64 Linux stub.  This exploration still
  owns replacing the `linux` module with the shared Linux personality and
  translating riscv64 Linux syscalls into the shared `BUNIX_LINUX_*` protocol;
  it should not re-open the broad boot-package refactor except where required
  to make the shared Linux server become usable.
- Exploration 035 has replaced the default static-musl boot path's `linux`
  module with the shared `user/linux/main.c` build and added a narrow riscv64
  syscall frontend for the observed static hello surface.  Remaining work here
  is to broaden the table beyond static hello, retire the UART package's old
  stub dependency, delete the `BUNIX_LINUX_RISCV64_*` action protocol after no
  path uses it, and add syscall-heavy/dynamic-loader runtime coverage.
- Exploration 035 later added dynamic-loader runtime coverage for
  `/bin/dyn-hello` through proc, VFS, squashfs, and the shared Linux server.
  The riscv64 frontend now forwards the observed musl-loader calls directly
  enough for smoke coverage, including `openat(2)` and `fstat(2)`.  This
  exploration should treat that as a working baseline and refactor it into a
  real riscv64 syscall ABI table rather than adding more one-off switch cases.
