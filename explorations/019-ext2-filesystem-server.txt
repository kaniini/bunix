Implement ext2 as a filesystem server.

Problem:

- Bunix needs a real disk-backed filesystem path beyond rootfs/tmpfs/unionfs
  scaffolding.
- ext2 is a small enough on-disk format to bring up as an initial persistent
  filesystem server and exercise the VFS/block-device boundary.

Constraints:

- ext2 should live in userspace as a filesystem server mounted through VFS.
- The server should consume a block-device capability rather than assume a
  specific storage backend.  It should work with the current block server first
  and later with VirtIO block.
- Keep the initial implementation conservative: correctness and clear failure
  modes before journaling-like behavior or aggressive caching.
- Do not put ext2 parsing policy in the kernel.

Interface inventory:

- VFS translators are mounted by sending `BUNIX_VFS_MOUNT_BUFFER` to the VFS
  service with a path buffer and a service FourCC in `words[1]`.  VFS resolves
  the service through names, records the translator path, stores the filesystem
  type as that FourCC, and forwards path operations to the longest matching
  mount.
- Existing filesystem servers register their own service FourCC, then mount
  themselves through VFS.  `tmpfs.server` is the writable path-operation model;
  `rootfs.server` is the readonly block-backed model; `unionfs.server` shows
  route pinning and mount layering, but ext2 does not need that for the first
  slice.
- Translator path operations use caller-owned buffers:
  `OPEN_BUFFER`, `STAT_PATH_META_BUFFER`, `READLINK_BUFFER`, `ACCESS_BUFFER`,
  `CREATE_BUFFER`, `MKDIR_BUFFER`, `MKNOD_BUFFER`, `SYMLINK_BUFFER`,
  `RENAME_BUFFER`, `LINK_BUFFER`, `UNLINK_BUFFER`, and `RMDIR_BUFFER`.
  Open-handle operations use server-local handles returned by `OPEN_BUFFER`:
  `STAT_META`, `READ_FILE_BUFFER`, `WRITE_FILE_BUFFER`, `TRUNCATE`, `CHMOD`,
  `CHOWN`, `READDIR_BUFFER`, and `CLOSE`.
- Stat metadata is the shared `BUNIX_VFS_STAT_BYTES` record written with
  `bunix_vfs_stat_pack()`: size, mode plus VFS type, uid/gid, device id,
  inode, link count, rdev, block size, and 512-byte block count.
- `READDIR_BUFFER` returns one directory entry per call.  `words[1]` contains
  the next cookie in the low 32 bits and the VFS file type in the high 32 bits;
  `words[2]` is the full name length; `words[3]` is the byte count written to
  the caller buffer.
- VFS mount routes can be resolved or pinned through
  `RESOLVE_MOUNT_BUFFER`, `PIN_ROUTE_BUFFER`, and `UNPIN_ROUTE`.  That is
  useful for future layered filesystems and block-capability handoff, but the
  first ext2 server can use the direct names service/block service path.
- The current block protocol is byte-offset based.  `GET_INFO` returns total
  backing bytes in `words[1]` and an advertised granularity in `words[2]`;
  `READ` returns small inline payloads; `READ_BUFFER` copies up to 128 KiB into
  a supplied buffer capability.  It is transport-neutral enough for ext2 to
  consume the current block server first and a VirtIO block-backed service
  later.

Initial ext2 mount contract:

- Add an `EXT2` FourCC service and userspace `ext2.server`.  The server owns
  ext2 parsing and exposes only the normal VFS translator protocol to clients.
- The first implementation mounts readonly.  Mutating VFS operations must fail
  with access/invalid errors until explicit write-support slices implement
  allocation, metadata updates, and fsck-compatible persistence.
- The server consumes an explicit block-device capability.  For the first
  boot/test path it may resolve `BUNIX_SERVICE_BLOCK`; the server API should
  still keep the block handle in per-mount state so tests and later VirtIO
  block mounts can supply a different handle without changing VFS behavior.
- A mount request validates the superblock before registering the VFS
  translator.  Mount state records the VFS mount path, block handle, readonly
  flag, device id, superblock fields, block-group table, root inode, and any
  feature diagnostics needed for logs/tests.
- Support ext2 block sizes 1024, 2048, and 4096 initially.  Reject unsupported
  block sizes, non-power-of-two values, impossible image sizes, bad group
  counts, corrupt inode sizes, and on-disk offsets that overflow the backing
  block-device byte length.
- Root inode is ext2 inode 2.  Directory lookup starts from that inode and maps
  ext2 file types to Bunix VFS types for regular files, directories, symlinks,
  FIFOs, and character devices.
- Feature policy for the readonly slice: accept only ext2 revisions and
  compatible/readonly-compatible features that are explicitly decoded.  Reject
  unsupported incompat features and unsupported ro-compat features with clear
  diagnostics rather than silently mounting a filesystem whose metadata cannot
  be interpreted correctly.
- Architecture and kernel code remain uninvolved.  Kernel faults, page tables,
  and block-device discovery are outside the ext2 parser; the only kernel-facing
  behavior is ordinary userspace IPC and buffer copying.

Implementation notes:

- Added `BUNIX_PROTO_EXT2`, `BUNIX_SERVICE_EXT2`, `BUNIX_VFS_DEV_EXT2`, and a
  buildable `ext2.server` userspace module.
- The initial server resolves the block service, reads the ext2 superblock at
  byte offset 1024 through `BUNIX_BLOCK_READ_BUFFER`, validates ext2 magic,
  revision, block size, image bounds, group geometry, inode size, and feature
  masks, then logs either a concise mount rejection diagnostic or accepted
  geometry.
- Readonly-safe feature support currently accepts common generated ext2 image
  bits: `filetype`, `sparse_super`, `large_file`, `ext_attr`, `resize_inode`,
  and `dir_index`.  Journaling/recovery, compression, meta block groups, and
  other undecoded features remain rejected.
- Block group descriptors are loaded from the ext2 descriptor table with bounds
  checks for descriptor-table size, bitmap block numbers, inode-table ranges,
  and byte-offset overflow.  The server has bounded inode reads and a generic
  bitmap-bit helper, currently used to verify that root inode 2 is allocated
  before accepting the mount state.
- Directory-entry parsing now has checked direct-block iteration, name-length
  and record-length validation, ext2-to-VFS type mapping, and root `"."`
  lookup validation.  The remaining directory task work is wiring those helpers
  into persistent mount state and the VFS `OPEN_BUFFER`/`READDIR_BUFFER`
  translator operations.
- Directory lookup and readdir are wired into the VFS translator surface for
  the initial readonly ext2 mount state: path lookup walks directory entries
  from inode 2, `OPEN_BUFFER` returns server-local handles, `READDIR_BUFFER`
  returns one non-dot entry per cookie, and `CLOSE` releases open handles.
  This currently uses direct directory blocks; indirect directory block support
  will come with the general file/data-block mapping task.
- File data reads now use a shared ext2 logical-block mapper that supports
  direct, singly indirect, doubly indirect, and triply indirect blocks with
  sparse block handling.  `READ_FILE_BUFFER` streams regular-file contents into
  the caller buffer and directory iteration uses the same mapper, so large
  directories can traverse indirect blocks too.
- The VFS stat record now carries atime, mtime, and ctime seconds in addition
  to the original identity fields.  Ext2 `STAT_PATH_META_BUFFER` and
  `STAT_META` replies map inode number, mode/type, uid/gid, size, link count,
  512-byte block count, device identity, rdev, block size, and timestamps into
  Linux stat/statx through the Linux server.  Filesystems that use the older
  stat helper continue to emit zero timestamps.
- Readonly boot coverage now builds a generated ext2 image as a Multiboot2
  data module named `ext2disk`, launches `ext2.server` only for the
  `ext2-test` command-line mode, mounts it at `/mnt/ext2`, and exercises
  regular-file reads, VFS symlink following, hardlink inode/link metadata,
  long-name readdir, and sparse-ish file size metadata before continuing to the
  normal login path.  The fixture is intentionally small enough to avoid GRUB
  standalone relocator pressure while still being verified by host `e2fsck`.
- Write-support groundwork now extends the generic block protocol with
  `WRITE_BUFFER` and `FLUSH`.  The boot-module-backed `block.server` remains
  explicitly readonly, while `virtio-blk.server` implements byte-range writes
  with sector read-modify-write for unaligned ranges and exposes flush when
  negotiated.  Ext2 metadata mutation should build on that writable block
  capability rather than writing to Multiboot2 module memory.
- The first ext2 mutating operation is in-place regular-file writes through
  `WRITE_FILE_BUFFER`.  It updates existing mapped data blocks only, refuses
  file growth/allocation for now, and uses a mutable server-owned copy of the
  generated boot test image so the ext2 boot test can prove VFS write
  forwarding and hardlink-visible data updates without claiming persistence.
- Shrink-only `TRUNCATE` is now implemented for open regular-file handles.  It
  updates inode size metadata and is covered through the same hardlink-visible
  ext2 boot selftest, but it intentionally does not allocate blocks for growth
  or free blocks on shrink yet.
