Map out and implement the VirtIO bus, VirtIO block, and VirtIO timer path.

Problem:

- Bunix currently relies on narrow bringup devices and ad hoc platform
  surfaces.  To run more realistic guests and root filesystems, it needs a
  proper VirtIO foundation.
- The immediate targets are VirtIO bus enumeration, a VirtIO block driver, and
  a timer source suitable for portable scheduling/time work.

Constraints:

- Keep device drivers in userspace where practical.  Kernel involvement should
  be limited to resource discovery, capability-scoped hardware access, DMA
  safety, interrupts, and task/VM primitives.
- Design the bus layer around transports, not only QEMU x86.  PCI/MMIO VirtIO
  should be separable so ARM MMIO VirtIO can be added later.
- Do not hand arbitrary I/O or DMA authority to drivers.  Capabilities should
  attenuate access to specific device BAR/MMIO ranges, IRQs, and DMA buffers.
- Keep block-device APIs compatible with the existing block/VFS direction so
  rootfs mounting can move from static module images to device-backed storage.

Tasks:

- [x] Inventory current block, timer, interrupt, PCI/platform discovery, and
      hardware-access paths.
- [x] Define a generic device-resource model for userspace drivers: MMIO/PIO
      ranges, IRQs, DMA windows, feature bits, queue memory, and reset.
- [x] Implement or plan the required VirtIO transport layer, starting with PCI
      on x86_64 and leaving a clean MMIO transport path for ARM.
- [ ] Implement VirtIO feature negotiation, status transitions, queue setup,
      descriptor rings, avail/used rings, notifications, and interrupt acks.
- [ ] Add a userspace VirtIO bus manager/server that discovers devices and
      grants attenuated device capabilities to matching drivers.
- [ ] Implement a VirtIO block userspace driver with read/write/flush support,
      request queueing, completion handling, and error translation.
- [ ] Connect VirtIO block to the existing block server or replace the current
      block backend behind the same VFS-facing protocol.
- [ ] Define the VirtIO timer requirement precisely.  If standard VirtIO does
      not provide the desired timer device, decide whether the right target is
      virtio-rng/console, HPET/APIC timer abstraction, or a platform timer
      server that shares the same driver framework.
- [x] Add QEMU run/test modes that boot with VirtIO block and deterministic
      device topology.
- [ ] Add tests for bus enumeration, feature negotiation failures, block reads,
      block writes, flushes, interrupt-driven completions, and clean poweroff.
- [ ] Document the ARM porting shape for VirtIO MMIO transport and timer
      selection.

Inventory:

- Block storage is currently a boot-module-backed userspace `block.server`.
  `kernel/server.c` records the GRUB `disk0` module and attaches its copied
  data range to the `block` module start record.  `user/block/main.c` exposes
  `BLK0` `GET_INFO`, small inline `READ`, and `READ_BUFFER` RPCs by copying
  from that boot module through `bunix_boot_module_read()`.
- The block protocol is read-only today.  There are no write, flush, discard,
  capacity-change, or completion/status operations, so VirtIO block needs either
  protocol extensions behind the existing VFS-facing service name or a new
  block-device protocol that `block.server` adapts.
- `user/rootfs/main.c` and `user/vfs/main.c` consume `BUNIX_SERVICE_BLOCK`
  synchronously.  They assume reads complete during the RPC call, and rootfs
  mounting depends on offset-addressed byte reads from one root block service.
- Timekeeping is kernel-owned.  `arch/x86_64/interrupts.c` programs PIT channel
  0 at 100 Hz, increments a global tick counter on vector 32, wakes sleepers,
  and invokes preemption from user-mode interrupt returns.  APs also receive
  periodic LAPIC timer interrupts on vector 65, but the global monotonic clock
  still comes from the PIT tick counter.
- The `time.server` is a thin userspace facade over native clock/sleep syscalls.
  It does not own timer hardware, deadlines, clocksource selection, or interrupt
  routing.
- Interrupt handling is fixed and architecture-local.  The IDT installs CPU
  exception stubs plus vectors 32, 64, and 65.  There is no generic IRQ
  allocator, no userspace IRQ bind/ack/mask implementation, no IOAPIC/MSI/MSI-X
  routing, and no event delivery path for device interrupts yet.
- ACPI parsing currently exists only for SMP MADT discovery in
  `arch/x86_64/smp.c`.  There is no PCI/PCIe/ECAM scanner, no PCI config-space
  access, no BAR discovery, no VirtIO vendor/device probing, and no platform
  bus manager.
- Hardware capability plumbing exists from the userspace console work:
  `TASK_HW_RESOURCE_PORT`, `TASK_HW_RESOURCE_MMIO`, and `TASK_HW_RESOURCE_IRQ`
  handles carry base/length/op masks and can be inherited/attenuated through
  task handles.  Only 8-bit PIO read/write syscalls are implemented, and the
  only boot-time grant is a static COM1 port range for `consoled`.
- DMA has no explicit resource model yet.  Drivers can receive normal shared
  buffer handles, but there is no DMA window object, physical-contiguous queue
  allocation contract, bounce-buffer support, or kernel validation tying device
  DMA to a driver-owned capability.
- QEMU run and test modes use `-machine q35`, KVM, OVMF pflash, and a FAT ESP
  drive containing GRUB modules.  No `virtio-blk`, `virtio-mmio`, deterministic
  PCI topology, or device-specific test mode exists yet.

Progress:

- Added a shared `DEV0` protocol/service namespace for generic device-bus RPCs:
  enumeration, device info, resource lookup, driver binding, reset, feature
  read/negotiation, queue setup/notification, and interrupt ack.
- Defined generic resource kinds for PIO, MMIO, IRQ, DMA windows, queue memory,
  feature bits, and reset authority, plus transport IDs for platform, PCI, and
  MMIO transports.
- Added driver-side descriptors for device identity, device resources, DMA
  windows, queue memory layout, feature sets, and whole devices.  These are
  intentionally capability-oriented: handles carry authority, while base/len,
  queue indices, feature masks, and alignments describe what that authority
  permits.
- This task defines the userspace contract only.  IRQ binding/delivery, DMA
  allocation/validation, PCI discovery, and VirtIO queue mechanics remain open
  implementation tasks in this exploration.
- Added a shared VirtIO transport header with PCI vendor/device constants,
  device status bits, common feature bits, packed virtqueue descriptor/avail/used
  ring layouts, queue layout descriptors, and transport/device-info records.
- Transport plan: build a userspace `virtio-bus` server that first discovers
  modern VirtIO PCI devices on x86_64, maps common/notify/isr/device config
  resources into `DEV0` descriptors, and hands attenuated resource capabilities
  to class drivers.  Keep the class-driver side transport-neutral by consuming
  `struct bunix_virtio_transport` records rather than hard-coding PCI config
  details.
- PCI transport plan: add a minimal x86_64 PCI config accessor and scanner,
  then match vendor `0x1af4` devices, parse modern VirtIO capabilities, expose
  BAR-backed MMIO/PIO resources, and later add MSI-X or INTx IRQ resources once
  the generic IRQ broker exists.
- MMIO transport plan: reuse the same `DEV0` and `bunix_virtio_transport`
  records with `BUNIX_DEV_TRANSPORT_MMIO`; ARM should discover platform/MMIO
  VirtIO ranges from firmware/device-tree/ACPI and grant MMIO plus IRQ resources
  without exposing PCI assumptions to block/net/timer drivers.
- Added transport-independent VirtIO helpers for feature negotiation, status
  construction, queue layout sizing, descriptor initialization, avail-ring
  submission, and used-ring indexing.  The VirtIO mechanics task remains open
  until these helpers are wired into a discovered device and real queue
  notification/completion path.
- Added an initial userspace `virtio-bus.server` skeleton registered as the
  `DEV0` service.  It boots as a normal server, reports an empty enumeration,
  and returns `NOENT` for device-specific requests until PCI/MMIO discovery is
  implemented.  Boot marker coverage now asserts `virtio-bus: ready devices=`.
- Extended capability-checked x86 port I/O from byte-only access to 16-bit and
  32-bit read/write operations.  This is required groundwork for PCI config
  access through a narrowly granted `0xcf8/0xcfc` resource instead of broad I/O
  privilege.
- Granted `virtio-bus.server` a narrow PCI config-port resource for
  `0xcf8..0xcff` and added a bus-0 PCI config scanner using the 32-bit port I/O
  helpers.  The server now dynamically records matching VirtIO PCI functions
  and reports the discovered count through `DEV0` enumeration.
- Fixed supplemental boot grants so bootstrap-launched module servers can still
  receive kernel-granted hardware resources.  A direct boot harness run with
  `-device virtio-blk-pci` now reports `virtio-bus: ready devices=1`.
- Added `run-virtio` and `test-boot-virtio` Make targets.  They attach the
  current rootfs image as a deterministic `virtio-blk-pci` device at
  `pcie.0/addr=0x6`; `test-boot-virtio` boots to poweroff and asserts
  `virtio-bus: ready devices=1`.
- Extended `virtio-bus.server` discovery to size PCI BARs, parse modern VirtIO
  vendor capabilities, store resource descriptors, and expose those descriptors
  through `DEV0 GET_RESOURCE`.  The deterministic VirtIO boot target now also
  asserts a discovered device log line with resource metadata.
- Added a kernel-mediated `hw_pci_bar_grant` native syscall for VirtIO PCI
  BAR subranges.  `virtio-bus.server` now attaches hardware-resource handles to
  whole BAR descriptors and attenuated VirtIO common/notify/ISR/device config
  subresources, while the scheduler tracks heap-backed hardware resources with
  owned refcounts so inherited or closed handles do not leak.  This is still
  only the bus resource handoff foundation; driver binding, IRQ resources, DMA
  windows, and queue ownership remain open.
- Added capability-checked native MMIO read/write syscalls for 8/16/32-bit
  accesses.  The kernel validates MMIO resource type, range, and operation bits,
  maps touched pages into kernel space on demand, and keeps userspace drivers on
  handles rather than raw I/O privilege.  This unblocks VirtIO common config,
  notify, ISR, and device config access once drivers begin consuming `DEV0`
  resources.
- Switched the deterministic QEMU VirtIO block topology to modern-only PCI with
  `disable-legacy=on`, enabled PCI command I/O/memory/bus-master bits for
  discovered VirtIO functions, and fixed 64-bit BAR base/size handling in both
  userspace discovery and the kernel BAR-grant validator.  This is required for
  modern VirtIO devices whose config BARs live above 4 GiB.
- `virtio-bus.server` now records common/notify/ISR/device config resource
  indices, reads nonzero device feature bits through the modern common config,
  exposes `DEV0 READ_FEATURES`, and implements an initial
  `DEV0 NEGOTIATE_FEATURES` path that resets status, sets ACKNOWLEDGE/DRIVER,
  writes both 32-bit driver feature banks, and verifies FEATURES_OK.  Queue
  setup, notification, interrupt ack, and driver ownership remain open.
- `test-boot-virtio` now asserts modern VirtIO enumeration and rejects a zero
  feature-read marker, giving coverage for PCI capability parsing, 64-bit BAR
  grants, and common-config MMIO reads during boot.
- Fixed IPC capability transport so hardware-resource caps are a distinct
  internal IPC cap type instead of being collapsed into task caps.  This makes
  `DEV0 GET_RESOURCE` hardware handles deliverable to drivers and preserves
  owned BAR resource lifetimes while messages are queued.  Added a
  `buffer_phys` native syscall for shared-buffer handles so upcoming VirtIO
  queue setup can program descriptor/driver/device ring physical addresses from
  a capability-held queue memory buffer.
- Added `DEV0 SETUP_QUEUE` support to `virtio-bus.server` for modern PCI
  devices.  The bus allocates a shared-buffer queue memory object, computes the
  descriptor/driver/device ring layout, programs queue size and physical ring
  addresses through common config, enables the queue, and returns the queue
  buffer capability plus layout metadata to the caller.  The server also parses
  the notify capability multiplier and implements `DEV0 NOTIFY_QUEUE` using the
  queue's notify offset.  A real block driver still needs to fill descriptors,
  consume used rings, handle interrupts, and exercise this path end-to-end.
