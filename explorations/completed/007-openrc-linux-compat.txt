Close Linux personality gaps discovered by OpenRC userspace.

Findings:

- Alpine reaches OpenRC startup and `/sbin/openrc` runs far enough for musl's
  dynamic linker to search for dependencies.
- The dynamic linker opens `/usr/lib/librc.so.1` and `/usr/lib/libeinfo.so.1`,
  but relocation still reports missing `libeinfo` symbols.
- Local musl `ldso` source shows loaded DSOs are de-duplicated with `fstat()`
  `st_dev`/`st_ino`; Bunix currently reports constant Linux stat identity for
  many files, so distinct shared objects can look identical.
- The real VFS-backed fix for `/sbin/openrc` dependency identity should
  happen in `011-vfs-stat-identity.txt`; until then, the Linux personality
  needs a narrow compatibility shim so OpenRC can keep booting.
- A later Alpine boot reaches OpenRC script startup, but script execution
  fails with `Invalid argument` for `/usr/libexec/rc/sh/init-early.sh`,
  `/usr/libexec/rc/sh/init.sh`, and `gendepends.sh`; this points at missing
  Linux `execve()` shebang handling rather than a VFS issue.
- Until `011-vfs-stat-identity.txt` lands, Linux `fstat()` must keep a
  temporary handle-derived inode identity so musl does not collapse distinct
  OpenRC DSOs such as `librc` and `libeinfo`.
- With the temporary `fstat()` identity shim restored and shebang handling in
  place, Alpine OpenRC starts, mounts `/proc`, and advances to
  `checkpath: owner 'root:uucp' not found`.
- The Alpine rootfs overlay must merge Bunix login credentials into the apk
  account database instead of replacing `/etc/passwd`, `/etc/shadow`, and
  `/etc/group`; replacing them drops system groups such as `uucp`.
- After preserving apk account entries and forwarding unionfs directory
  `chown`/`chmod` to upper handles, OpenRC passes `/run/lock` ownership
  correction and reaches a later fork/pipe supervision phase.
- Linux TTY reads must not call the blocking early console read syscall inside
  the Linux server.  A process waiting for terminal input can otherwise stop
  unrelated Linux IPC such as `wait4`, pipe wakeups, and process exits.
- OpenRC `gendepends.sh` exposed another kernel/server split bug: kernel-side
  `poll` treated numeric fd 0 as the console even after shell redirection
  duped `/etc/init.d/binfmt` onto stdin.  Linux poll readiness must be resolved
  by the Linux server against the current Linux fd table.
- OpenRC's dependency cache path uses `flock(2)`.  Advisory locking can be a
  compatibility no-op for now, but valid fds must return success so OpenRC does
  not report `Function not implemented`.
- With Linux-server poll readiness and minimal `flock(2)`, Alpine OpenRC
  caches service dependencies, reaches Bunix login, starts a root shell, and
  powers off through `/sbin/poweroff`.
- `tools/audit-linux-syscalls.sh` reports no missing arch dispatch.  Its
  remaining forwarded-without-server-case entries are compatibility rewrites:
  `stat`/`lstat` route as `newfstatat`, `symlink` routes as `symlinkat`, and
  `exit` is completed in the arch syscall path.
- The Alpine boot log no longer shows unknown syscalls, `EINVAL`,
  `ENOSYS`/`Function not implemented`, or shebang exec failures during the
  OpenRC startup path.  The remaining OpenRC warnings are missing service
  dependency references for `dev`.
- Alpine boot verification now runs BusyBox `id` and `ps` after login before
  guest poweroff, proving the root shell can launch applets after OpenRC
  dependency caching and login.
