Implement a VirtIO-net driver and attach it to the core networking stack.

Problem:

- After the core networking stack exists, Bunix needs a real paravirtual NIC
  driver so guests can communicate beyond localhost under QEMU.
- VirtIO-net should plug into the VirtIO bus framework and the networking
  stack without bypassing capability boundaries.

Dependencies:

- Requires the VirtIO bus/resource framework from
  `016-virtio-bus-block-timer.txt`.
- Requires the network interface/packet-driver boundary from
  `017-core-networking-stack.txt`.

Constraints:

- Keep the VirtIO-net driver in userspace.
- Kernel hardware authority should be limited to scoped device resources,
  DMA-safe buffers, queue notifications, and interrupt/event delivery.
- The driver should expose a packet interface to the networking stack rather
  than Linux socket semantics directly.
- Preserve a path for both PCI VirtIO on x86_64 and MMIO VirtIO for future ARM.

Tasks:

- [x] Inventory QEMU VirtIO-net device modes, feature bits, queue layout, MAC
      address configuration, and interrupt behavior.
- [x] Define the packet-device protocol between NIC drivers and the network
      interface manager.
- [x] Implement VirtIO-net feature negotiation, RX/TX virtqueues, buffer
      replenishment, transmit completion, receive delivery, and device reset.
- [x] Add support for MAC address discovery/configuration and basic link state.
- [x] Integrate the driver with the network stack interface registry and route
      management.
- [x] Add QEMU run/test modes with a deterministic VirtIO-net topology.
- [x] Add tests for driver attach, link-up, transmit path, receive path,
      localhost unaffected by NIC state, and clean shutdown.
- [x] Add external-network smoke tests once a deterministic host-side tap/user
      networking setup exists.
- [x] Document future work for checksum offload, segmentation offload,
      multiqueue, promiscuous mode, and packet filtering.

Inventory:

- The existing VirtIO foundation from exploration 016 already provides a
  userspace `virtio-bus.server` registered as `DEV0`.  It scans bus 0 PCI
  config space through a narrowly granted `0xcf8/0xcfc` PIO capability, matches
  modern VirtIO PCI functions, sizes BARs, parses common/notify/ISR/device
  config capabilities, grants attenuated BAR subresources, reads feature banks,
  negotiates driver features, allocates split-queue memory, programs queue
  physical addresses, notifies queues, acknowledges ISR status, and moves a
  device to `DRIVER_OK`.
- `virtio-blk.server` is the class-driver template to follow.  It resolves the
  `DEV0` service, locates a device by `BUNIX_VIRTIO_DEVICE_BLOCK`, negotiates
  `VERSION_1`, sets up queue 0, builds descriptor chains in a shared queue
  buffer, notifies the queue, polls used-ring completion, and optionally
  registers behind the existing block protocol.
- The current NET0 server owns loopback, route lookup, UDP/TCP socket state,
  packet counters, and `/proc/net` observability.  It does not yet have a
  driver-facing packet-interface registry, TX packet queue, RX Ethernet frame
  ingress path, MAC identity, non-loopback interface IDs, or route-management
  updates for external interfaces.
- Installed QEMU exposes `virtio-net-pci`, `virtio-net-pci-non-transitional`,
  and `virtio-net-pci-transitional`.  The deterministic Bunix target should
  follow the block target and use modern-only PCI with `disable-legacy=on`, a
  fixed PCI slot, and a fixed MAC address.  Useful QEMU properties for the
  first slice are `netdev=`, `mac=`, `disable-legacy=on`, `rx_queue_size=`,
  `tx_queue_size=`, `status=on`, `mrg_rxbuf=off`, `csum=off`, `gso=off`,
  `guest_csum=off`, `guest_tso4=off`, `guest_tso6=off`, `guest_ufo=off`,
  `host_tso4=off`, `host_tso6=off`, and `ctrl_vq=off`.
- Installed QEMU netdev backends include `user`, `passt`, `tap`, `socket`,
  `stream`, `dgram`, `bridge`, `hubport`, `vhost-user`, and `vhost-vdpa`.
  The first deterministic attach test should use `-netdev user,id=bunix-net0`
  or an isolated socket backend, then defer external-network smoke tests until
  the harness owns a stable host-side topology.
- VirtIO-net is device type 1.  The minimum modern split-queue model uses queue
  0 for receive and queue 1 for transmit.  Queue 2 is the control virtqueue
  only if `CTRL_VQ` is negotiated, and multiqueue adds queue pairs only if
  `MQ` is negotiated.  The first driver should require only `VERSION_1`, avoid
  offload features, and treat packet buffers as complete Ethernet frames
  preceded by `struct bunix_virtio_net_header`.
- MAC address discovery comes from device config when `MAC` is available or a
  deterministic locally administered fallback when it is not.  Link state comes
  from the device config `status` field when `STATUS` is negotiated; otherwise
  the driver should report link-up once queues are initialized.  Interrupts can
  initially follow the block driver's bounded used-ring polling path until
  exploration 025 provides generic IRQ-driven completions.

Packet-Device Protocol:

- `virtio-net.server` is a userspace NIC class driver.  It talks to `DEV0` for
  hardware resources and queue operations, and to `NET0` only through a
  packet-interface protocol.  It must not expose Linux socket operations, Linux
  PIDs, or protocol-stack internals.
- NET0 remains the network interface manager and protocol owner.  It assigns
  interface IDs, stores MAC/MTU/link state, owns route entries, accepts inbound
  Ethernet frames from drivers, queues outbound Ethernet frames for drivers,
  and accounts RX/TX/drop counters for `/proc/net/dev` and future Bunix-native
  tools.
- The shared NET0 driver ABI now reserves:
  `BUNIX_NET_PACKET_INTERFACE_ATTACH`,
  `BUNIX_NET_PACKET_INTERFACE_LINK`,
  `BUNIX_NET_PACKET_RX_SUBMIT`,
  `BUNIX_NET_PACKET_TX_DEQUEUE`, and
  `BUNIX_NET_PACKET_TX_COMPLETE`.
- A NIC driver attaches by sending `struct bunix_net_packet_interface_info`
  containing requested flags, MTU, and MAC address.  NET0 replies with the
  assigned interface ID and current counters.  `LINK` updates
  `UP/RUNNING/BROADCAST` state without granting route-management authority.
- RX uses `RX_SUBMIT` with a buffer containing
  `struct bunix_net_packet_info` followed by one Ethernet frame.  NET0 validates
  the interface ID, MTU, link state, and buffer length before passing the frame
  into ARP/IPv4/IPv6 handling once those paths exist.
- TX uses pull-based driver dequeue: the driver calls `TX_DEQUEUE` with an
  output buffer and NET0 either copies one queued Ethernet frame into it or
  returns an empty/not-ready status.  The driver calls `TX_COMPLETE` after the
  VirtIO transmit descriptor reaches the used ring so NET0 can update counters
  and release queued packet state.  This preserves OCAP shape because NET0 never
  calls directly into an arbitrary driver port.
- The first VirtIO-net implementation can prove attach/link and loopback
  non-regression before external packet routing exists.  Actual external
  IPv4/IPv6 traffic requires later ARP/ND, Ethernet encapsulation, route
  installation, and deterministic host networking tests.

Progress:

- Added the initial shared VirtIO-net ABI in `bunix/virtio.h`: device type 1,
  common VirtIO-net feature bits, link-status bits, packed device config, and
  packed packet header records.  The first driver should require only
  `VERSION_1`, avoid offloads, and optionally consume `MAC`/`STATUS`/`MTU`.
- Reserved NET0 packet-interface RPCs and records in `bunix/syscall.h` for
  driver attach, link updates, RX submission, TX dequeue, and TX completion.
- Implemented the first NET0 packet-interface registry.  Packet devices receive
  dynamic interface IDs starting at 2, can publish MTU/MAC/UP/RUNNING/BROADCAST
  state, can submit RX frames for accounting, and expose counters through the
  existing interface info/stats RPCs.  TX dequeue currently returns no packet
  because Ethernet encapsulation and route-backed egress queues are not in this
  slice.
- Bootstrap now self-tests the packet-interface registry by attaching a
  synthetic NIC, setting link state, submitting one RX frame, checking interface
  count/stats, and emitting `bootstrap: net packet iface ok`.
- Added deterministic QEMU VirtIO-net topology support through
  `QEMU_VIRTIO_NET_ARGS`, `run-virtio-net`, and `test-boot-virtio-net`.  The
  narrow boot test uses modern-only `virtio-net-pci` at fixed PCI slot `0x7`
  with MAC `52:54:00:18:00:01`, boots the normal server set, and verifies that
  `virtio-bus.server` enumerates one device with modern VirtIO-net PCI device
  ID `0x1041`.
- Added the first userspace `virtio-net.server` slice.  It resolves `DEV0` and
  `NET0`, finds VirtIO device type 1, negotiates `VERSION_1` with optional
  `MAC`/`STATUS`/`MTU`, sets up RX queue 0 and TX queue 1, binds the device to
  `DRIVER_OK`, reads MAC/MTU/link state from device config where available,
  and attaches a packet interface to NET0.  It is launched only by the
  dedicated `virtio-net-test` boot image for now.
- `test-boot-virtio-net` now includes `virtio-net.server` in a dedicated ESP,
  boots with a fixed VirtIO-net PCI topology, verifies device enumeration, and
  checks for `virtio-net: negotiated`, `virtio-net: queues ready`, and
  `virtio-net: attached iface=` logs.  Actual RX buffer replenishment, TX
  dequeue/completion, Ethernet encapsulation, route integration, and device
  reset handling remain open.
- `virtio-net.server` now replenishes the receive virtqueue with guest-owned
  buffers, notifies queue 0, strips the VirtIO-net header from completed
  buffers, submits completed Ethernet frames to NET0 through
  `BUNIX_NET_PACKET_RX_SUBMIT`, requeues consumed RX descriptors, and polls
  completions in a bounded loop until exploration 025 supplies IRQ-driven
  completions.  The dedicated boot test checks `virtio-net: rx ready`.  Real
  external receive testing still needs deterministic host traffic and the
  network stack still needs Ethernet/ARP/ND route integration.
- `virtio-bus.server` now implements `BUNIX_DEV_RESET` for discovered VirtIO
  devices by clearing device status, dropping queue buffers, and clearing
  negotiated feature state.  `virtio-net.server` resets its device before
  negotiation, and the dedicated boot test checks `virtio-net: reset`.
- Added driver-side TX plumbing in `virtio-net.server`: it polls NET0 with
  `BUNIX_NET_PACKET_TX_DEQUEUE`, formats dequeued frames with a zeroed
  VirtIO-net header, submits them to TX queue 1, polls the TX used ring, and
  reports completions back through `BUNIX_NET_PACKET_TX_COMPLETE`.  NET0 does
  not yet provide route-backed Ethernet egress packets, so the dedicated boot
  test currently covers TX idle readiness with `virtio-net: tx ready`.
- Added NET0 route-management and indexed interface RPCs.  Packet interfaces
  can now be discovered by index, dynamic routes can be added and enumerated,
  route lookup considers dynamic routes plus static loopback routes, and
  `/proc/net/dev` plus `/proc/net/route` are built from NET0 state instead of
  hardcoded loopback rows.  Bootstrap covers the route API with
  `bootstrap: net route ok`; `virtio-net.server` installs a deterministic
  QEMU-user `10.0.2.0/24` route for its interface and logs
  `virtio-net: route ready`.
- NET0 packet interfaces now have real transmit queues.  Protocol-stack code
  can enqueue an Ethernet frame with `BUNIX_NET_PACKET_TX_ENQUEUE`, drivers
  pull frames with `BUNIX_NET_PACKET_TX_DEQUEUE`, and `TX_COMPLETE` accounts
  completed sends.  Bootstrap now exercises enqueue, dequeue, completion, and
  TX stats using the synthetic packet interface.
- Test coverage now spans the first driver and packet-interface boundary:
  `test-boot-virtio-net` verifies deterministic device enumeration, reset,
  feature negotiation, RX/TX queue setup, NET0 attachment, route installation,
  RX readiness, TX readiness, existing localhost NET0 selftests, and clean
  harness poweroff.  Bootstrap's synthetic packet-interface selftest exercises
  RX submission plus TX enqueue/dequeue/complete and counter accounting without
  depending on host traffic.

Future Work:

- Checksum offload should stay disabled until NET0 has packet metadata that can
  describe partial checksums without exposing VirtIO-specific headers to the
  protocol stack.  The future ABI should carry checksum start/offset and an
  explicit verified/needs-completion state so non-VirtIO drivers and software
  fallback can share one path.
- Segmentation offload should be added only after TCP output owns real MSS,
  PMTU, and retransmission accounting.  Until then, the driver should continue
  negotiating no GSO/TSO/UFO features and should treat every packet-interface
  frame as one complete Ethernet frame.
- Multiqueue support needs a NET0 queue-selection policy, per-queue counters,
  and CPU-affinity rules.  The first cut should map queue pairs onto scheduler
  CPUs through an architecture-neutral device/IRQ abstraction rather than
  hardcoding x86 PCI assumptions in the driver.
- Promiscuous mode and packet filtering need explicit capabilities.  A future
  packet-capture/debug server should receive attenuated observe-only authority,
  while normal protocol delivery should remain owned by NET0.
- IRQ-driven completions are intentionally deferred to exploration 025.  The
  current bounded polling loops are adequate for attach tests but should be
  replaced by IRQ/event delivery before external traffic or performance work.
- External-network smoke tests require ARP for IPv4, neighbor discovery for
  IPv6, Ethernet encapsulation/decapsulation in NET0, deterministic host-side
  traffic generation, and a harness-owned QEMU backend.  QEMU user networking
  is useful for the first topology, but the real smoke target should probably
  use an isolated socket or tap backend so tests can inject and observe packets
  without depending on host network policy.
- External-network smoke tests are split into
  `025-virtio-net-external-smoke.txt` because the prerequisite host-side
  topology and Ethernet/ARP/ND protocol work do not exist yet.  Exploration 018
  stops at deterministic device attach, queue readiness, NET0 packet-interface
  routing state, and synthetic packet-path coverage.
