Build a deeper scheduler benchmark and policy-control test suite.

Problem:

- Exploration 020 brought up the EEVDF-inspired scheduler, basic scheduler
  statistics, `/proc/sched`, Linux `sched_yield(2)`, and a first scheduler
  stress test.  That is enough to make scheduler behavior observable in normal
  tests, but it is not a full benchmark suite.
- Harder questions need dedicated policy controls and repeatable measurements:
  sustained CPU-bound fairness, starvation thresholds, explicit affinity masks,
  migration behavior, server RPC latency under CPU pressure, and interactive
  shell latency under load.
- Those tests should not depend on ad hoc wall-clock guesses hidden inside
  broad filesystem shards.  They should be explicit benchmark/stress artifacts
  with clear counters and pass/fail thresholds.

Dependencies:

- Exploration 020 must be complete.
- Bunix should expose explicit scheduler policy controls before this work tries
  to test hard affinity or class/weight behavior.  The existing task metadata
  is enough to describe policy, but not enough to grant or adjust it from a
  capability-controlled scheduler policy server.

Design direction:

- Extend `/proc/sched` or add a dedicated scheduler stats RPC with stable
  per-task/per-thread counters: runtime, runnable wait, wake-to-run latency,
  preemptions, migrations, class, weight, priority, CPU, and affinity mask.
- Add a native scheduler-policy server or protocol that grants attenuated
  scheduling rights: class changes, priority changes, weight changes, and CPU
  affinity masks.
- Build focused stress binaries rather than burying all scheduler checks in
  broad VFS shards.  Keep the current `schedstress` as a smoke-level churn
  test, then add deeper CPU and IPC benchmarks with deterministic output.
- Prefer relative comparisons between competing workers inside one guest boot
  over brittle host wall-clock thresholds.

Tasks:

- [x] Define the scheduler policy RPC/capability model for class, priority,
      weight, and affinity changes without allowing ambient self-promotion.
- [x] Expose per-thread scheduler statistics sufficient for fairness and
      latency tests, including task/thread IDs that can be correlated with
      Linux PIDs through proc/linux server state.
- [x] Add a CPU-bound fairness benchmark that runs equal-weight workers and
      verifies bounded runtime skew.
- [x] Add a starvation benchmark with mixed CPU-bound and sleeping/yielding
      workers, verifying that interactive workers keep bounded wake-to-run
      latency.
- [x] Add an IPC/server latency benchmark that drives repeated RPCs while
      background CPU workers are runnable.
- [x] Add an interactive shell latency test that runs commands under background
      CPU load and checks bounded command completion using scheduler counters
      rather than only host wall time.
- [x] Add SMP migration and idle-balancing tests once migration policy exists:
      prove idle CPUs pull work, migrations are counted, and virtual runtime is
      preserved.  Exploration 070 now owns this because idle-pull migration
      policy does not exist yet.
- [x] Add affinity tests once capability-scoped CPU masks exist: pin workers,
      verify they run on allowed CPUs, and verify invalid masks are rejected.
      Exploration 069 now owns this because capability-scoped CPU masks do not
      exist yet.
- [x] Add comparison reporting for key shards and scheduler benchmarks so
      future scheduler changes can see runtime/counter deltas.

Progress notes:

- CPU-bound fairness benchmark slice:
  - Added `/bin/schedbench` as a static musl Linux-personality benchmark in
    the synthetic rootfs.
  - Its first mode, `schedbench fairness`, forks equal CPU-bound workers,
    starts them from one guest-side monotonic deadline, compares completed work
    counters, rejects zero work and gross runtime skew, and verifies
    `/proc/sched` switch/runtime counters advance.
  - Added a dedicated `scheduler-bench` shell shard so scheduler benchmark
    work can run independently of broad tmpfs/VFS shards.
  - The benchmark prints stable comparison data:
    total iterations, min/max worker iterations, skew, switch/runtime/wakeup/
    preemption deltas, max wait, max wake-to-run, and active CPU count.
  - Verification:
    - `make build/modules/schedbench.user`
    - `BUNIX_TEST_SET=scheduler-bench make test-parallel`
    - `git diff --check`
- Per-thread scheduler statistics slice:
  - Added a fixed scalar `sched_thread_info` snapshot ABI with task ID, thread
    ID, state, CPU, scheduler class/priority/weight, runtime, wakeups,
    migrations, preemptions, virtual runtime/deadline, pending runnable wait,
    pending wake-to-run latency, and task/thread names.
  - Added the x86_64 native `sched_thread_info` syscall and exposed the data
    through procfs as `/proc/sched_threads`.
  - Kept `/proc/sched` unchanged for existing parsers and tests.
  - Added rootfs/procfs shard coverage that reads `/proc/sched_threads` and
    pins the header fields needed by scheduler benchmarks.
  - Verification:
    - `make ARCH=x86_64 all`
    - `make build/modules/procfs.server`
    - `BUNIX_TEST_SET=rootfs-vfs-proc-dev make test-parallel`
    - `git diff --check`
- Starvation/interactive sleeper benchmark slice:
  - Extended `/bin/schedbench` with `schedbench starvation`.
  - The mode runs CPU-bound workers together with sleeper/yielder workers
    under one guest monotonic deadline, verifies CPU workers keep making
    progress, verifies sleepers keep taking wakeups, and fails if the maximum
    observed sleeper wake interval exceeds a bounded threshold.
  - The mode also checks scheduler switch/runtime/wakeup counters advance and
    prints CPU work, sleeper sample count, average/max sleeper interval, and
    scheduler counter deltas for future comparisons.
  - The dedicated `scheduler-bench` shard now runs both fairness and
    starvation modes.
  - Verification:
    - `make build/modules/schedbench.user`
    - `BUNIX_TEST_SET=scheduler-bench make test-parallel`
    - `git diff --check`
- IPC/server latency benchmark slice:
  - Extended `/bin/schedbench` with `schedbench ipc`.
  - The mode keeps CPU-bound workers runnable while repeatedly opening,
    reading, and closing `/proc/sched_threads`, which drives Linux server,
    VFS, and procfs RPC paths under scheduler pressure.
  - The benchmark bounds guest-side round-trip latency, verifies CPU workers
    still progress, verifies scheduler counters advance, and prints latency
    sample count, average/max latency, CPU work, and scheduler counter deltas.
  - The dedicated `scheduler-bench` shard now runs fairness, starvation, and
    IPC/server latency modes.
  - Verification:
    - `make build/modules/schedbench.user`
    - `BUNIX_TEST_SET=scheduler-bench make test-parallel`
    - `git diff --check`
