Debug and fix Linux-visible uptime accounting.

Problem:

- `uptime(1)` can report `0` seconds even after Bunix has been running for
  several minutes.
- This is likely a Linux personality accounting bug, a time server/kernel tick
  accounting bug, or an incomplete `/proc/uptime`/`sysinfo(2)` implementation.

Constraints:

- Keep Linux process time separate from Bunix task identity, but expose
  Linux-compatible uptime semantics to Linux userspace.
- Preserve Bunix-native observability where useful; Linux-visible uptime should
  not require pretending Bunix boot and Linux init are the same event unless
  that is explicitly chosen.
- Prefer a narrow reproducer over long manual waits.

Tasks:

- [x] Identify whether BusyBox `uptime` reads `/proc/uptime`, calls
      `sysinfo(2)`, consults utmp, or combines multiple sources on Bunix.
- [x] Add strace/log coverage showing the exact Linux ABI path used by
      `uptime(1)`.
- [x] Audit kernel timer, time server, procfs, and Linux server uptime sources
      for inconsistent epochs or unadvanced counters.
- [x] Define the intended epochs for Bunix kernel uptime, Linux personality
      uptime, and per-session/login accounting.
- [x] Implement the missing or incorrect uptime source using a monotonic
      kernel/time-server primitive rather than wall-clock time.
- [x] Add a narrow test that waits a bounded interval and asserts Linux-visible
      uptime advances.
- [x] Verify `busybox uptime`, `/proc/uptime`, and any implemented `sysinfo(2)`
      path agree closely enough for Linux userspace.

Progress:

- Found that procfs `/proc/uptime` already used
  `bunix_clock_monotonic_ns()`, while Linux `sysinfo(2)`,
  `clock_gettime(2)`, `gettimeofday(2)`, and `time(2)` treated raw
  `bunix_timer_ticks()` as nanoseconds.  That made Linux-server time advance
  far too slowly and could leave Linux-visible uptime at zero.
- Chose Bunix kernel monotonic time as the current shared epoch for Bunix
  uptime and Linux personality uptime.  Login/session accounting remains
  separate through utmp/session state; BusyBox uptime still reports users from
  that path.
- Added `/bin/uptimetest`, a static musl regression binary that samples
  `sysinfo(2)`, `clock_gettime(CLOCK_MONOTONIC)`, and `/proc/uptime`, sleeps
  for a bounded two seconds, then verifies all sources advance and agree within
  one second.
- Added `uptimetest` to the login smoke shard beside the existing `uptime` and
  `busybox uptime` checks.  A structured debug run with
  `KERNEL_CMDLINE='log=debug strace=structured'` kept the runtime log and
  reported `uptimetest ok sys=20 clock=20 proc=20`, proving the Linux ABI
  sources and procfs source advanced together in-guest.

Validation so far:

- `make build/modules/linux.server`
- `make build/modules/uptimetest.user`
- `BUNIX_SHELL_PART=login-smoke make test-shell-part`
- `KEEP_TMP=1 BUNIX_SHELL_PART=login-smoke KERNEL_CMDLINE='log=debug strace=structured' BUNIX_STRACE_LINES=800 make test-shell-part`
- `BUNIX_SHELL_PART=login-smoke KERNEL_CMDLINE='log=info' make test-shell-part`
- `BUNIX_SHELL_PART=procfs-cmdline KERNEL_CMDLINE='log=info' make test-shell-part`
- `BUNIX_TEST_SET=openrc KERNEL_CMDLINE='log=info' make test-parallel`
- `make test`

Follow-up fix while validating:

- Full-test validation exposed existing `ps | grep`/procfs PID fragility in
  `procfs-cmdline` and `alpine-openrc-full`.  Procfs numeric paths now accept
  Linux PID aliases by asking proc for `BUNIX_PROC_INFO_BY_LINUX_PID`, while
  keeping Bunix proc IDs as the canonical internal IDs.  This lets shell `$!`
  and Linux-visible PIDs resolve through `/proc/<pid>` without changing the
  underlying Bunix process identity model.
