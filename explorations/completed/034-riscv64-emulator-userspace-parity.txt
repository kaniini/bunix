Finish riscv64 userspace bringup in the QEMU virt emulator before starting
board-specific hardware work.

Problem:

- The initial riscv64 port proves early boot, timer traps, kernel-thread
  context switching, U-mode entry, a native syscall ABI, boot-package payload
  discovery, and one Sv39-backed crt0 payload.
- That is still an early harness.  It does not yet run scheduler-owned native
  tasks through the normal proc/bootstrap path, does not have general
  userspace VM/copyin/copyout on riscv64, and does not run the Linux
  personality in the emulator.
- Board bringup should wait until the QEMU `virt` emulator path is genuinely
  working.  Otherwise hardware debugging will mix platform defects with
  unfinished generic riscv64 architecture work.

Intended outcome:

- `make run-riscv64-early` or a successor emulator target boots a riscv64
  kernel under QEMU `virt`, starts scheduler-owned native Bunix userspace
  tasks, runs at least one native server-shaped payload through the normal
  launch path, and powers off cleanly.
- A static riscv64 musl hello binary can be built with the external musl.cc
  `riscv64-linux-musl-cross.tgz` toolchain, packaged into the riscv64 boot
  carrier, loaded through the normal proc/VFS/server path, executed by the
  Linux personality server using riscv64 Linux syscall numbers, and observed
  in a QEMU smoke test.
- The BPI-F3 exploration remains blocked until this emulator target is
  complete and stable.

Constraints:

- Keep hardware/board-specific code out of this exploration.  The target is
  `qemu-system-riscv64 -machine virt`.
- Prioritize further riscv64 emulator bringup over BPI-F3 work.  Board
  bringup must not start while any emulator userspace parity tasks in this
  exploration remain open.
- Preserve the microkernel shape.  Do not pull VFS, proc, Linux personality,
  or filesystem policy into the riscv64 kernel path to simplify bringup.
- Keep x86_64 behavior and tests green while adding riscv64 implementations of
  existing architecture hooks.
- Do not fake a riscv64 musl target using headers or libraries that are not
  actually available.  Use the external musl.cc cross toolchain when the Linux
  personality slice needs libc.
