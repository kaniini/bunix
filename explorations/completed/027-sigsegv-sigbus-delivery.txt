Implement SIGSEGV and SIGBUS delivery for Linux-personality processes.

Problem:

- User faults currently behave like kernel-facing task failures instead of
  Linux-compatible synchronous signals.
- Real userland expects invalid memory accesses, protection violations, and
  bus errors to be reported through `SIGSEGV`/`SIGBUS` with useful `siginfo_t`
  contents and default process termination semantics.
- Debugging and shell behavior will remain rough until crashes look like
  Linux process crashes rather than unexplained task death or kernel logging.

Constraints:

- Keep task faults and Linux process signals conceptually separate: Bunix tasks
  are lower-level execution objects, while Linux PIDs are process-level objects
  managed by the Linux/proc servers.
- Signal delivery should be implemented through the Linux personality server
  wherever practical.  Kernel architecture code should classify faults, capture
  machine context, and hand off enough structured state to deliver the signal.
- Preserve native observability.  Native Bunix programs should still be able to
  see low-level task fault information without pretending every task is a Linux
  process.
- Do not bake x86-only signal-frame assumptions into generic Linux-server
  policy.  Keep architecture-specific frame construction isolated so ARM can
  supply its own frame shape later.

Progress notes:

- The current slice adds a generic Bunix task-fault event shape and routes
  user-mode synchronous faults to the Linux server as `BUNIX_LINUX_TASK_FAULT`
  messages.
- The kernel fault path now uses a reply-capable Linux-server RPC so the
  exception path can either build a signal frame and resume into a handler or
  let the server record default process termination before the task exits.
- The Linux server currently implements default termination only: page faults
  and general-protection faults become SIGSEGV-style wait status, while x86
  alignment-check faults become SIGBUS-style wait status.
- File-backed mmap-hole SIGBUS remains open until the VFS mmap path can report
  that condition distinctly; this is split into
  `050-vfs-backed-mmap-sigbus.txt`.
- Synchronous fault delivery now reuses the Linux signal disposition tables:
  if a SIGSEGV/SIGBUS handler and restorer are installed, the x86 exception
  path builds a signal frame on the interrupted user stack and resumes into the
  handler; otherwise the Linux server records the default signal exit status.
- `SA_SIGINFO` handlers for synchronous SIGSEGV/SIGBUS receive a minimal
  Linux-compatible `siginfo_t` with signal number, code, and address.  Page
  faults report `cr2` as `si_addr`; x86 alignment-check SIGBUS reports
  `BUS_ADRALN` with a zero address because #AC does not expose the effective
  memory address.
- The focused default SIGSEGV/SIGBUS regression is:

  ```
  make BUNIX_USER=root BUNIX_PASSWORD=root BUNIX_PROMPT='~ # ' \
    BUNIX_CMD=/bin/faulttest \
    BUNIX_MARKER='faulttest sigsegv sigbus wait status ok' test-command
  ```
- The focused caught SIGSEGV/SIGBUS regression is:

  ```
  make BUNIX_USER=root BUNIX_PASSWORD=root BUNIX_PROMPT='~ # ' \
    BUNIX_CMD=/bin/signaltest \
    BUNIX_MARKER='linux signaltest ok' test-command
  ```

Remaining Signal Roadmap:

- File-backed mmap-hole SIGBUS is tracked separately in
  `050-vfs-backed-mmap-sigbus.txt`, because the current mmap implementation
  eagerly copies file bytes into anonymous mappings and loses the file extent
  metadata needed to distinguish a late mapping hole from an ordinary unmapped
  page.
- Asynchronous signal delivery should grow beyond the current syscall-return
  polling model so runnable tasks can be interrupted promptly for job-control,
  timer, and externally generated signals.
- Job-control signals need process-group and terminal foreground semantics
  connected to the TTY/session model, including SIGSTOP, SIGTSTP, SIGTTIN,
  SIGTTOU, SIGCONT, and shell-visible wait status changes.
- Timer-generated signals should be integrated with the time server and Linux
  personality signal queues, including interval timers and per-process timer
  ownership rules.
- ptrace/strace interactions should preserve the current native observability
  while adding Linux-compatible stop reasons, signal injection, and fault
  inspection.
