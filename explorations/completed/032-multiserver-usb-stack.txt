Implement a basic multiserver USB stack.

Problem:

- Bunix needs USB support for real hardware bringup: keyboards, serial
  adapters, storage, hubs, and board-level debug devices are common on both
  x86_64 machines and future non-x86 targets.
- USB is too broad to collapse into one driver without losing the microkernel
  shape.  Host-controller policy, bus enumeration, hub management, transfer
  scheduling, and class drivers should be separated by explicit protocols and
  capabilities.
- The riscv64 hardware work should not inherit x86_64-only assumptions about
  PCI transport or QEMU-only devices.  A basic USB design before riscv64
  bringup gives the port a clearer story for USB host controllers discovered
  through PCIe, device tree, ACPI, or board-specific resources.

Scheduling:

- Run this after `030-pci-pcie-server.txt` and before
  `033-riscv64-initial-port.txt`.
- The first implementation may target deterministic x86_64 QEMU hardware, but
  the protocol split must be transport-neutral enough for device-tree described
  host controllers on riscv64 and future ARM work.
- Keep existing boot, shell, VFS, VirtIO, and networking tests passing while
  USB servers are introduced behind explicit boot/test flags.

Design direction:

- Introduce a userspace USB bus manager server, likely `usb-bus.server`, with a
  FourCC protocol such as `USB0`.  It owns device addressing, descriptor
  parsing, configuration selection, interface claiming, and child-device
  namespace publication.
- Introduce host-controller driver servers as separate providers below the bus
  manager.  Initial candidates:
  - `xhci.server` for modern PCI/PCIe xHCI controllers.
  - Later `ehci.server`/`ohci.server`/`uhci.server` only if needed for target
    hardware.
  - Later device-tree or platform host-controller providers for riscv64/ARM.
- Keep class drivers as separate servers above `USB0`: HID keyboard/mouse,
  mass-storage/Bulk-Only Transport, CDC ACM serial, hub, and future network
  classes should claim interfaces through attenuated interface capabilities.
- Preserve OCAP boundaries: the kernel grants raw MMIO/I/O/IRQ/DMA resources to
  the host-controller server; the host-controller grants only endpoint/transfer
  capabilities upward; class drivers never receive broad hardware authority.
- Model transfers as asynchronous requests with completion events, cancellation,
  timeouts, and bounded buffers.  Avoid baking polling into the class-driver
  contract, even if the first backend uses polling internally.

Constraints:

- Do not put USB enumeration or class policy in the kernel.
- Do not let a class driver talk to arbitrary endpoints.  Interface claims
  should produce capabilities scoped to endpoint IDs, transfer types, max packet
  sizes, and direction.
- Do not assume PCI is the only transport.  xHCI over PCI can be first, but
  the bus-manager protocol must also work with host controllers discovered from
  device tree or firmware resources.
- Keep the first slice small enough to test.  A minimal xHCI path that
  enumerates a QEMU USB device and logs descriptors is preferable to a broad
  untested class stack.
- Avoid GPL/copyleft USB stack reuse in Bunix kernel/servers.  Treat GPL
  sources as behavioral references only unless the code stays in allowed
  userspace tooling.

USB DMA model:

- Class drivers never hand host controllers raw arbitrary task memory.  A class
  driver submits a bounded shared buffer and an endpoint capability to `USB0`.
- `usb-bus.server` validates endpoint scope, transfer type, direction, maximum
  packet size, timeout, and transfer length before forwarding work to a host
  controller.
- Host-controller servers receive only per-transfer DMA authority.  The
  intended primitive is a pinned or bounce-backed DMA buffer capability derived
  from the caller's shared buffer for the exact byte range needed by that
  transfer.
- Until Bunix has an IOMMU server, all real USB DMA must use bounded
  bounce buffers allocated by the trusted host-controller server or a dedicated
  DMA broker.  Direct device access to arbitrary userspace physical pages is
  not acceptable for class drivers.
- x86_64 can initially treat coherent shared/bounce buffers as cache coherent,
  but the API must include explicit sync points before submit and after
  completion so riscv64/ARM can add cache maintenance without changing
  class-driver contracts.
- Ownership rules:
  - OUT transfers copy or pin caller data before submit; the caller may not
    mutate the submitted range until completion or cancellation.
  - IN transfers write into a controller-owned bounce buffer first, then copy
    into the caller buffer only after a successful completion.
  - Setup packets for control transfers are copied into controller-owned memory
    and are never exposed as mutable hardware state.
  - Cancellation returns either completed bytes or a cancelled status, never
    silent partial mutation.
- Completion records should include endpoint ID, transfer cookie, status,
  actual length, short-packet indication, stall/reset indication, and timeout
  status.  Completion delivery should be through an event port or wait RPC, not
  by polling shared memory from class drivers.
- Follow-up needed before real hardware exposure:
  - Add a generic DMA/bounce-buffer broker or extend the buffer server with
    pinned DMA grants.
  - Add IOMMU domain design for PCI/PCIe controllers.
  - Add cache maintenance hooks in an architecture abstraction.
  - Add transfer lifetime accounting so cancellation and task exit revoke DMA
    authority promptly.
