Align OpenRC runlevels with a normal Alpine image.

Problem:

- The Alpine rootfs builder currently installs Alpine packages and then
  overlays a small Bunix-specific OpenRC setup.  It creates only the service
  links needed for the current boot/login smoke path, notably
  `default/bunix-login`.
- A normal Alpine image has a broader and more intentional OpenRC runlevel
  layout across `sysinit`, `boot`, `default`, `shutdown`, and sometimes
  additional runlevels.  If Bunix diverges without documenting why, OpenRC
  behavior, service ordering, dependency caching, and future multiuser boot
  work can drift from Alpine expectations.
- We need Bunix-specific login integration, but the rest of the runlevel
  structure should be aligned with Alpine defaults unless a service is
  impossible or intentionally replaced by a Bunix server.

Constraints:

- Keep Bunix's login program.  BusyBox login is not a substitute because Bunix
  login contacts the user server.
- Do not cargo-cult host state.  Use a controlled Alpine rootfs generated by
  `apk add --root ... --initdb` or an explicitly documented reference image.
- Do not enable services that require unsupported kernel/device features
  unless the exploration also documents the missing Bunix capability and marks
  the service intentionally deferred.
- Preserve deterministic tests: generated runlevel links should not depend on
  the developer machine's ambient `/etc/runlevels`.

Design direction:

- Treat Alpine's generated OpenRC runlevel directories as the reference input,
  then apply a small Bunix policy layer for services we replace, suppress, or
  add.
- Keep the rootfs builder responsible for materializing runlevel links in the
  rootfs image.  Runtime VFS should just expose the resulting filesystem state.
- Record service parity decisions in a manifest so tests can distinguish
  expected Bunix deviations from missing links.

Implementation notes:

- A controlled plain Alpine rootfs generated with the same package set
  (`alpine-baselayout busybox musl openrc ifupdown-ng`) creates the standard
  OpenRC runlevel directories (`sysinit`, `boot`, `default`, `nonetwork`,
  `shutdown`) but no service links by default.  It installs the OpenRC init
  scripts and conf.d files used as the reference inventory.
- `tools/alpine-openrc-runlevels.policy` now records Bunix policy decisions:
  add the temporary Bunix `networking` service to `boot`, replace getty/login
  with `/bin/login`, and defer devfs/procfs/sysfs OpenRC services because those
  filesystems are currently mounted by Bunix servers through VFS.
- `tools/build-alpine-rootfs.sh` snapshots reference runlevels, init scripts,
  and conf.d files into `build/alpine-rootfs`, applies only policy-approved
  runlevel links, snapshots the Bunix result, and records the policy paths in
  the rootfs manifest.
- Alpine `inittab` generation now follows the normal OpenRC flow:
  `openrc sysinit`, `openrc boot`, `openrc default`, serial `/bin/login`
  respawn, and `openrc shutdown`.  The old direct
  `rc-service networking start` wait entry was removed; networking is now
  started through the `boot` runlevel policy link.
- `tools/test-alpine-rootfs.sh` builds a controlled rootfs stage and verifies
  runlevel directories, the Bunix networking policy link, policy manifest
  entries, reference artifacts, and the getty-to-Bunix-login replacement.

Remaining parity gaps:

- The temporary Bunix `networking` init script remains a replacement for stock
  Alpine networking service behavior.  Restoring the stock Alpine networking
  service is tracked separately by the existing stock-networking exploration.
- OpenRC devfs, procfs, and sysfs services remain deferred until those services
  can either be harmless under Bunix server-mounted filesystems or replaced by
  explicit policy that models the server-provided mounts.
- The controlled APK package root does not enable a broad default service set
  by itself; this exploration therefore records Bunix policy over the package
  reference rather than copying ambient host `/etc/runlevels` state.
