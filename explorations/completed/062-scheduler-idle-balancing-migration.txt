Implement scheduler idle balancing and migration tests.

Problem:

- Exploration 040 added scheduler benchmarks and per-thread observability, but
  the scheduler still does not have a deliberate idle-pull migration policy.
- Existing migration counters mostly cover placement on a non-current CPU, not
  a policy where idle CPUs pull runnable work from busy CPUs while preserving
  virtual runtime/deadline fairness.
- SMP migration tests should only be added once migration is an explicit
  scheduler policy rather than an accidental side effect of task placement.

Idle-balancing invariants:

- An idle CPU may pull only after its local run queue is empty.
- A source CPU is stealable only when it has at least two queued runnable
  user/batch threads, leaving singleton interactive work and server/kernel
  placement alone.
- The destination CPU must be present in the candidate thread's affinity mask.
- Each idle pass pulls at most one eligible thread from the busiest eligible
  source CPU.
- Migration preserves the thread's virtual runtime/deadline state except that
  virtual runtime is clamped up to the destination run queue minimum before the
  deadline is refreshed, avoiding unfair credit from moving to another CPU.
- Enqueue only kicks idle pullers when queued user/batch work reaches the same
  bounded source threshold.

- [x] Count policy-driven migrations separately enough that tests can
