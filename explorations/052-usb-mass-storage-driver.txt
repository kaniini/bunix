Implement USB mass-storage over the USB stack.

Problem:

- Bunix has block-device protocols and filesystem servers, but USB storage is
  not available through the USB stack yet.
- USB mass storage should map into the existing block-device path rather than
  inventing a separate filesystem or disk interface.

Constraints:

- Keep mass-storage policy in a userspace class-driver server above `USB0`.
- Use scoped interface and bulk endpoint capabilities from `USB0`; do not give
  the class driver raw xHCI/MMIO/DMA authority.
- Publish discovered USB disks through the existing block-device protocol.
- Preserve VirtIO block and existing VFS tests.

Tasks:

- [ ] Extend `USB0` bulk endpoint opening and transfer completion APIs enough
      for class drivers to submit bounded transfers and wait on completions.
- [ ] Add `usb-storage.server` for Bulk-Only Transport command block wrappers,
      data phase, command status wrappers, reset/retry, and basic inquiry/read
      capacity/read/write commands.
- [ ] Map USB storage devices into `BLK0` or the existing block-device
      namespace without adding a new filesystem path.
- [ ] Add a deterministic QEMU USB storage smoke test with read, write, flush
      where supported, and guest poweroff.
- [ ] Keep `test-boot-virtio-blk`, `test-boot-usb`, and default boot passing.

Completion criteria:

- A QEMU USB mass-storage device can be enumerated through xHCI, claimed by
  `usb-storage.server`, and exposed through the existing block-device protocol.
- Existing VirtIO block behavior is unchanged.
