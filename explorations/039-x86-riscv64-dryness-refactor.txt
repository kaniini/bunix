Refactor x86_64 and riscv64 code for general DRYness after riscv64 Alpine
bringup.

Status:

- Pending.
- Start this after `035-riscv64-cleanup-alpine-environment.txt` and the
  focused riscv64 refactor explorations `036` through `038`.

Problem:

- x86_64 is the mature architecture path and riscv64 was brought up
  incrementally through focused smoke tests.
- Some duplication is useful while an architecture is young, but once riscv64
  can run an Alpine environment, duplicated kernel lifecycle, syscall
  forwarding, server packaging, rootfs boot, and test orchestration logic
  should be collapsed into shared abstractions.
- The goal is not aesthetic deduplication; it is to make future architecture
  ports and board work cheaper without hiding architecture-specific behavior.

Tasks:

- [ ] Audit x86_64/riscv64 duplication across kernel entry, runtime init,
      boot module handling, user entry, syscall forwarding, server packaging,
      rootfs image wiring, and QEMU test targets.
- [ ] Identify which differences are architectural contracts and which are
      accidental bringup duplication.
- [ ] Create shared helpers for generic lifecycle and boot-module sequencing
      where both architectures already agree.
- [ ] Move Linux-personality syscall forwarding into a shared registry or
      dispatch layer with arch-specific ABI tables.
- [x] Replace the riscv64 dynamic-loader smoke's temporary `mprotect(2)`
      success stub with real VM permission updates shared with the x86_64
      Linux mmap/mprotect semantics.
- [x] Replace the temporary riscv64 frontend behavior that re-sends
      `BUNIX_LINUX_REGISTER_PROCESS` before forwarded scalar syscalls with
      per-task Linux frontend state.  This must support concurrent riscv64
      Linux tasks without a fixed single-task cache.
- [ ] Unify server module build/package rules where the only difference is
      target triple, crt0, or linker mode.
- [ ] Factor freestanding userspace runtime support such as `memcpy`,
      `memset`, and future compiler-emitted helper symbols into a shared user
      support object instead of carrying one-off definitions in individual
      servers.
- [ ] Unify common QEMU test harness behavior while preserving architecture
      machine, firmware, and shutdown differences.
- [ ] Update documentation to describe the architecture boundary explicitly:
      what belongs in `arch/*`, what belongs in generic kernel code, and what
      belongs in userspace servers.
- [ ] Keep BPI-F3 artifacts and x86_64 Alpine boot tests passing throughout.

Progress notes:

- Created as the broad follow-on exploration requested after riscv64 cleanup
  and Alpine environment work, then moved after the focused riscv64 refactor
  explorations created from the `035` audit.
- Exploration 035 intentionally kept the riscv64 shared-Linux smoke narrow:
  the frontend treats duplicate registration as already registered so
  `/bin/rv64-syscall-smoke` and `/bin/musl-hello` can run concurrently.  This
  should become a real per-task frontend table once the riscv64 Alpine path is
  far enough along to justify broader cleanup.
- Exploration 035 added a local freestanding `memcpy` definition to
  `user/proc/main.c` because the riscv64 compiler lowered a struct copy into
  an external helper while linking proc without libc.  Keep that local unblock
  narrow; the DRYness pass should replace it with shared user runtime support.
- Exploration 035 later added an interim `user/runtime/string.c` object for
  the riscv64 filesystem-server builds because squashfs and future larger
  userspace servers need compiler-emitted string helpers too.  The DRYness
  pass should fold proc and server builds onto one explicit freestanding
  userspace runtime policy instead of keeping mixed local and per-rule helper
  wiring.
- First cleanup slice:
  - Removed proc's local freestanding `memcpy` implementation.
  - Linked the riscv64 proc server against the shared `user/runtime/string.c`
    object already used by the riscv64 filesystem-server builds.
  - Verification:
    - `make test-riscv64-proc-server-build`
    - `make test-boot-riscv64-alpine`
    - `git diff --check`
- Riscv64 `mprotect(2)` slice:
  - Replaced the riscv64 frontend's temporary success return with real
    `vm_protect_user_range()` and `task_protect_vm_region()` updates using
    the same Linux-prot to task-prot mapping as the x86_64 path.
  - `make test-boot-riscv64-alpine` logs `linux-riscv64: mprotect` for both
    `/bin/dyn-hello` and the Alpine shell path before the deterministic
    poweroff.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make test-boot-riscv64-alpine`
    - `git diff --check`
- Riscv64 Linux frontend-state slice:
  - Added a protected per-task frontend-state list keyed by Bunix task ID so
    positive Linux syscalls do not resend `BUNIX_LINUX_REGISTER_PROCESS` after
    the current task is known to the Linux server.
  - Preserved the current proc/Linux ownership path by caching a one-time
    `-EINVAL` "already registered" response instead of retrying it on every
    forwarded syscall.
  - Linux `exit_group` now drops the frontend entry before the task exits.
  - Added exploration 068 for generic task-exit cleanup hooks so future
    frontend-owned task state can be released on non-Linux-exit paths without
    expanding this DRYness slice.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make test-boot-riscv64-early`
    - `make test-boot-riscv64-alpine`
    - `git diff --check`
