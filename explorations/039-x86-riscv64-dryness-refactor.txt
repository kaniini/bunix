Refactor x86_64 and riscv64 code for general DRYness after riscv64 Alpine
bringup.

Status:

- Pending.
- Start this after `035-riscv64-cleanup-alpine-environment.txt` and the
  focused riscv64 refactor explorations `036` through `038`.

Problem:

- x86_64 is the mature architecture path and riscv64 was brought up
  incrementally through focused smoke tests.
- Some duplication is useful while an architecture is young, but once riscv64
  can run an Alpine environment, duplicated kernel lifecycle, syscall
  forwarding, server packaging, rootfs boot, and test orchestration logic
  should be collapsed into shared abstractions.
- The goal is not aesthetic deduplication; it is to make future architecture
  ports and board work cheaper without hiding architecture-specific behavior.

Tasks:

- [x] Audit x86_64/riscv64 duplication across kernel entry, runtime init,
      boot module handling, user entry, syscall forwarding, server packaging,
      rootfs image wiring, and QEMU test targets.
- [x] Identify which differences are architectural contracts and which are
      accidental bringup duplication.
- [x] Create shared helpers for generic lifecycle and boot-module sequencing
      where both architectures already agree.
- [x] Move Linux-personality syscall forwarding into a shared registry or
      dispatch layer with arch-specific ABI tables.
- [x] Replace the riscv64 dynamic-loader smoke's temporary `mprotect(2)`
      success stub with real VM permission updates shared with the x86_64
      Linux mmap/mprotect semantics.
- [x] Replace the temporary riscv64 frontend behavior that re-sends
      `BUNIX_LINUX_REGISTER_PROCESS` before forwarded scalar syscalls with
      per-task Linux frontend state.  This must support concurrent riscv64
      Linux tasks without a fixed single-task cache.
- [ ] Unify server module build/package rules where the only difference is
      target triple, crt0, or linker mode.
- [x] Factor freestanding userspace runtime support such as `memcpy`,
      `memset`, and future compiler-emitted helper symbols into a shared user
      support object instead of carrying one-off definitions in individual
      servers.
- [x] Unify common QEMU test harness behavior while preserving architecture
      machine, firmware, and shutdown differences.
- [x] Update documentation to describe the architecture boundary explicitly:
      what belongs in `arch/*`, what belongs in generic kernel code, and what
      belongs in userspace servers.
- [ ] Keep BPI-F3 artifacts and x86_64 Alpine boot tests passing throughout.

Progress notes:

- Created as the broad follow-on exploration requested after riscv64 cleanup
  and Alpine environment work, then moved after the focused riscv64 refactor
  explorations created from the `035` audit.
- Exploration 035 intentionally kept the riscv64 shared-Linux smoke narrow:
  the frontend treats duplicate registration as already registered so
  `/bin/rv64-syscall-smoke` and `/bin/musl-hello` can run concurrently.  This
  should become a real per-task frontend table once the riscv64 Alpine path is
  far enough along to justify broader cleanup.
- Exploration 035 added a local freestanding `memcpy` definition to
  `user/proc/main.c` because the riscv64 compiler lowered a struct copy into
  an external helper while linking proc without libc.  Keep that local unblock
  narrow; the DRYness pass should replace it with shared user runtime support.
- Exploration 035 later added an interim `user/runtime/string.c` object for
  the riscv64 filesystem-server builds because squashfs and future larger
  userspace servers need compiler-emitted string helpers too.  The DRYness
  pass should fold proc and server builds onto one explicit freestanding
  userspace runtime policy instead of keeping mixed local and per-rule helper
  wiring.
- First cleanup slice:
  - Removed proc's local freestanding `memcpy` implementation.
  - Linked the riscv64 proc server against the shared `user/runtime/string.c`
    object already used by the riscv64 filesystem-server builds.
  - Verification:
    - `make test-riscv64-proc-server-build`
    - `make test-boot-riscv64-alpine`
    - `git diff --check`
- Riscv64 `mprotect(2)` slice:
  - Replaced the riscv64 frontend's temporary success return with real
    `vm_protect_user_range()` and `task_protect_vm_region()` updates using
    the same Linux-prot to task-prot mapping as the x86_64 path.
  - `make test-boot-riscv64-alpine` logs `linux-riscv64: mprotect` for both
    `/bin/dyn-hello` and the Alpine shell path before the deterministic
    poweroff.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make test-boot-riscv64-alpine`
    - `git diff --check`
- Riscv64 Linux frontend-state slice:
  - Added a protected per-task frontend-state list keyed by Bunix task ID so
    positive Linux syscalls do not resend `BUNIX_LINUX_REGISTER_PROCESS` after
    the current task is known to the Linux server.
  - Preserved the current proc/Linux ownership path by caching a one-time
    `-EINVAL` "already registered" response instead of retrying it on every
    forwarded syscall.
  - Linux `exit_group` now drops the frontend entry before the task exits.
  - Added exploration 068 for generic task-exit cleanup hooks so future
    frontend-owned task state can be released on non-Linux-exit paths without
    expanding this DRYness slice.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make test-boot-riscv64-early`
    - `make test-boot-riscv64-alpine`
    - `git diff --check`
- Riscv64 freestanding user-runtime slice:
  - Added a single `RISCV64_USER_CRT0_OBJ` build rule instead of recompiling
    `user/crt0-riscv64.S` inside every riscv64 server/module rule.
  - Promoted `user/runtime/string.c` to the explicit
    `RISCV64_USER_RUNTIME_OBJS` list and linked it into every riscv64
    freestanding server/ABI module, not only the modules that had already
    tripped compiler-emitted string helper references.
  - Left the larger module-rule/package-rule unification task open; this slice
    only normalized startup/runtime support.
  - Verification:
    - `make test-riscv64-shared-linux-server-build test-riscv64-proc-server-build test-riscv64-fs-server-build`
    - `make test-boot-riscv64-alpine`
- Architecture-boundary audit/doc slice:
  - Added `docs/architecture-boundary.md` to define ownership for `arch/*`,
    generic kernel mechanisms, and userspace server policy.
  - Classified accepted architecture differences: boot protocol/carrier,
    trap/syscall register ABI, context switch code, MMU format, timer/interrupt
    controller, SMP startup, and poweroff/reset mechanics.
  - Classified cleanup targets: normalized boot-module sequencing, shared
    Linux syscall forwarding machinery with arch ABI tables, server module
    build/package repetition, freestanding runtime helper policy, and QEMU
    harness behavior.
  - Added a README pointer so future port work has a stable local reference.
- Generic boot-package parser slice:
  - Moved the riscv64 text boot-package parser out of
    `arch/riscv64/early.c` and into `kernel/bootpkg.c`.
  - Riscv64 still owns FDT initrd discovery and decides when to run the
    low-level boot smoke; the parser now owns firmware-neutral package magic,
    cmdline extraction, module lookup, and module recording.
  - This is partial progress toward the lifecycle/boot-module helper task; the
    top-level x86_64 Multiboot2 and riscv64 FDT boot sequencing is still not
    unified.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make ARCH=x86_64 all`
    - `make test-riscv64-bootpkg test-boot-riscv64-early`
    - `make test-smoke`
- Shared Linux ABI helper slice:
  - Moved the duplicated Linux `PROT_*` to Bunix task VM permission mapping
    into `kernel/linux_abi.h`.
  - x86_64 and riscv64 mmap/mprotect paths now use the same
    `linux_prot_to_task()` helper.
  - This is a small first step toward the shared Linux syscall registry; the
    larger forwarding table and marshalling helpers remain architecture-local.
  - Verification:
    - `make ARCH=x86_64 all`
    - `make ARCH=riscv64 all`
    - `make test-boot-riscv64-alpine`
- Riscv64 boot-package argument slice:
  - Centralized the repeated riscv64 boot-package module/file argument lists
    into `RISCV64_BOOTPKG_*_ARGS` Make variables.
  - The standard, multi-record, Alpine, and UART boot-package recipes now
    share their package composition policy instead of spelling module pairs in
    every recipe body.
  - This is partial progress toward the server module build/package rule task;
    the x86_64 EFI module list and generic server link rules still need a
    broader cleanup.
  - Verification:
    - `make test-riscv64-bootpkg`
- Riscv64 QEMU harness slice:
  - Added `tools/test-riscv64-boot.sh` to own shared riscv64 QEMU command
    lookup, timeout, memory, serial log capture, and optional marker checking.
  - Moved the long riscv64 boot marker grep chains into
    `tools/test-riscv64-markers-early.txt`,
    `tools/test-riscv64-markers-alpine.txt`, and
    `tools/test-riscv64-markers-uart-console.txt`.
  - Updated `test-boot-riscv64-early`, `test-boot-riscv64-alpine`, and
    `test-boot-riscv64-uart-console` to use the shared harness while keeping
    the target-specific initrd, timeout, and QEMU `virt` machine policy.
  - Per-target serial-log isolation remains tracked in exploration 066; this
    slice preserves the existing shared `build/riscv64-serial.log` behavior.
  - Verification:
    - `make test-boot-riscv64-early test-boot-riscv64-alpine test-boot-riscv64-uart-console`
    - `git diff --check`
- Shared Linux message ID slice:
  - Added explicit `LINUX_MSG_*` protocol IDs to `kernel/linux_abi.h` for the
    Linux server message types shared by the x86_64 and riscv64 frontends.
  - Converted the riscv64 frontend away from local `LINUX_SHARED_*` aliases so
    its syscall table names the shared `LINX` protocol operation directly.
  - This is incremental progress toward a shared Linux syscall registry; the
    architecture-local dispatch and marshalling code still remains.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make ARCH=x86_64 all`
    - `make test-boot-riscv64-alpine`
- Shared Linux forwarding registry slice:
  - Added `enum linux_forward_op`, `struct linux_syscall_forward_entry`, and
    `linux_syscall_forward_lookup()` to `kernel/linux_abi.h`.
  - Converted the riscv64 Linux syscall table to use the shared forwarding
    entry shape and lookup helper while keeping the riscv64 syscall-number ABI
    table in `arch/riscv64/user.c`.
  - The mature x86_64 path still has its large switch-based frontend because
    it covers many more Linux operations, signal delivery, exec, and networking
    marshalling.  The shared registry boundary now exists for new architecture
    tables and future x86_64 migration without mixing those policies into this
    slice.
  - Verification:
    - `make ARCH=riscv64 all`
    - `make ARCH=x86_64 all`
    - `make test-boot-riscv64-alpine`
- Runtime boot-module lifecycle slice:
  - Added `kernel_runtime_boot_modules_init()` as the shared lifecycle hook for
    boot-module registry setup after IPC/name services are online.
  - x86_64 now initializes the boot-module registry through the generic
    runtime path before scheduler startup; `server_start_boot_modules()` only
    records Multiboot2 modules and launches the initial module graph.
  - riscv64 uses the same runtime hook before scheduler startup, while still
    preserving its FDT/bootpkg discovery and low-level self-test sequencing.
  - Verification:
    - `make ARCH=x86_64 all`
    - `make ARCH=riscv64 all`
    - `make test-smoke`
    - `make test-boot-riscv64-early`
