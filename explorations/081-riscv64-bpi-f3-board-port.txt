Target the riscv64 port to boot on Banana Pi BPI-F3 hardware.

Status:

- Paused.  Hardware proof remains open, but this exploration is intentionally
  deferred while the riscv64 generic port is cleaned up enough to boot an
  Alpine environment in QEMU.  Resume this exploration after
  `035-riscv64-cleanup-alpine-environment.txt` and
  `039-x86-riscv64-dryness-refactor.txt` have reduced the riscv64-only
  bringup scaffolding.
- `034-riscv64-emulator-userspace-parity.txt` is complete, moved to
  `explorations/completed/`, and passed the full `make test` gate after the
  QEMU `virt` path booted scheduler-owned native userspace plus the static
  riscv64 musl/Linux-personality smoke.
- Hardware execution is gated on emulator parity.  Do not start new BPI-F3
  physical-board work unless the QEMU `virt` riscv64 userspace path is known
  to work end-to-end.  If that emulator path regresses, pause BPI-F3 hardware
  bringup and repair the emulator path first.

Problem:

- The generic riscv64 port can start on QEMU `virt`, but real hardware will
  surface board-specific boot, firmware, device-tree, timer, interrupt,
  console, storage, and reset behavior.
- Banana Pi BPI-F3 is a useful first real riscv64 board target because it is a
  SpacemiT K1-based development board with real SMP, onboard storage, serial
  console, Ethernet, PCIe, USB, and published vendor firmware/BSP sources.
- A board plan needs to be explicit about what is proved on physical hardware
  versus what remains QEMU-only.

Reference facts:

- Banana Pi documents BPI-F3 as an industrial RISC-V development board using
  the SpacemiT K1 8-core RISC-V SoC.
- The documented CPU is SpacemiT K1 X60 `RV64GCVB`, RVA22, with RVV 1.0.
- Board resources include 2/4/8/16 GB LPDDR4, 8/16/32/128 GB eMMC options,
  microSD, dual GbE, USB 3.0 host ports, USB-C OTG, PCIe/M.2, HDMI, GPIO, and
  multiple UARTs.
- The vendor publishes Bianbu OpenSBI, U-Boot, and Linux 6.1 BSP source links,
  plus a BPI-F3 schematic and K1 documentation references.

Relationship to generic riscv64 port:

- This exploration depends on `033-riscv64-initial-port.txt` for the generic
  riscv64 architecture skeleton, trap handling, VM, native syscall ABI, and
  initial QEMU smoke path.
- This exploration must not begin until
  `034-riscv64-emulator-userspace-parity.txt` is complete.  Generic riscv64
  userspace, VM, proc/bootstrap launch, and the static musl/Linux-personality
  smoke must work end-to-end in QEMU `virt` before BPI-F3 hardware debugging
  starts.  A partial emulator boot, early harness success, or kernel-only
  milestone is not sufficient to unblock this exploration.
- The same gate remains active during the exploration: any BPI-F3 work that
  depends on a generic riscv64 facility must first be reproducible in QEMU
  `virt`; hardware is for board-specific proof, not for debugging generic
  emulator regressions.
- BPI-F3 support should be a board layer under the riscv64 port, not a forked
  architecture path.
- QEMU tests remain the fast development gate; BPI-F3 hardware tests become
  the proof that the riscv64 port is not merely `virt`-machine-specific.

Constraints:

- Keep GPL/vendor BSP code out of Bunix unless it remains userspace or tooling
  where the project explicitly allows it.  Use permissively licensed reference
  material directly only when license-compatible; otherwise treat BSPs as
  documentation for register layout, boot flow, and device-tree behavior.
- Prefer consuming firmware-provided device tree data over hardcoded board
  constants wherever possible.
- Do not require replacing board firmware for the first milestone.  Booting
  through vendor OpenSBI/U-Boot is acceptable while Bunix learns the platform.
- Physical-board tests must have clear operator steps and expected serial
  output because they cannot run inside normal CI.

- Refactored shared kernel runtime initialization:
  - Added `kernel_runtime_memory_init()`,
    `kernel_runtime_services_init()`, and
    `kernel_runtime_scheduler_init()` as shared generic lifecycle helpers.
  - x86_64 `kernel_main()` now uses those helpers after Multiboot2-specific
    boot parsing and SMP setup.
  - riscv64 keeps FDT, boot-package, and smoke sequencing in
    `riscv64_early_main()`, but now reaches the same generic VM/slab/buffer,
    IPC/name/VM-server, and scheduler initialization helpers.
  - This is not a single fully generic `kernel_main()` yet because x86_64 and
    riscv64 still enter through different firmware protocols, but the generic
    runtime lifecycle is no longer duplicated.
- Added the first native riscv64 console driver hook:
  - `console_init()` is now called from the riscv64 boot lifecycle.
  - `riscv64_console_init_from_fdt()` can bind the firmware-selected
    `/chosen/stdout-path` UART when the boot command line includes
    `riscv64-uart-console`.
  - The backend supports ns16550/8250-compatible UARTs, including QEMU
    `ns16550a` and `snps,dw-apb-uart`, and honors FDT `reg-shift` and
    `reg-io-width`.
  - The default riscv64 path remains SBI-backed so the normal emulator and
    BPI-F3 smoke gates keep their existing conservative behavior.
- Added supervisor MMIO identity mappings for early riscv64 device drivers:
  - The UART backend registers its MMIO page with the riscv64 VM layer.
  - New task address spaces inherit that mapping as supervisor-only, which
    keeps kernel syscall/trap-time console output working after a user task
    page table is active.
- Added `make test-boot-riscv64-uart-console`:
  - The target builds a separate riscv64 boot package whose command line
    includes `riscv64-uart-console`.
  - It boots QEMU `virt`, requires the ns16550 console-driver marker, then
    requires bootstrap completion, linux-riscv64 server poweroff, and machine
    poweroff.
  - This gives the current driver slice a concrete test the user can run
    before trying the equivalent stdout UART binding on physical hardware.
