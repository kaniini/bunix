Use HOT structures for IPv4 and IPv6 route tables.

Problem:

- NET0 currently does longest-prefix route lookup by scanning dynamic routes
  and a small static route array.  That is fine for loopback and early smoke
  tests, but it will not scale once VirtIO-net, external routing, localhost,
  IPv4, IPv6, and multiple interfaces grow real route state.
- Route lookup sits on hot UDP/TCP send paths and future packet-forwarding
  paths.  It should use a cache-aware data structure with predictable lookup
  behavior instead of list traversal.
- IPv4 and IPv6 have different key sizes and common prefix distributions, so
  the route-table abstraction should share policy while allowing separate
  optimized implementations.

Design direction:

- Replace NET0's route list lookup with Height Optimized Trie-style route
  tables for both IPv4 and IPv6.
- Preserve Linux-compatible longest-prefix match semantics: choose the most
  specific matching prefix, then apply metric/tie-break policy consistently.
- Keep route ownership in the net server.  The data structure should be a
  normal userspace-server component, allocated with Bunix userspace allocator
  primitives, and independent of kernel or architecture code.
- Keep the route-table API protocol-neutral enough that future routing
  protocols, static configuration, `/proc/net/route`, `/proc/net/ipv6_route`,
  and admin tools can all use the same backing tables.

Constraints:

- Do not change external NET0 RPC behavior until compatibility tests prove the
  new table preserves current route add/count/enumeration/lookup results.
- Avoid fixed-size route arrays.  The structure must grow dynamically and have
  explicit cleanup paths.
- IPv4 and IPv6 lookup should not share a stringly or byte-by-byte generic path
  when native integer key handling gives clearer and faster code.
- Keep enumeration deterministic for tests even if HOT internal node layout is
  optimized for lookup rather than insertion order.
- Tie-break behavior for duplicate prefixes must be documented and tested.

Tasks:

- [ ] Inventory current route add, route enumeration, route lookup,
      `net_addr_is_local`, packet-interface route state, and `/proc/net`
      consumers.
- [ ] Define a route-table API for insert, delete, lookup, count, enumerate,
      and snapshot/debug dumping, with separate IPv4 and IPv6 key types.
- [ ] Define longest-prefix and tie-break policy: prefix length first, then
      metric, then deterministic insertion or stable route id order.
- [ ] Implement an IPv4 HOT route table with dynamic node allocation, checked
      prefix masks, default-route support, duplicate-prefix handling, and
      deterministic enumeration.
- [ ] Implement an IPv6 HOT route table with 128-bit keys, prefix ranges from
      `/0` through `/128`, default-route support, duplicate-prefix handling,
      and deterministic enumeration.
- [ ] Refactor NET0 route add/count/at/lookup to use the route-table API while
      preserving the current RPC contract and static loopback route behavior.
- [ ] Add route deletion and replacement primitives if current admin-facing
      protocol lacks enough operations to keep the HOT tables maintainable.
- [ ] Add focused unit or userspace-server tests for IPv4 lookup, IPv6 lookup,
      default routes, host routes, overlapping prefixes, duplicate prefixes,
      metrics, enumeration stability, and deletion/replacement.
- [ ] Add NET0 integration tests proving UDP/TCP localhost behavior and
      packet-interface route selection still pass with HOT-backed lookup.
- [ ] Add route-table stress tests with many IPv4 and IPv6 prefixes to compare
      lookup behavior against a simple reference longest-prefix implementation.
- [ ] Document follow-up work for routing observability: `/proc/net/route`,
      `/proc/net/ipv6_route`, route cache counters, miss counters, and admin
      tooling.
