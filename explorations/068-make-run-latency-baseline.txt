Baseline and explain `make run` latency.

Parent exploration: `067-user-copy-vm-contract-refactor.txt`

Problem:

- Interactive `make run` has become noticeably slow.
- The current default run path boots the Alpine squashfs image with networking,
  OpenRC, many userspace servers, `log=info`, serial stdio, and QEMU external
  network plumbing.
- Recent logs show high-volume diagnostics such as `user-syscall:` dumps and
  repeated service chatter on the console path.  Console I/O can dominate boot
  time under `-serial stdio`, so we need measurements before changing policy.
- Full parallel tests also exposed a procfs self-pipe hang while `make test`
  was interrupted for this planning slice.  That belongs primarily to
  exploration 079, but the same instrumentation may help explain slow runs.

Goal:

- Produce a repeatable latency profile for `make run` from host command start
  to login prompt, network readiness, and first shell command completion.
- Identify whether the dominant cost is host build/staging work, GRUB/OVMF,
  kernel boot, server startup, VFS/rootfs setup, OpenRC, networking, logging,
  or QEMU/device configuration.

Tasks:

- [x] Split host-side time into rootfs/ESP rebuild, QEMU launch, firmware,

- Current explanation:
  - `make run` defaults to `RUN_ROOTFS_FLAVOR=alpine-squashfs` and runs the
    virtio-net Alpine profile.
  - Warm host-side work is not the problem: rootfs rebuild, ESP rebuild, ESP
    copy, and QEMU launch complete in about 55 ms when outputs are current.
  - Firmware plus kernel/server bringup reaches the Linux init handoff in
    about 6-7s on the measured runs.
  - The dominant delay is the guest userspace interval between
    `bootstrap: linux init exec` and `login:`, which remains about 31-34s even
    when the external QEMU network backend is removed.
  - The synthetic shell profile reaches login in about 8.6s, so the expensive
    behavior is specific to the default Alpine/OpenRC/login/networking run
    policy rather than the kernel boot path alone.
