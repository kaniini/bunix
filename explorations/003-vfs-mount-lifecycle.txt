Project plan: fix repeated mount lifecycle on the same mountpoint

Context carried forward:

- `root-mount-soak` now passes by using unique `/mnt/soak-*` mountpoints.
- During implementation, repeatedly mounting and unmounting tmpfs on the same
  `/mnt` mountpoint left `/mnt` busy after the first cycle.
- The parallel gate is green, so this can be investigated as a focused VFS and
  mount lifecycle bug rather than hidden inside the parallel harness work.

Milestone: make same-path mount/umount reuse reliable

Goal:

- Allow repeated mount and unmount operations on the same mountpoint when no
  live cwd/open route pins remain.
- Preserve busy unmount denial when a task genuinely pins a mount route.
- Replace the `root-mount-soak` unique-mountpoint workaround with same-path
  coverage once the lifecycle bug is fixed.

Plan:

1. Reproduce same-path mountpoint busy state [DONE]
   Add or run a focused root shell reproducer that mounts tmpfs on `/mnt`,
   writes and verifies a file, unmounts, checks hidden state, then repeats the
   same sequence on `/mnt` and captures the exact failing operation.

   Result:

   - Ran a focused root `test-command` reproducer.
   - Artifact: `build/failures/20260706T015254Z-8468/20260706T015435Z-8468`.
   - First cycle succeeded:
     `mount -t tmpfs tmpfs /mnt`, write/read `/mnt/repro.txt`, `umount /mnt`,
     and hidden-state check.
   - Second cycle failed at `mount -t tmpfs tmpfs /mnt` with
     `Resource busy`.
   - This confirms stale mountpoint busy state after a clean unmount, not a
     first-mount authorization or tmpfs creation issue.

2. Trace VFS route pin and mount refcounts [DONE]
   Inspect VFS mount, route pin, cwd, open-file, and unmount paths. Add
   temporary or trace-level diagnostics if needed to distinguish stale route
   pins from a genuine live reference.

   Result:

   - VFS unmount removed the mount node when explicit route pins were zero, but
     did not ask tmpfs to forget its internal root entry.
   - A later tmpfs mount request for the same path failed because
     `tmpfs.mount_root()` rejected any path still present in the tmpfs `roots`
     tree.
   - A separate busy-invariant check showed VFS unmount also ignored open VFS
     handles under the mountpoint, such as cwd handles.

3. Fix stale mount lifecycle state [DONE]
   Update the VFS and relevant filesystem/server code so a clean unmount fully
   releases route state for the mountpoint while still rejecting unmounts pinned
   by cwd or open handles.

   Result:

   - `tmpfs` now recycles a stale root on same-path mount only when no tmpfs
     open directory or file handle points at or under that root. Recycling
     prunes old files so the remounted tmpfs is fresh.
   - VFS unmount now rejects unmount when an open VFS handle path is at or under
     the mountpoint, preserving busy denial for cwd/open-file pins.
   - Focused same-path `/mnt` two-cycle mount/write/read/unmount/hide
     reproducer passed.
   - Focused pinned-cwd `/mnt` busy-unmount reproducer passed.

4. Convert `root-mount-soak` to same-path reuse [DONE]
   Change the root mount soak shard to repeatedly reuse `/mnt` directly, keeping
   the existing pinned-cwd busy denial coverage.

   Result:

   - `root-mount-soak` now mounts and unmounts `/mnt` directly for all repeated
     cycles.
   - The pinned-cwd busy denial check also uses `/mnt` directly.

5. Validate and complete
   Run narrow same-path mount coverage, `root-mount-soak`, and then full
   `make test`. Move this exploration to `explorations/completed/` when all
   tasks pass.
