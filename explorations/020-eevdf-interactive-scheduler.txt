Extend the scheduler into a high-quality EEVDF-inspired interactive scheduler.

Problem:

- Bunix has enough task/thread/preemption machinery to run real userspace, but
  the scheduler still needs a more deliberate policy for latency, fairness,
  SMP placement, and interactive behavior.
- The target is an EEVDF-style scheduler that favors interactivity and low
  latency over strict throughput fairness where those goals conflict.

Design direction:

- Use EEVDF concepts for virtual deadlines, eligible time, lag accounting, and
  proportional fairness.
- Use Con Kolivas' scheduler work, including BFS and MuQSS, as inspiration for
  simple global reasoning, strong desktop/interactivity latency, bounded
  wakeup behavior, and avoiding excessive scheduler complexity.
- Keep the scheduler portable and SMP-aware.  Per-CPU run queues may exist for
  scalability, but the policy should have a clear global fairness/latency
  model and avoid hiding starvation behind per-CPU artifacts.

Constraints:

- Preserve Bunix task/thread semantics: tasks are low-level objects, not Linux
  processes.
- Linux process/LWP behavior should map onto Bunix scheduling primitives
  without making Linux PID identity the scheduler's primary model.
- Interactive wakeups, IPC wakeups, console input, and server responsiveness
  should get good latency under load.
- Server threads and user processes should have tunable weights/classes without
  creating ambient privilege or ad hoc hardcoded special cases.
- The design must handle SMP placement, migration, idle balancing, affinity,
  preemption, timer ticks, and future real-time-ish classes cleanly.

Current scheduler inventory:

- `kernel/sched.c` owns Bunix task/thread objects, task handles, task VM-region
  metadata, per-task IPC affinity, per-CPU run queues, sleep queues, blocking,
  wakeups, direct IPC handoff, preemption, and teardown/reaping.
- Tasks are low-level Bunix objects with kernel-assigned `task->pid` values;
  those IDs are not Linux PIDs.  Threads are the scheduler entities.  Linux
  processes/LWPs are layered over Bunix tasks by the Linux/proc servers.
- The old policy was per-CPU FIFO: `run_queue` had only `head`, `tail`,
  `count`, and `idle`; enqueue appended, pick-next popped from the head, and a
  fixed 5-tick user-mode quantum triggered preemption when another local
  thread was runnable.
- SMP placement was conservative: new threads used explicit CPU placement,
  task IPC affinity, or least-loaded round-robin auto placement.  Wakeups
  returned a blocked thread to its recorded `thread->cpu_id`; cross-CPU wakeups
  sent a scheduler IPI only when the target run queue had been idle.
- Blocking/yield/sleep paths switch back to the per-CPU scheduler thread.
  Sleepers live in a global tick-ordered sleep list protected by `sleep_lock`.
  Direct IPC handoff can bypass the scheduler when sender and receiver are on
  the same CPU and the handoff is not nested.
- Timer preemption currently happens only from user-mode timer interrupts.
  Kernel/server code normally yields, blocks, or sleeps explicitly, so server
  responsiveness depends heavily on wakeup ordering and IPC handoff.

Initial EEVDF policy slice:

- The scheduler entity is the Bunix thread.  Each thread now carries a base
  weight, virtual runtime, eligible time, virtual deadline, deterministic
  enqueue sequence, execution-start tick, blocked-since tick, runtime ticks,
  wakeup count, migration count, and preemption count.
- Initial weights are uniform (`1024`) for all threads.  That preserves
  existing semantics while making weight/class policy an explicit next layer
  instead of baking Linux process identity into the scheduler.
- Runtime accounting charges running threads on timer ticks and voluntary
  exits from the CPU: yield, block, sleep, direct handoff, preemption, and
  thread exit.  Virtual runtime advances by `runtime * 1024 / weight`.
- Per-CPU run queues are now ordered by virtual deadline with deterministic
  tie-breaking on eligible time, enqueue sequence, and TID.  This replaces
  FIFO pick-next with an EEVDF-inspired earliest-deadline policy while keeping
  bounded O(n) insertion cost suitable for the current small-kernel stage.
- Interactive wakeups get bounded sleep credit.  A thread that was blocked can
  receive up to 20 ticks of virtual-runtime credit when it wakes, clamped to
  the target CPU's minimum virtual runtime.  This favors IPC, terminal input,
  and server replies without allowing unbounded negative lag.
- Direct IPC handoff is preserved.  When handoff cannot happen, the receiver is
  enqueued through the same wakeup path and receives the bounded interactive
  credit.

Instrumentation slice:

- The kernel now tracks aggregate and per-CPU scheduler counters for enqueues,
  switches, wakeups, preemptions, migrations, runtime ticks, runnable wait
  ticks, maximum runnable wait, wakeup-to-run ticks, maximum wakeup-to-run
  time, current run-queue load, and minimum virtual runtime.
- Scheduler accounting covers both ordinary run-queue dispatch and direct IPC
  handoff, so the QNX-style fast path remains visible instead of disappearing
  from scheduler observability.
- A native Bunix `sched_stats` syscall snapshots the kernel counters, and
  `procfs.server` exposes them as `/proc/sched` for shell tests and future
  Bunix/Linux-aware tooling.
- Verification for this slice:
  - `make`
  - `BUNIX_TEST_SET=rootfs-vfs-proc-dev make test-parallel`

Scheduler class metadata slice:

- Bunix tasks now have explicit scheduler metadata: class, priority, and future
  policy-right bits.  Threads inherit that metadata when they are created, but
  all classes still use the same weight in this slice to preserve behavior.
- The initial class set is `KERNEL`, `SERVER`, `USER`, `BATCH`, and `IDLE`.
  Kernel scheduler threads are `KERNEL`, boot module servers are marked
  `SERVER`, ordinary task creation defaults to `USER`, and forked Bunix tasks
  inherit their parent's scheduler metadata.
- Policy rights are named but not yet exposed as ambient authority.  Future
  policy-server work can grant class, priority, or weight adjustment rights as
  capabilities instead of letting arbitrary tasks self-promote.

SMP policy design:

- Keep per-CPU run queues for simple locking and good cache behavior, but make
  policy decisions visible enough that starvation cannot hide behind a local
  queue.  Each CPU tracks load, minimum virtual runtime, migrations, wakeups,
  and runnable latency through `/proc/sched`.
- New task placement remains least-loaded round-robin with explicit placement
  and IPC-affinity overrides.  IPC affinity is a latency hint for servers: a
  task that is created with a send capability to an affinity-bearing port can
  start on that port's CPU so the first request/reply path is local.
- Wakeups normally return a blocked thread to its last CPU.  This preserves
  locality and keeps server state warm.  If that CPU was idle, a scheduler IPI
  wakes it immediately.  Cross-CPU stealing should be added only after metrics
  show a real imbalance because eager migration can damage interactive latency.
- Migration is allowed when a thread is newly placed, explicitly moved by a
  future scheduler-policy server, or selected by a future idle-balancing pass.
  A migration must preserve the thread's virtual runtime/deadline state and
  must be counted in scheduler stats.
- CPU affinity should eventually be represented as a capability-scoped mask,
  not as a Linux PID property.  Linux CPU affinity can map onto that lower
  Bunix mask without making Linux process identity part of the scheduler core.
- Idle balancing should prefer pulling the earliest-deadline eligible thread
  from the busiest CPU only when the local CPU is idle and the source CPU has
  spare runnable work.  That gives idle CPUs useful work while avoiding
  continuous load-chasing.

Policy tradeoffs:

- Bunix deliberately favors interactive wakeup latency for IPC-heavy servers,
  terminal input, and short-lived shell commands over perfect throughput
  fairness.  The bounded wakeup credit is capped at 20 ticks and clamped to the
  CPU's minimum virtual runtime so sleepers cannot accumulate unbounded
  negative lag.
- Direct IPC handoff can bypass the run queue when sender and receiver are on
  the same CPU.  That trades strict fairness for lower RPC latency, but the
  path is still accounted as a wakeup and switch so it remains observable.
- Weights, classes, and priorities are present as metadata, but they do not yet
  alter runtime share.  That keeps this exploration behaviorally conservative
  until a capability-controlled policy surface exists.
- Per-CPU queues are an implementation choice, not the semantic boundary of
  fairness.  Future balancing must compare virtual runtime and runnable
  latency across CPUs rather than relying only on queue length.

Stress coverage slice:

- Added `/bin/schedstress` as a static musl Linux-personality test in the
  synthetic rootfs.  It forks multiple workers, drives repeated short sleeps,
  waits for every child, and verifies `/proc/sched` switch counters advance.
- The test is wired into the `tmpfs-basic-linux-tests` shard alongside the
  existing syscall race coverage.  This proves basic sleep/wakeup/fork/wait
  churn against the EEVDF accounting path, but it does not yet cover CPU-bound
  fairness, affinity, SMP stealing, or interactive latency under sustained CPU
  load.
- Verification for this slice:
  - `make build/modules/schedstress.user`
  - `BUNIX_TEST_SET=tmpfs-basic-linux-tests make test-parallel`
