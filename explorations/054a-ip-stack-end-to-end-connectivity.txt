Fix IP stack correctness and end-to-end userspace networking.

  either completed or explicitly postponed.
- This must stay before VFS-backed mmap/SIGBUS and later unrelated runtime
  work, because a default Alpine boot is not useful until the configured
  network is actually usable from userspace.
- Exploration 054 is about getting Alpine networking initialized by the
  normal OpenRC/ifupdown-ng service path.  This exploration is the next slice
  for proving that the IP stack itself is correct once the interface is up.
- Exploration 054 is currently postponed on an OpenRC `/run/openrc/deptree`
  runtime-state failure.  Until that is resumed, run this exploration against

- The default generated-service Alpine image has been restored to a minimal

Problem:

- The current default Alpine networking path can bring up interface
  configuration, but the IP stack still has serious correctness gaps after
  DHCP succeeds.
- This exploration is specifically for the post-DHCP "network is up but the
  stack is broken" state observed from a normal Alpine shell.
- The immediate follow-up to stock OpenRC networking is to examine why the IP
  stack is still utterly broken under normal Alpine userspace, not to add more
  boot-time networking policy.
- This exploration exists to examine why the IP stack is still utterly broken
  once link and DHCP appear to be working, instead of treating DHCP success as
  proof of usable networking.
- The IP stack appears substantially broken after link and DHCP success:
  ping TTLs are inconsistent, ping replies can be attributed to the wrong
  source address, DNS does not work, and `wget http://example.com` does not
  work.
- Current reported failures to reproduce and fix:
  - Ping TTLs are all over the place.
  - DNS does not work.
  - `wget http://example.com` does not work.
- Reported external ping behavior includes replies from `4.2.2.1` appearing
  as `0.0.0.0`, missing replies after the first packet, and later
  `recvfrom: Resource temporarily unavailable` errors.
- Reported higher-level failures include DNS resolution failing and
  `wget http://example.com` failing from Alpine userspace.
- ICMP replies have incorrect or unstable TTL/source metadata, which means
  receive-side packet attribution, route lookup, packet header handling, or
  ICMP socket delivery is not trustworthy enough yet.
- DNS resolution does not work from Alpine userspace.
- `wget http://example.com` does not work, so TCP, DNS, socket, routing, and
  Linux syscall compatibility are not proven end to end.

Required failure cases:

- `ping` must no longer report wildly inconsistent TTLs for the same remote
  endpoint.
- `ping -c 4 4.2.2.1` must produce four replies attributed to `4.2.2.1`,
  not `0.0.0.0`, and must not degrade into repeated `recvfrom: Resource
  temporarily unavailable` errors after the first reply.
- DNS resolution must work through Alpine's normal resolver path.
- `wget http://example.com` must work from the guest and fetch real HTTP
  content over the configured network.
- The DNS and HTTP checks must run after the interface has been initialized by
  the default Alpine boot networking path, not through a manually preconfigured
  test shortcut.  While exploration 054 is postponed, that default path is the
  generated Bunix networking service; once stock OpenRC networking is restored,
  these checks should be revalidated through the stock service path too.

Intended outcome:

- A fresh `make run` Alpine environment boots, logs in, and has usable
  userspace networking through the normal configured interface path.
- IPv4 ICMP replies report the real remote source address and sane TTL values.
- DNS works through Alpine's normal resolver configuration.
- `wget http://example.com` succeeds from the Bunix guest and retrieves HTTP
  content over TCP.
- The DNS and HTTP validations run from the guest's normal Alpine userspace
  environment after OpenRC networking has brought the interface up.
- The validation exercises the real userspace stack rather than a synthetic
  kernel-only shortcut.

Constraints:

- Do not write a new DHCP client; use the Alpine/BusyBox userspace already in
  the image.
- Do not special-case `example.com`, DNS, or ICMP in the test harness.
- Keep network policy in userspace and keep packet mechanics in the network
  servers.
- Preserve localhost IPv4/IPv6 behavior while fixing external networking.
- Treat incorrect TTL/source reporting as a stack correctness bug, not a
  display-only problem.
- Do not call the exploration complete just because DHCP succeeds.  It must
  also prove DNS and TCP/HTTP through normal Alpine userspace commands.

Investigation targets:

- Audit IPv4 receive dispatch from virtio-net through the network server into
  Linux socket wakeups.
- Verify that qemu/virtio-net receive packets are delivered to the IP stack and
  then to the correct userspace socket, with no stale packet metadata reused
  between receives.
- Verify ICMP echo reply parsing, source-address selection, TTL propagation,
  route lookup, and socket receive metadata.
- Verify UDP sockets used by DNS, including ephemeral port binding, reply
  demultiplexing, timeouts, and `recvfrom()`/`poll()` semantics.
- Verify resolver inputs: `/etc/resolv.conf`, DHCP-provided nameservers, and
  any Bunix-generated fallback configuration.
- Verify TCP connect/send/recv/close behavior against an external HTTP server,
  including nonblocking/error paths used by BusyBox `wget`.
- Check missing Linux networking APIs by running the failing BusyBox commands
  under `log=debug` strace and accounting for every unsupported or incorrect
  syscall result.
- Trace the packet path far enough to prove whether bad TTL/source values are
  introduced by virtio-net receive, IPv4 parsing, ICMP handling, route/socket
  metadata, or Linux `recvfrom()` translation.

Implementation Notes:

- Added `tools/guest-network-external-ping.sh` and
  `make test-boot-virtio-net-external-ping-strict`.  The guest script checks
  that default Alpine networking is up, runs `ping -c 4 -W 4 4.2.2.1`, rejects
  replies attributed to `0.0.0.0`, requires four replies from `4.2.2.1`, and
  requires all four TTL values to be in range and stable across replies.
- Added `tools/guest-network-dns-wget.sh` and
  `make test-boot-virtio-net-dns-wget`.  The guest script checks default
  Alpine networking, verifies `/etc/resolv.conf`, resolves `example.com` with
  BusyBox `nslookup`, then fetches `http://example.com` with BusyBox `wget`.
- Added `make test-boot-virtio-net-external-stack` as a convenience target for
  running the strict ping and DNS/wget checks sequentially.
- These targets currently cannot reach their guest assertions because Alpine
  previously failed before login in the OpenRC `/run/openrc/deptree` path.
  After restoring generated mode to a minimal OpenRC policy, the strict ping
  target reaches login and runs the guest assertions.
- Fixed the external IPv4 ping source/TTL path.  The net server now carries
  the IPv4 TTL from received ICMP packets, and the Linux server preserves
  socket type so raw IPv4 ICMP sockets receive a synthetic IPv4 header plus
  ICMP payload instead of interpreting the ICMP payload as an IP header.  This
  fixes the observed drifting TTL values.
- `make test-boot-virtio-net-external-ping-strict` now passes.  The target
  validates four replies from `4.2.2.1` and requires all four reply lines to
  report `ttl=255` under QEMU user networking.
- `tools/guest-network-external-ping.sh` now accumulates failure state and
  returns a failing final command when source, count, or TTL assertions fail,
  which avoids masking failures when the harness types command files into an
  interactive shell.
- The serial command-file path is still fragile for long scripts.  The strict
  ping target now uses a compact one-line command instead of
  `BUNIX_CMD_FILE`, because command-file injection can overrun the shell input
  path while long-running commands such as `ping` are still executing.  The
  next test-efficiency slice should replace typed command files with a real
  guest-side script transport or module-backed runner instead of further
  tuning serial pacing.
- A DNS/wget probe now reaches the guest and fails at the expected resolver
  layer.  `/etc/resolv.conf` is missing, BusyBox `nslookup example.com` falls
  back to `127.0.0.1` and reports `write to '127.0.0.1': Bad file descriptor`,
  and `wget http://example.com` fails with `bad address 'example.com'`.
  Artifact: `build/failures/20260711T105731Z-20972/20260711T110331Z-20972`.
  The DNS/wget guest script also masks failures today and needs the same
  final-status hardening as the strict ping test.
