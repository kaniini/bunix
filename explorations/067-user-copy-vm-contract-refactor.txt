Refactor user-copy and VM authority contracts after identity-map removal.

Parent exploration: `043-user-address-space-identity-map-escape.txt`

Problem:

- Removing the user-accessible low identity map made user-copy behavior depend
  on architecture-specific "is this really user memory?" checks.
- The generic VM layer, Linux server, native servers, and arch page-table code
  still share a fairly implicit contract for translating, copying, protecting,
  cloning, and unmapping user memory.
- That contract needs to be explicit before more server-side mmap, SIGBUS,
  dynamic-linker, and cross-architecture work lands.

Why this is a refactor:

- The immediate security fix should remain small and auditable.
- Follow-up work can improve API shape, diagnostics, and tests without
  changing the intended isolation policy.

Plan:

- Document the generic user-memory contract: which APIs accept task IDs,
  which require explicit user mappings, and which may touch kernel mappings.
- Introduce named helper families for user translation, kernel translation,
  user copy, mapping clone, protection changes, and unmap operations.
- Add trace/debug hooks for failed user-copy paths that report task, address,
  length, and permission reason without dumping user data.
- Audit all Linux server, VFS, procfs, tmpfs, user server, and test program
  call sites for the new contract.
- Add cross-architecture regression coverage for low-address faults, unmapped
  pointers, read-only pages, copied mappings, and denied kernel-only mappings.

Progress:

- Added `docs/user-memory-contract.md` to define the current contract between
  architecture page-table translation, generic VM helpers, kernel task
  authority, Linux syscall frontends, and userspace task-memory clients.
- Added generic `vm_translate_user_read()`, `vm_translate_user_write()`,
  `vm_translate_kernel_read()`, and `vm_translate_kernel_write()` wrappers.
  Generic VM code now uses the explicit user wrappers instead of direct
  architecture user-translation calls.
- Added debug-level VM failure diagnostics for user read, user write, and user
  protection operations.  Logs include operation, reason, current task, target
  VM space, address, and length without dumping user data.
- Audited task-memory wrapper users and kernel `vm_*_user*` call sites.  The
  direct userspace clients are proc, the shared syscall wrappers, allocator
  self-growth, and the proc exec-stack host unit test; Linux, VFS, procfs,
  tmpfs, and user do not directly issue task-memory mapping calls.
- Added `usermemtest`, a static Linux test program installed into the
  synthetic x86_64 rootfs and the riscv64 smoke rootfs.  It verifies low
  unmapped pointers, kernel-only pointer denial, read-only user-buffer denial,
  and valid `fstat()` user copies.  On x86_64 it also verifies private
  mapping copy behavior through `fork()`.  The riscv64 Linux frontend does not
  yet implement Linux `clone`, so riscv64 runs the user-copy/protection subset
  and treats Linux fork-copy coverage as an x86_64 gate for now.
- Riscv64 `arch_user_copy_from()` and `arch_user_copy_to()` now use the
  generic VM user-copy helpers instead of raw SUM memory copies, so bad user
  pointers fail with `EFAULT` instead of trapping in the kernel.
- The riscv64 bootstrap now treats nonzero child exit status from
  `BUNIX_PROC_WAIT` as a failed spawn-wait, making runtime test markers
  trustworthy.
