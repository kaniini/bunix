Shorten the boot and service startup critical path.

Parent exploration: `068-make-run-latency-baseline.txt`

Problem:

- Even with logging fixed, `make run` may still spend too much time launching
  servers, mounting translators, waiting for names, configuring networking,
  starting OpenRC, and spawning login.
- Several boot paths still use broad waits, fixed sleeps, or serialized
  readiness checks that were reasonable during bringup but are expensive for
  everyday interactive runs.
- Exploration 069 found that some visible log output is accidentally pacing
  early boot: broad raw-log filtering, or demoting module launch/start lines,
  can make default `make run-profile` hang before `bootstrap: fs ready` while
  `log=debug` still succeeds.
- During exploration 069 validation, `make test-boot` failed on both the dirty
  tree and a clean `HEAD` archive because the synthetic boot-spawn phase
  reported `proc: exec open failed` before `/bin/first` markers appeared,
  even though `/bin/first` exists in the generated squashfs image.
- The default run target starts a feature-rich environment.  That is valuable,
  but routine development needs boot phases to be explicit and measurable.

Goal:

- Reduce time from kernel entry to usable login shell without weakening the
  multi-server architecture or hiding readiness failures.
- Replace accidental console/serial pacing with explicit ordering,
  readiness, and wakeup behavior.

Tasks:

- [ ] Reproduce the `log=info` hang from exploration 069 with broad raw-log
      filtering, capture the last successful server readiness point, and prove
      whether `log=debug` succeeds only because of extra serial pacing.
- [ ] Identify the exact launch-order or wakeup dependency that makes module
      start logs observable in control flow; replace it with explicit
      readiness RPCs, notifications, or scheduler wakeups.
- [ ] Debug the synthetic `run_boot_spawns()` failure where proc cannot open
      `/bin/first` after `bootstrap: fs ready`; determine whether this is VFS
      namespace resolution, proc exec registry state, subject authority, or a
      race with Linux init/login startup.
- [ ] Use exploration 068 phase data to identify the slowest guest-side boot
      phases and the server responsible for each delay.
- [ ] Replace fixed sleeps and polling loops on the interactive path with
      explicit readiness RPCs or event notifications where practical.
- [ ] Audit bootstrap server launch ordering for unnecessary serialization;
      allow independent servers to start in parallel when capability and mount
      dependencies permit.
- [ ] Separate critical-path services needed for login from deferred services
      that can start after the prompt.
- [ ] Profile OpenRC and networking startup under the default Alpine run path,
      and decide which services must block login.
- [ ] Add boot timing markers to dmesg or `/proc` so future regressions can be
      diagnosed from inside the guest.
- [ ] Re-run `make run-profile`, shell smoke, networking smoke, and full test
      shards affected by boot ordering.

Baseline from exploration 068:

- Repeated warm default Alpine networking profile
  `build/run-profile/20260712T232619Z-17014` reached the Linux init handoff at
  7071 ms, but the login prompt did not appear until 40126 ms.
- No-external-QEMU-network-device Alpine profile
  `build/run-profile/20260712T232817Z-17851` reached the Linux init handoff at
  6068 ms and login at 37119 ms.
- Synthetic shell profile `build/run-profile/20260712T232231Z-14525` reached
  login at 8639 ms.
- First read: the largest cost is the default Alpine/OpenRC/login path after
  Linux init starts.  External virtio-net networking adds several seconds, but
  removing it does not collapse the roughly 31-34s login delay.  This
  exploration should focus on OpenRC service ordering, readiness waits, login
  spawn policy, and any blocking Linux/VFS calls made during that interval.
- Classified output profile `build/run-profile/20260712T233726Z-21452` shows
  the Linux-init-to-login interval emitted only 19 serial lines and 640 bytes:
  14 OpenRC lines, 1 VM grant line, 1 login line, and 3 blank lines.  The next
  investigation should therefore prioritize silent blocking work: OpenRC
  dependency evaluation, clock skew handling, VFS metadata/read behavior,
  Linux syscall compatibility waits, login spawn timing, and readiness polling.
- Safe log-reduction profile from exploration 069
  `build/run-profile/20260713T000744Z-7395` reached `bootstrap: fs ready` at
  5360 ms and login at 34410 ms with only 12433 bytes of serial output.  This
  confirms that the remaining slowdown is not primarily console bandwidth.
- Failed log-filtering profile from exploration 069
  `build/run-profile/20260713T000028Z-3709` stopped after `console: driver
  online` and before `bootstrap: fs ready`; a comparable `log=debug` run
  `build/run-profile/20260713T000319Z-5138` completed.  Treat this as a
  high-priority boot ordering race before further console demotion work.
- Clean-HEAD synthetic boot check from exploration 069:
  - `make test-boot` fails at `tools/test-boot-markers-squashfs.txt` because
    `proc: exec /bin/first` never appears.
  - The guest log shows `bootstrap: fs ready`, then `bootstrap: linux init
    exec`, then `proc: exec /sbin/init`, then `proc: exec open failed` /
    `proc: spawn failed`.
  - `unsquashfs -ll build/modules/disk0.sqfs` confirms `/bin/first`,
    `/etc/execs`, and `/etc/spawns` are present in the image, so the first
    suspect is runtime VFS/proc authority or namespace state rather than
    rootfs construction.

Priority order:

1. Fix the boot ordering/log-pacing dependency so quiet logs cannot change
   whether servers reach filesystem readiness.
2. Fix the clean-HEAD synthetic proc-spawn failure so the boot marker gate is
   trustworthy again.
3. Instrument the silent Linux-init-to-login interval from inside the guest,
   including OpenRC service timing, networking startup, VFS calls, and Linux
   syscall wait sites.
4. Defer nonessential services until after login where the full Alpine
   environment permits it, and keep full networking/OpenRC validation in a
   named faithful profile.
