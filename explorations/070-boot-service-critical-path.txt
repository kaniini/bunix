Shorten the boot and service startup critical path.

Parent exploration: `068-make-run-latency-baseline.txt`

Problem:

- Even with logging fixed, `make run` may still spend too much time launching
  servers, mounting translators, waiting for names, configuring networking,
  starting OpenRC, and spawning login.
- Several boot paths still use broad waits, fixed sleeps, or serialized
  readiness checks that were reasonable during bringup but are expensive for
  everyday interactive runs.
- Exploration 069 found that some visible log output is accidentally pacing
  early boot: broad raw-log filtering, or demoting module launch/start lines,
  can make default `make run-profile` hang before `bootstrap: fs ready` while
  `log=debug` still succeeds.
- During exploration 069 validation, `make test-boot` failed on both the dirty
  tree and a clean `HEAD` archive because the synthetic boot-spawn phase
  reported `proc: exec open failed` before `/bin/first` markers appeared,
  even though `/bin/first` exists in the generated squashfs image.
- The default run target starts a feature-rich environment.  That is valuable,
  but routine development needs boot phases to be explicit and measurable.

Goal:

- Reduce time from kernel entry to usable login shell without weakening the
  multi-server architecture or hiding readiness failures.
- Replace accidental console/serial pacing with explicit ordering,
  readiness, and wakeup behavior.

Tasks:

- [x] Reproduce the `log=info` hang from exploration 069 with broad raw-log
      filtering, capture the last successful server readiness point, and prove
      whether `log=debug` succeeds only because of extra serial pacing.
- [x] Identify the exact launch-order or wakeup dependency that makes module
      start logs observable in control flow; replace it with explicit
      readiness RPCs, notifications, or scheduler wakeups.
- [x] Debug the synthetic `run_boot_spawns()` failure where proc cannot open
      `/bin/first` after `bootstrap: fs ready`; determine whether this is VFS

      race with Linux init/login startup.
- [x] Use exploration 068 phase data to identify the slowest guest-side boot
      phases and the server responsible for each delay.
- [ ] Replace fixed sleeps and polling loops on the interactive path with

- [ ] Audit bootstrap server launch ordering for unnecessary serialization;

- [x] Separate critical-path services needed for login from deferred services

- [x] Profile OpenRC and networking startup under the default Alpine run path,

Progress:

- The clean-HEAD synthetic boot-spawn failure was a subject-authority problem
  in the proc server, not a missing squashfs entry.  Native proc-spawned tasks
  were being used as the VFS subject for executable opens without first being
  registered with the user server, so permission checks could deny readable
  files such as `/bin/first`.
- The proc server now registers non-Linux spawned tasks with the user server
  immediately after task creation and unregisters them during process reset.
  Linux-personality tasks still follow the Linux server registration path so
  PID/user-session state is not duplicated.
- `make test-boot` passes with the synthetic `/bin/first`,
  `/bin/longpath-first`, alloctest, phdrstress, and Linux init markers
  restored.
- Default generated Alpine networking profiles showed the main remaining
  interactive delay was OpenRC sysinit:
  - `build/run-profile/20260713T011353Z-16473` measured
    `openrc-sysinit-start` at 3447 ms and `openrc-sysinit-end` at 41506 ms;
    boot/default were much smaller.
  - A generated `/var/cache/rc/deptree` avoids the expensive OpenRC dependency
    regeneration path, while a Bunix-specific sysinit avoids generic
    `/proc`/`/run` mount probing that Bunix already handles through servers.
  - The generated-networking flavor now runs one `bunix-openrc-bringup`
    sysinit action, starts the generated `networking` service, advances the
    OpenRC softlevel to default, and then lets init spawn login.  This removes
    BusyBox init's one-second gaps between separate sysinit/wait entries on
    the default `make run-alpine-net` path.
  - The stock-networking flavor keeps the fuller OpenRC boot path for the
    future stock-service exploration.
- `tools/run-profile.sh` now uses short polling with real deadlines, so
  marker attribution no longer adds up to one second per waited marker.
- Latest generated Alpine networking profile
  `build/run-profile/20260713T013155Z-2809`:
  - QEMU start at 498 ms, login prompt at 9405 ms: about 8.9 seconds from
    QEMU start to usable login prompt.
  - `bootstrap: fs ready` at 3463 ms, Linux init handoff at 3568 ms,
    sysinit end at 6433 ms, boot end at 8480 ms, default end at 8687 ms.
  - The generated networking service exited with status 0 and the post-login
    `/proc/net/config` probe found `default_ipv4 1`.
- Direct `make run-alpine-net` was still slow/misleading because invoking the
  target directly used the global default `ROOTFS_FLAVOR=squashfs`; the
  `BLOCK_IMAGE := ...` assignment is expanded during Makefile parsing, so the
  target embedded the synthetic rootfs unless `make run` wrapped it with an
  explicit `ROOTFS_FLAVOR=alpine-squashfs`.
- `run-alpine-net` now recurses as the Alpine networking wrapper, while
  `run-alpine-net-qemu` is the internal QEMU launch target.  This keeps
  `make run` configurable through `RUN_ROOTFS_FLAVOR` and makes direct
  `make run-alpine-net` match its name.
- Warm validation profile `build/run-profile/20260713T015610Z-21128` reached
  `login:` at 8393 ms from profile start and 7697 ms from QEMU start.  A warm
  direct `timeout 10s make run-alpine-net` smoke saw `login:` before the
  cutoff, including Make wrapper overhead.
- `/proc/boot_timing` now exposes structured boot milestones from inside the
  guest.  The kernel stores bounded monotonic boot events and exposes a native
  record/read syscall; procfs renders those events as `index ms delta_ms name`
  and accepts writes to `/proc/boot_timing` as additional markers.  Bootstrap
  records critical-path milestones including names readiness, input readiness,
  network readiness, rootfs selection, netcfg readiness, filesystem namespace
  readiness, filesystem readiness, and Linux init handoff.  Generated Alpine
  OpenRC wrappers append sysinit/boot/default start/end markers to the same
  file, so the Linux-init-to-login interval is diagnosable without serial log
  scraping.
- The early boot log-pacing dependency was a timer-polled IPC readiness path:
  the names server only noticed private claim ports while polling, and the
  proc, user, and Linux servers slept for 1 ms whenever neither their
  management port nor service port had an immediately queued message.  Extra
  serial logging could accidentally create enough scheduling/timer pacing to
  hide that missing wakeup.
- Names claim registration now sends the registration to the private claim
  port, nudges the names service with an explicit `BUNIX_NAMES_CLAIM_READY`
  message, and waits on a reply port.  The names server blocks on its service
  port and polls claim ports only when it has a reason to do so.
- The kernel now exposes `ipc_recv_any` as a native syscall on x86_64 and
  riscv64.  It blocks one thread on a bounded set of receive ports, reports
  which port delivered the message, and guards against stale multi-port waits
  being overwritten while the receiver wakes.  The proc, user, and Linux
  servers use it to receive from their management and service ports without
  fixed sleeps.
- Latest generated Alpine networking profile
  `build/run-profile/20260713T032407Z-15933` reached `login:` at 3651 ms from
  profile start, `root-shell` at 3755 ms, completed the network probe at
  3961 ms, and exited QEMU at 4019 ms.  The profile stayed inside the log
  volume gates with 183 serial lines and 9103 bytes.
- The historical quiet-log failure is preserved by exploration 069 profiles:
  `build/run-profile/20260713T000028Z-3709` stopped after `console: driver
  online`, before `bootstrap: fs ready`, while `log=debug` completed.  After
  replacing the polling dependency, a stronger `log=error` generated Alpine
  networking smoke reaches login, restarts OpenRC networking, validates
  `/proc/net/config`, and pings the host gateway plus external IPv4
  addresses.  That proves normal info-level server output is no longer
  required for early boot progress.
