Implement a SquashFS filesystem server, migrate the boot root filesystem to it,
and retire the legacy rootfs server once the migration is complete.

Problem:

- Bunix currently carries a custom boot-module rootfs path that was useful for
  early bringup, but it is not a realistic filesystem format and keeps too much
  bespoke root filesystem behavior alive.
- Alpine/OpenRC and future package-manager workflows are better served by a
  standard readonly image format with mature host tooling, compression support,
  directories, symlinks, permissions, timestamps, and stable inode metadata.
- SquashFS is a good fit for an initial readonly system root: compact,
  deterministic, widely supported by Linux tooling, and simpler operationally
  than making the early root image format grow into a real filesystem.

Constraints:

- Implement SquashFS as a userspace filesystem server mounted through VFS.
  Do not move SquashFS parsing or root filesystem policy into the kernel.
- The server should consume a block-device capability, not assume a GRUB module
  or a specific storage backend.  It must work first with the current block
  server and later with VirtIO block without changing the VFS-facing contract.
- Keep the first SquashFS server readonly.  SquashFS write support is a host
  image-generation concern, not a runtime filesystem mutation target.
- Preserve OCAP direction: mounting a SquashFS image requires an explicit block
  capability, and the mounted translator should only expose VFS operations
  authorized by that mount.
- Migrate root selection cleanly.  During transition, both the old rootfs
  server and the SquashFS server may coexist, but the old rootfs server should
  be deleted once tests prove the SquashFS-backed root covers the boot,
  BusyBox, login, OpenRC, and shell use cases.
- Host-side image generation may use standard tools such as `mksquashfs`, but
  runtime code must remain compatible with the project licensing constraints.

Inventory:

- Boot modules:
  - `boot/grub.cfg` and `boot/grub-standalone.cfg` always include
    `rootfs.server` and `disk0.img`.
  - `kernel/server.c` treats `disk0` as the boot data module for the `block`
    server. This is the current block handoff path SquashFS should reuse first.
  - `kernel/server.c` still knows `rootfs` as a boot server name, so retiring
    the legacy server requires removing it from boot server discovery and all
    GRUB/module construction rules.
- Build rules and host images:
  - `Makefile` defines `ROOTFS_MODULE`, `ROOTFS_TOOL`, `SYNTHETIC_BLOCK_IMAGE`,
    `ALPINE_BLOCK_IMAGE`, `ROOTFS_FLAVOR`, `BLOCK_IMAGE`, and
    `ROOTFS_FLAVOR_STAMP` around the custom `mkrootfs` image format.
  - `tools/mkrootfs.c` writes the custom `RFS0` image with fixed rootfs entry
    records and data payloads. Synthetic root generation passes explicit file,
    directory, and symlink entries to this tool.
  - `tools/build-alpine-rootfs.sh` stages Alpine with `apk --root ... --initdb`
    and then calls `mkrootfs --tree "$root"` to produce
    `build/modules/alpine-disk0.img`.
  - `tools/test-rootfs-tool.sh` and `tools/test-alpine-rootfs.sh` directly
    parse the custom rootfs format and must either be replaced with SquashFS
    tests or kept only while the legacy image path exists.
- Runtime servers and protocols:
  - `user/rootfs/main.c` is the active readonly translator for the custom
    image. It reads through `BUNIX_SERVICE_BLOCK`, parses `ROOTFS_MAGIC`, and
    registers `BUNIX_SERVICE_ROOTFS`.
  - `user/include/bunix/syscall.h` defines `BUNIX_PROTO_ROOTFS`,
    `BUNIX_SERVICE_ROOTFS`, `BUNIX_ROOTFS_MOUNT_PATH`, and
    `BUNIX_VFS_DEV_ROOTFS`; those protocol and device IDs become obsolete when
    the legacy server is deleted.
  - `user/vfs/main.c` still contains rootfs entry structures, lookup, stat,
    readdir, symlink, and read helpers backed by a `root_block`, but current
    VFS startup only initializes that tree and does not populate it. Treat this
    as legacy/dead rootfs behavior to delete after the mounted-translator path
    is fully SquashFS-backed.
  - `user/procfs/main.c` reports `rootfs` in filesystems/mount output and maps
    `BUNIX_VFS_DEV_ROOTFS` to `"rootfs"`.
- Bootstrap mount flow:
  - `user/bootstrap/main.c` launches `vfs`, `procfs`, `tmpfs`, `devfs`,
    `sysfs`, and `utmpfs`; mounts tmpfs roots at `/.upper`, `/tmp`, `/run`,
    `/var/tmp`; then launches `rootfs` and mounts it at `/.lower`.
  - Bootstrap then launches `unionfs`, sets lower to `/.lower`, upper to
    `/.upper`, and mounts unionfs at `/`.
  - Ext2 test modes bypass part of this by mounting ext2 at `/` or
    `/mnt/ext2`, but normal synthetic and Alpine boots use the
    `rootfs + tmpfs + unionfs` stack.
- Tests and docs:
  - Boot marker files look for `rootfs: online`, `rootfs: module`, and
    `kernel: launching module server rootfs`.
  - Shell tests named `rootfs-vfs-proc-dev` and content tests in
    `tmpfs-extended`, `union-root-user`, and `tmpfs-basic-linux-tests` assume
    the lower readonly filesystem reports as rootfs and contains the synthetic
    fixture files.
  - `tools/test-parallel.sh` only accepts `synthetic` and `alpine` rootfs
    flavors today. SquashFS needs either a new flavor or a compatible mapping
    before parallel tests can cover it.
  - `README.md` documents the custom rootfs builder, `disk0` image,
    `rootfs.server`, rootfs lower layer, and rootfs-specific tests.

SquashFS server/VFS mount contract:

- Service identity:
  - Add a new `BUNIX_PROTO_SQUASHFS`/`BUNIX_SERVICE_SQUASHFS` FourCC and a
    VFS device ID such as `BUNIX_VFS_DEV_SQUASHFS`.
  - The SquashFS server registers itself through names like other filesystem
    translators and receives ordinary `BUNIX_PROTO_VFS` requests once mounted.
- Block capability handoff:
  - First slice may mirror ext2/rootfs by resolving `BUNIX_SERVICE_BLOCK` with
    send rights during bootstrap bringup, because that is how the current block
    server exposes the boot `disk0` image.
  - The target contract is explicit: bootstrap or a mount manager sends the
    SquashFS server a mount/setup RPC carrying a send-capable block service
    capability and the mount path/options.  The server must not assume GRUB
    modules or global block service resolution after this transition point.
  - SquashFS only needs `BUNIX_BLOCK_GET_INFO`, `BUNIX_BLOCK_READ`, and
    `BUNIX_BLOCK_READ_BUFFER`; it must never issue write or flush operations.
- Mount behavior:
  - Bootstrap mounts SquashFS at `/.lower` during transition so unionfs can
    continue to mount `/` with tmpfs at `/.upper`.
  - Later migration may mount SquashFS directly as `/` only if the writable
    overlay story changes; the initial contract keeps the existing
    readonly-lower/writable-upper behavior.
  - Mount failure must be explicit in logs: unsupported compression, unsupported
    feature flags, malformed superblock, invalid table bounds, and missing root
    directory should produce distinct diagnostics.
- Readonly VFS surface:
  - Supported operations: `OPEN_BUFFER`, `STAT_PATH_META_BUFFER`,
    `ACCESS_BUFFER`, `READLINK_BUFFER`, `STAT_META`, `READ_FILE_BUFFER`,
    `READDIR_BUFFER`, and `CLOSE`.
  - Mutating operations must return `BUNIX_VFS_ERR_ACCESS` or an equivalent
    readonly error mapping: create, write, truncate, unlink, chmod, chown,
    mkdir, rmdir, mknod, symlink, rename, and link.
  - Open handles should follow the existing translator shape for the first
    implementation, while noting that `039-vfs-global-open-handle-authority.txt`
    will later replace ambient integer handle authority.
- Path and lookup semantics:
  - VFS remains responsible for global path normalization and symlink
    traversal across mount boundaries.  SquashFS receives already-rooted
    translator-relative paths and should resolve only within its own image.
  - `READLINK_BUFFER` returns the symlink payload without following it.
  - Directory lookup and readdir must preserve image names exactly, reject
    invalid/corrupt duplicate entries when detected, and expose deterministic
    ordering from the image.
- Root inode and stat metadata:
  - The mounted root directory uses the SquashFS root inode from the superblock,
    not a synthetic inode, so `/`, `/.lower`, and procfs mountinfo can report a
    stable inode identity.
  - `stat`/`statx` metadata maps SquashFS inode type, mode, uid/gid table
    entries, size, link count, mtime, block count, device numbers, and inode
    number into the existing `BUNIX_VFS_STAT_BYTES` layout.
  - Unsupported or out-of-range uid/gid IDs should fail mount validation or map
    to a documented overflow value, not silently become root.
- Error translation:
  - Missing path: `BUNIX_VFS_ERR_NOENT`.
  - Non-directory in path traversal or readdir on file: `BUNIX_VFS_ERR_NOTDIR`.
  - Directory opened as regular file read: `BUNIX_VFS_ERR_ISDIR` or existing
    read failure convention, then normalize through Linux errno mapping.
  - Unsupported/corrupt image discovered after mount should fail the request
    with `BUNIX_VFS_ERR_INVAL` and log a bounded diagnostic.

Initial SquashFS format scope:

- Superblock:
  - Accept SquashFS magic `0x73717368` and major version 4 only.
  - Treat images as little-endian.  Big-endian images, older format revisions,
    and filesystem sizes larger than the block device are unsupported.
  - Validate `bytes_used`, `block_size`, `block_log`, inode count, fragment
    count, ID count, and all table start offsets before reading any table.
  - Initial block-size limit is 128 KiB to match the current block server
    buffered-read cap and `mksquashfs` default.  Larger block sizes should fail
    with an explicit unsupported-block-size diagnostic until the reader/cache
    is generalized.
- Tables:
  - Required: inode table, directory table, and ID table.
  - Optional/unsupported for first boot slice: export table and xattr table.
    Generate images without export/xattr requirements and reject images that
    require xattr interpretation.
  - Fragment table support can start as "no fragments allowed" by generating
    images with `-no-fragments`; the parser should still validate fragment
    table offsets/counts and reject nonzero fragment use until fragment reads
    are implemented.
  - Metadata table blocks must be bounds-checked by on-disk start and stored
    size, and cached by table block address once decoded.
- Compression:
  - Runtime supports only blocks stored uncompressed in the first slice.
  - Mount validation rejects any metadata, directory, inode, data, or fragment
    block that is marked compressed.  The superblock compression field may
    still reflect the host tool's selected compressor, so actual per-block
    compressed/uncompressed flags are the authority for whether a decompressor
    is needed.
- Inodes and file types:
  - Required inode types for synthetic root: basic directory, basic regular
    file, and basic symlink.
  - Required inode types for Alpine/OpenRC root: directory, regular file,
    symlink, and character/block device metadata if present in the staged tree.
    FIFOs/sockets can be rejected until tests require them.
  - Extended inode variants should be decoded where the field layout is needed
    for size/link/timestamp correctness; otherwise reject with an explicit
    unsupported-inode diagnostic rather than silently mis-statting.
- Directory semantics:
  - Support directory lookup by exact byte name and readdir from the directory
    table.
  - Reject corrupt names that exceed Bunix `PATH_MAX`, names containing `/`, or
    entries whose decoded inode references are outside validated metadata
    ranges.
  - Preserve image order for readdir so tests can rely on deterministic output
    from deterministic image generation.
- Regular files and symlinks:
  - Support regular reads from uncompressed data blocks and empty/sparse files.
    Reads crossing block boundaries must be handled before this server becomes
    a root candidate.
  - Fragment-backed tail reads are deferred if images are generated with
    `-no-fragments`; otherwise they are required before enabling general
    package-built SquashFS images.
  - Symlink payloads are returned exactly as stored for VFS-level resolution.

Compression policy:

- First runtime target: no decompressor in Bunix.
- Host generation uses `mksquashfs -no-compression -no-fragments -no-exports
  -no-xattrs -b 128K -repro-time 0` or the closest supported equivalent.
- Compressed SquashFS images are explicitly unsupported until a permissively
  licensed decompressor is imported and isolated behind a small decompression
  dispatch interface.
- Gzip is the preferred first compressed extension if needed, because the
  format is ubiquitous and small permissive inflate implementations exist, but
  compressed support is not on the critical path for the first bootable
  SquashFS root.
