Refactor the TTY layer around a userspace device-driver framework.

Problem:

- Console and TTY behavior has grown from early bringup code into kernel and
  server paths that know too much about the serial console.
- The next design step is to split early console from real TTY/device handling,
  move the serial console into a userspace driver, and keep privileged hardware
  operations behind narrow kernel RPCs.

Constraints:

- Preserve a minimal early console for boot diagnostics before userspace
  drivers are online.
- The serial console driver should run in userspace and communicate with the
  kernel only through explicit privileged hardware-operation RPCs.
- Do not expose arbitrary I/O privilege to drivers.  Capabilities should
  attenuate access to specific port/MMIO/interrupt resources.
- Keep the design portable: x86 port I/O is one backend, but the framework
  should be able to support ARM MMIO UARTs later.
- Existing Linux TTY behavior must keep working through the refactor:
  canonical input, echo, password suppression, Ctrl-C, backspace, poll
  readiness, session/login behavior, and `/dev/console`/`/dev/tty` routing.

Tasks:

- [x] Inventory current console, devfs, Linux TTY, login, interrupt, and input
      paths and identify which pieces are early-console-only versus real TTY.
- [x] Define a device-driver RPC protocol for privileged hardware operations:
      port I/O, MMIO read/write, interrupt binding, interrupt ack/mask, and
      timer-safe notification.
- [x] Add kernel capability objects for hardware resources with attenuation to
      specific port ranges, MMIO ranges, IRQ vectors, and operation masks.
- [x] Create an initial userspace driver framework with driver registration,
      resource discovery, interrupt/event delivery, and lifecycle diagnostics.
- [ ] Move the serial console implementation into a userspace driver using the
      hardware RPC protocol instead of direct kernel console ownership.
- [ ] Keep early console output available until the serial driver is online,
      then hand off normal console writes/input to the userspace path.
- [ ] Refactor TTY line discipline into a clear server-side layer that can be
      shared by serial console, future virtual terminals, and PTYs.
- [ ] Route Linux `/dev/console`, `/dev/tty`, stdin/stdout/stderr, `poll(2)`,
      and terminal ioctls through the TTY layer rather than serial-specific
      shortcuts.
- [ ] Add tests for boot console handoff, login/password echo behavior,
      backspace, Ctrl-C, poll readiness, foreground process interruption, and
      shell logout/login respawn.
- [ ] Document how the framework will map to future ARM UART drivers without
      relying on x86 port I/O assumptions.

Inventory:

- `kernel/console.c` still owns both early diagnostics and the real serial
  backend.  It initializes COM1 at `0x3f8`, writes bytes through direct x86
  port I/O, polls serial input, queues pending input, provides the dmesg ring,
  and has legacy kernel-side line input helpers.
- `user/console/main.c` is currently a console service facade.  Other servers
  send `CONS` RPCs to it, but it still calls early-console syscalls, so the
  actual serial hardware remains kernel-owned.
- `kernel/server.c` starts a `console-input` kernel thread that polls
  `console_try_read_char()` and forwards bytes to the Linux server as
  `BUNIX_LINUX_TTY_INPUT`.
- `user/linux/main.c` owns the real Linux TTY behavior: canonical input,
  echo, password echo suppression through termios, erase handling, Ctrl-C,
  foreground process interruption, CPR response, `poll(2)` readiness, terminal
  ioctls, and `/dev/tty`/`/dev/ttyS0`/`/dev/console` file descriptors.
- `user/devfs/main.c`, `user/sysfs/main.c`, and `user/utmpfs/main.c` expose
  serial-console-shaped device names and metadata, while `user/login/main.c`
  depends on Linux termios to disable password echo.
- Early-console-only pieces are boot diagnostics, the dmesg ring, log
  verbosity, and fallback panic/bringup writes.  Real TTY pieces are serial
  device output/input, line discipline, readiness notification, foreground
  process signaling, termios state, and `/dev/console`/`/dev/tty` routing.

Progress:

- Added `HWR0` as the hardware-driver protocol namespace, with resource kinds
  for port I/O, MMIO, and IRQ ranges; operation masks for read, write,
  interrupt bind, interrupt ack, and interrupt mask; and message IDs for the
  planned driver RPC surface.
- Added native byte port I/O syscalls that require a hardware resource handle.
  The x86 backend validates handle type, operation mask, range bounds, and
  overflow before touching an I/O port.
- Added hardware resources as task handle objects.  They can be inherited and
  attenuated through existing handle rights, while the resource object carries
  the hardware-specific operation mask and range.
- Granted `consoled` a COM1-only port resource for `0x3f8..0x3ff` with read
  and write authority.  Normal console writes now use that capability-backed
  userspace serial path, with early-console writes kept as fallback and for the
  dmesg ring.
- Added a small userspace driver helper for service registration, static
  resource discovery, and lifecycle diagnostics.  Consoled now declares its
  COM1 resource through that framework, registers through the shared helper,
  and emits a driver lifecycle message; interrupt/event delivery is represented
  in the `HWR0` protocol and remains to be wired to real IRQ dispatch.
