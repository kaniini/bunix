Use HOT structures for IPv4 and IPv6 route tables.

Problem:

- NET0 currently does longest-prefix route lookup by scanning dynamic routes
  and a small static route array.  That is fine for loopback and early smoke
  tests, but it will not scale once VirtIO-net, external routing, localhost,
  IPv4, IPv6, and multiple interfaces grow real route state.
- Route lookup sits on hot UDP/TCP send paths and future packet-forwarding
  paths.  It should use a cache-aware data structure with predictable lookup
  behavior instead of list traversal.
- IPv4 and IPv6 have different key sizes and common prefix distributions, so
  the route-table abstraction should share policy while allowing separate
  optimized implementations.

Design direction:

- Replace NET0's route list lookup with Height Optimized Trie-style route
  tables for both IPv4 and IPv6.
- Preserve Linux-compatible longest-prefix match semantics: choose the most
  specific matching prefix, then apply metric/tie-break policy consistently.
- Keep route ownership in the net server.  The data structure should be a
  normal userspace-server component, allocated with Bunix userspace allocator
  primitives, and independent of kernel or architecture code.
- Keep the route-table API protocol-neutral enough that future routing
  protocols, static configuration, `/proc/net/route`, `/proc/net/ipv6_route`,
  and admin tools can all use the same backing tables.

Constraints:

- Do not change external NET0 RPC behavior until compatibility tests prove the
  new table preserves current route add/count/enumeration/lookup results.
- Avoid fixed-size route arrays.  The structure must grow dynamically and have
  explicit cleanup paths.
- IPv4 and IPv6 lookup should not share a stringly or byte-by-byte generic path
  when native integer key handling gives clearer and faster code.
- Keep enumeration deterministic for tests even if HOT internal node layout is
  optimized for lookup rather than insertion order.
- Tie-break behavior for duplicate prefixes must be documented and tested.

Inventory:

- NET0 stores static loopback routes in the fixed `routes[]` array and dynamic
  routes in the `dynamic_routes` singly linked list.
- `BUNIX_NET_ROUTE_ADD` rejects invalid prefixes and unknown interfaces,
  deduplicates only exact route records, then inserts new dynamic routes at the
  list head.
- `BUNIX_NET_ROUTE_DELETE` deletes only exact dynamic route records. Static
  loopback routes cannot be deleted.
- `BUNIX_NET_ROUTE_COUNT` returns dynamic plus static route count.
- `BUNIX_NET_ROUTE_AT` enumerates dynamic routes first in current list order
  (reverse insertion order), then static loopback routes in array order.
- `net_route_lookup()` scans dynamic routes, then static routes, and currently
  chooses the highest prefix length only. Metrics do not participate in lookup
  tie-breaking yet, so equal-prefix ties keep the first matching route in
  dynamic enumeration order and dynamic routes beat equal-prefix static routes.
- IPv4 route matching uses `hi == 0`, `prefix_hi == 0`, and a 32-bit mask over
  `lo`. IPv6 route matching supports `/0` through `/128` over the `hi:lo`
  128-bit key.
- `net_addr_is_local()` is independent of routes and scans dynamic addresses
  followed by static addresses.
- Packet-interface send paths using `net_route_lookup()` are external IPv4 and
  IPv6 UDP, TCP SYN, and ICMP/ICMPv6.
- `default_route()` relies on `net_route_lookup()` and then requires
  `prefix_len == 0`.
- `/proc/net/route` consumes `BUNIX_NET_ROUTE_COUNT` and `BUNIX_NET_ROUTE_AT`,
  filters to IPv4, and prints route records in NET0 enumeration order.
- Current route tests live in the bootstrap packet-interface self-test and
  NETCFG installs DHCP fallback IPv4 and deterministic static IPv6 routes
  through the public route RPCs.

Route Table API:

- Store route entries in a common `struct net_route_entry` payload containing
  the public route fields plus a monotonic `order` used only for deterministic
  tie-breaking and enumeration.
- Provide separate IPv4 and IPv6 table wrappers:
  `route_table4_insert/delete/lookup/count/at` and
  `route_table6_insert/delete/lookup/count/at`.
- Keep a common route collection facade for RPCs:
  `route_collection_add/delete/count/at/lookup`.
- `lookup(family, hi, lo)` returns the best route entry or null without
  allocating.  It must handle IPv4 default routes, IPv6 `/0`, and IPv6 `/128`
  host routes.
- `at(index)` preserves deterministic RPC enumeration independent of internal
  trie node layout by enumerating entries sorted by `order`.
- Static loopback routes are represented as route entries in the same
  collection with reserved low `order` values and a `static_route` flag that
  prevents deletion.
- Insert deduplicates exact route records to preserve current
  `BUNIX_NET_ROUTE_ADD` behavior.  Exact duplicates are success no-ops.
- Delete matches exact route records and refuses static entries, preserving the
  current inability to delete built-in loopback routes.

Tie-Break Policy:

- Longest prefix wins.
- For equal prefix lengths, lower metric wins.
- For equal prefix length and metric, newer dynamic routes win over older
  dynamic routes and over static routes.  This keeps current dynamic-route
  behavior where head-inserted equal-prefix dynamic routes win, while making
  the rule explicit.
- Enumeration remains stable by route `order`: dynamic routes appear in reverse
  insertion order followed by static routes, matching the current RPC behavior.
  The lookup tie-break uses a separate descending dynamic order so lookup can
  prefer newer equal-prefix routes without making trie layout observable.
