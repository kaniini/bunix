Fix console TTY input and foreground-control authority.

Problem:

- The Linux server accepts `BUNIX_LINUX_TTY_INPUT` before resolving the message
  sender to a Linux process or checking that the sender is the console driver.
- The console TTY is a single global object.  Foreground process group state is
  also global, and `TIOCSPGRP` updates it without checking session membership
  or controlling-terminal ownership.
- A caller with a send-capable Linux service capability can potentially inject
  keystrokes, synthesize control characters, deliver signals through the TTY
  path, or redirect foreground terminal state.

Threat model:

- Primary attacker: an unprivileged Linux process if it can obtain or influence
  a Linux service send capability through native/hybrid Bunix RPC paths or
  service-resolution bugs.
- Secondary attacker: a compromised userspace server that can resolve or was
  granted the Linux service port.
- Impact candidates: cross-session input injection, password/command injection
  at the login shell, unauthorized SIGINT delivery to foreground jobs,
  foreground process-group confusion, and denial of service of the interactive
  console.

Evidence:

- `user/console/main.c` polls serial input and forwards bytes to the Linux
  server as `BUNIX_LINUX_TTY_INPUT`.
- `user/linux/main.c` handles `BUNIX_LINUX_TTY_INPUT` before
  `linux_process_for(&message)`, so the message is not bound to a Linux
  process or to a console-driver authority.
- `linux_tty_input_event()` appends bytes to the global `console_tty` queue,
  echoes according to global termios state, and handles interrupt characters by
  calling `linux_tty_interrupt_foreground()`.
- `linux_set_foreground_pgid()` assigns the global `foreground_pgid` without
  validating controlling-terminal ownership, process-group membership, or
  session relationship.

Root cause:

- Console input is represented as an ordinary Linux server message instead of
  an authority held only by the console driver or a TTY server.
- TTY state is global and not modeled as a session-owned device object.

Fix plan:

- Introduce an explicit console/TTY input authority delegated only to the
  console driver or future TTY server.
- Reject `BUNIX_LINUX_TTY_INPUT` unless the caller proves that authority.
- Move toward per-TTY objects with owner/session/foreground state instead of a
  global `console_tty` and global `foreground_pgid`.
- Enforce Linux controlling-terminal rules for `TIOCSPGRP`, `TIOCGPGRP`,
  reads, writes, and generated terminal signals.
- Add regression tests that prove ordinary Linux processes and unrelated
  servers cannot inject input or change another session's foreground group.

Notes:

- Added `BUNIX_LINUX_GRANT_TTY_INPUT_TASK` as a Linux management RPC.  Bootstrap
  resolves the kernel-launched `consoled` task and grants that task as the only
  authorized source of Linux TTY input after linux-server registers.
- Linux-server now tracks authorized TTY input senders and rejects both
  `BUNIX_LINUX_TTY_INPUT` and `BUNIX_LINUX_TTY_INPUT_BUFFER` with `-EPERM`
  unless the authenticated message sender has that authority.  Fire-and-forget
  console input still works, while tests can use reply ports to observe
  denial.
- Added `test-tty-input-authority`, which launches `mgmt-test` with only
  names/console authority, resolves linux-server directly, and proves a
  non-console sender cannot inject a TTY input buffer.
- Focused verification passed: `make test-tty-input-authority` and
  `BUNIX_TEST_SET=interactive-tty make test-shell-parallel`.
- Moved foreground process-group state from the global `foreground_pgid` to
  `struct linux_tty`, with a session owner and foreground pgid on the console
  TTY object.  Linux registration, session attach, and `setsid` now update the
  console TTY through that object rather than ambient global state.
- `TIOCGPGRP`, `TIOCSPGRP`, and terminal-generated SIGINT now consult the
  console TTY object.  `TIOCSPGRP` requires the caller to own the controlling
  TTY session and the requested process group to exist in the same Linux
  session; bogus or cross-session pgids are rejected instead of overwriting
  foreground state.
- The console TTY controlling-session key is the Linux session ID (`sid`), not
  the Bunix user-server login session ID.  A failed attempt to test
  `setsid sleep` surviving foreground Ctrl-C exposed that using the login
  session ID let `setsid` children still look like controlling-TTY members.
  That was corrected, but a stable cross-session generated-SIGINT regression
  still needs to be designed before the regression-test task is complete.
- Added `/bin/ttytest` and a `tty-foreground` shell shard.  The test exercises
  real musl `ioctl(1, TIOCGPGRP/TIOCSPGRP, ...)` calls, verifies a bogus
  foreground pgid is denied, and verifies setting/restoring the caller's own
  process group still works.
- Focused verification for the foreground slice passed: `make
  test-tty-input-authority`,
  `BUNIX_TEST_SET=interactive-tty,login-smoke,relogin-session make
  test-shell-parallel`, and `BUNIX_TEST_SET=tty-foreground make
  test-shell-parallel`.
- After correcting the TTY session key, focused verification passed again with
  `BUNIX_TEST_SET=interactive-tty make test-shell-parallel` and
  `BUNIX_TEST_SET=tty-foreground make test-shell-parallel`.
- Added `/bin/ttysigtest` and a `tty-sigint-session` shell shard.  The test
  forks a foreground reader and a separate `setsid()` sleeper, sends terminal
  Ctrl-C from the harness, and verifies the generated SIGINT reaches only the
  foreground reader while the separate session survives.
- Focused verification for the generated-signal regression passed:
  `BUNIX_TEST_SET=tty-sigint-session make test-shell-parallel`.
