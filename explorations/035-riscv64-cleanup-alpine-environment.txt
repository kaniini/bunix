Clean up the riscv64 port and get to an Alpine environment on riscv64.

Status:

- Active.
- This exploration pauses `058-riscv64-bpi-f3-board-port.txt`.  BPI-F3
  hardware proof remains important, but the generic riscv64 port should be
  cleaned up first so board work does not accumulate around temporary
  emulator-only scaffolding.
- The next explorations after this one split future refactors into focused
  work:
  - `036-shared-linux-personality-riscv64.txt`
  - `037-riscv64-boot-test-split.txt`
  - `038-riscv64-server-rootfs-packaging.txt`
  - `039-x86-riscv64-dryness-refactor.txt`

Problem:

- The riscv64 path can boot QEMU `virt`, run scheduler-owned native userspace,
  launch a tiny Linux-personality smoke server, and run a static riscv64 musl
  hello binary.
- The riscv64 Linux server is still a small architecture-specific stub rather
  than the real Linux personality used by x86_64.
- The riscv64 boot package and runtime path still contain bringup-era special
  cases that are acceptable for smoke tests but not for an Alpine environment.
- Alpine-on-riscv64 should be proved in QEMU before returning to BPI-F3
  hardware so board debugging does not hide generic architecture defects.

Target outcome:

- `make run` remains x86_64-focused unless intentionally changed elsewhere.
- A dedicated riscv64 QEMU target boots an Alpine-style riscv64 rootfs or
  rootfs image far enough to start the normal Bunix bootstrap/proc/Linux
  stack and enter an Alpine userspace init path.
- The riscv64 Linux personality is no longer a separate toy server for the
  static hello smoke; it either reuses the shared Linux server implementation
  or has a thin arch syscall frontend over shared Linux semantics.
- The riscv64 test suite has narrow gates that prove static musl smoke,
  dynamic-loader/openat/fstat behavior, and Alpine init-shell progress before
  the full image path is considered usable.

Constraints:

- Do not regress the existing x86_64 Alpine/busybox/OpenRC flow.
- Do not require BPI-F3 hardware for this exploration.
- Keep riscv64 BPI-F3 artifact generation working while this exploration is
  active, but do not close BPI-F3 hardware tasks without real board logs.
- Prefer shared Linux-personality code over growing `user/riscv64-linux`.
- Keep architecture-specific syscall ABI details explicit and table-driven
  where possible.

Progress notes:

- Created as the active exploration when `058-riscv64-bpi-f3-board-port.txt`

- Completed the first audit of bringup-only riscv64 pieces:
  - `user/riscv64-linux/main.c` is a toy Linux personality server.  It only
    recognizes riscv64 Linux `write`, `exit`, `exit_group`, and
    `set_tid_address`, emits `linux-riscv64-server:*` markers, and powers off
    directly after `exit_group`.  This must be replaced by the shared Linux
    server semantics before Alpine can work.
  - `arch/riscv64/user.c` forwards every nonnegative Linux syscall as one
    special `BUNIX_LINUX_RISCV64_SYSCALL` message and expects action words
    telling the kernel to perform console write or task exit side effects.
    The x86_64 path already uses a broad Linux RPC protocol with explicit
    message types for process registration, exec, VFS, TTY, signals, wait,
    and fault handling.  The riscv64 path needs a thin arch syscall frontend
    over that shared protocol, not a parallel action protocol.
  - `user/include/bunix/syscall.h` currently exposes the riscv64 bridge as
    public protocol IDs: `BUNIX_LINUX_RISCV64_SYSCALL` plus
    `BUNIX_LINUX_RISCV64_ACTION_*`.  Those should disappear once the riscv64
    Linux syscall frontend uses the shared Linux server protocol.
  - The Makefile has separate hand-written riscv64 module rules for
    `abi-smoke.user`, `bootstrap.server`, `linux.server`, and static musl
    hello.  Some target-triple/linker differences are legitimate, but the
    duplication currently bakes in `user/riscv64-linux` and a smoke-specific
    boot package instead of a reusable riscv64 server/rootfs packaging path.
  - `user/riscv64-bootstrap/main.c` is a smoke-only bootstrap.  It logs
    `bootstrap-riscv64:*`, launches `abi-smoke.user`, launches `linux`, then
    launches `/bin/musl-hello`.  Alpine needs the normal bootstrap/proc/Linux
    stack and an init/rootfs path, not this fixed hello sequence.
  - `arch/riscv64/early.c` still owns many QEMU smoke assertions directly in
    the boot path: boot layout markers, platform discovery markers, scheduler
    self-test, timer wait, context-switch self-test, syscall self-test, user
    mode self-test, boot package parsing, module validation, ELF loader
    self-test, user-copy self-test, and only then module launch.  These are
    valuable gates, but the Alpine path needs a cleaner split between
    always-on boot initialization and test-only diagnostics.
  - `tools/build-riscv64-bootpkg.sh` and the `RISCV64_BOOTPKG` Makefile rules
    are an initrd-style smoke carrier containing `bootstrap`, `abi-smoke.user`,
    `linux`, and `/bin/musl-hello`.  The next Alpine slice needs a riscv64
    rootfs/module carrier that can deliver the normal server graph and Alpine
    filesystem content.
  - `docs/riscv64-port.md` still documents dynamic linking, Alpine rootfs
    parity, BusyBox, console input, dmesg reads, and real Linux signal/syscall
    parity as later slices.  These map directly onto this exploration's
    remaining work and should be updated as each item becomes real.
- Legitimate architecture boundaries to keep explicit:
  - riscv64 FDT/SBI/OpenSBI boot entry is not the same as x86_64 EFI
    Multiboot2, so boot protocol parsing should stay arch-specific even when
    generic runtime initialization is shared.
  - riscv64 syscall register mapping, syscall numbers, user trap return,
    signal frame ABI, TLS/thread-pointer setup, and ELF machine ID are real
    arch boundaries.
  - riscv64 user CRT0 and static/dynamic musl toolchain wiring are target
    specific, but should feed shared server/rootfs packaging once the Linux
    personality is portable.
