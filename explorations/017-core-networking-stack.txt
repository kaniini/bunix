Implement the core networking framework: localhost, IPv4, IPv6, TCP, and UDP.

Problem:

- Linux userspace will increasingly expect sockets to behave like a real
  network stack, not just minimal compatibility stubs.
- Before real NIC drivers, Bunix needs an internal networking framework with
  loopback, IPv4, IPv6, UDP, and TCP semantics.

Constraints:

- Keep the network stack as userspace servers where possible, with kernel
  involvement limited to IPC, capabilities, event delivery, and low-level
  driver resources.
- Preserve OCAP direction: sockets, interfaces, routes, and packet flows should
  be represented by explicit capabilities rather than ambient global authority.
- Linux PID/process identity may differ from Bunix task identity, so socket
  ownership, credentials, and visibility must be mediated through the Linux
  personality and user/proc servers.
- Start with localhost and deterministic tests before external networking.

- [x] Define the networking server split: socket API/personality adapter,

Inventory:

- Linux socket compatibility is currently a narrow AF_UNIX stream shim in
  `user/linux/main.c`.  `linux_socket()` only accepts `AF_UNIX`,
  `SOCK_STREAM`, and protocol 0, then creates `LINUX_FD_SOCKET` entries whose
  handle is either `LINUX_SOCKET_UNIX_STREAM` or the special
  `LINUX_SOCKET_UTMPD` endpoint.
- `linux_connect()` only understands sockaddr_un and accepts the hard-coded
  `/run/utmps/.utmpd-socket` path.  Any non-AF_UNIX family returns
  `EAFNOSUPPORT`; any other AF_UNIX path returns `ENOENT`.
- `linux_sendto()` and `linux_recvfrom()` are utmps-specific.  `sendto()` only
  accepts one-byte utmps rewind/getent commands on a connected utmps socket,
  and `recvfrom()` delegates directly to the utmpfs response helper.
- `poll`/`ppoll` are forwarded to the Linux server one fd at a time by
  `arch/x86_64/user.c`, and the Linux server marks every `LINUX_FD_SOCKET` as
  immediately readable and writable.  There is no socket wait queue, readiness
  edge, timeout integration, or per-socket blocking state yet.
- The syscall bridge names and forwards only `socket`, `connect`, `sendto`,
  and `recvfrom` for socket operations.  The Bunix Linux RPC enum does not yet
  contain `bind`, `listen`, `accept`, `accept4`, `sendmsg`, `recvmsg`,
  `getsockname`, `getpeername`, `setsockopt`, `getsockopt`, or `shutdown`, so
  those currently fall through the generic unknown-syscall path in the arch
  Linux syscall layer.
- There are no `AF_INET`, `AF_INET6`, `SOCK_DGRAM`, `SOCK_NONBLOCK` readiness
  semantics beyond fd creation flags, address structures, route/interface
  records, port allocation state, packet buffers, or TCP/UDP protocol objects.

Server split:

- Keep the Linux personality as the socket ABI adapter.  It should translate
  Linux syscall arguments, fd flags, sockaddr layouts, credentials, and
  wait-status behavior, then hold a capability to a native socket object rather
  than implementing protocol mechanics inline.
- Add a native `net.server` that owns protocol state for loopback IPv4/IPv6,
  UDP endpoints, TCP control blocks, listen queues, route lookup, interface
  registration, packet counters, and socket readiness notifications.  The
  server should expose a Bunix protocol such as `NET0` rather than Linux
  syscall numbers.
- Keep loopback as the first built-in interface inside `net.server`: bring up
  `lo` with `127.0.0.1/8` and `::1/128`, route those prefixes locally, and
  deliver packets through in-memory queues without requiring a packet driver.
- Future NIC drivers, including VirtIO-net, should register packet interfaces
  with `net.server` through a separate driver-facing protocol.  Drivers should
  exchange packet buffers and link-state events, not Linux sockets or process
  IDs.
- Privileged network configuration belongs behind explicit capabilities issued
  by bootstrap or a future network-management server.  Ordinary Linux
  processes should receive only socket capabilities mediated by the Linux
  server and their process credentials.

Capability model:

- Socket capability: authority to operate one endpoint.  It carries domain,
  type, protocol, nonblocking/close-on-exec state, owner credentials, local and
  peer addresses, readiness state, receive/send queues, and shutdown state.
- Listener capability: authority to accept queued TCP streams for one bound
  local address/port.  It should be derived from a socket capability by
  `listen` and should not grant route or interface configuration authority.
- Interface capability: authority for one network interface such as `lo` or a
  future VirtIO-net device.  Driver-facing interface caps allow RX/TX packet
  queue attachment and link-state updates; management caps allow address and
  MTU changes.
- Route capability: authority to install, replace, or remove routes for a
  constrained address family/prefix/table.  The initial route caps should cover
  only the loopback prefixes and should be held by `net.server` during bringup.
- Packet-queue capability: authority for a driver or loopback path to submit
  and receive packet buffers on one interface.  Packet queues should be bounded
  and observable, with drop counters rather than unbounded allocation.
- Privileged port/config capability: authority to bind restricted ports or
  configure addresses/routes.  This keeps Linux uid checks as policy in the
  Linux/user servers while `net.server` enforces that the caller actually holds
  the needed authority.

Progress:

- Added a first userspace `net.server`, registered as the `NET0` service.  It
  owns the built-in loopback interface, brings `lo` up with `127.0.0.1` and
  `::1`, exposes interface count/info/stats RPCs, implements a raw in-memory
  loopback packet send/receive queue, and is included in the normal EFI, ISO,
  and VirtIO block test boot payloads.
- Boot marker coverage now asserts `net: loopback lo ipv4=127.0.0.1 ipv6=::1`,
  and `bootstrap: net loopback ok`, proving that the core networking server
  starts before Linux init, that localhost interface lifecycle no longer
  depends on a future physical NIC, and that a raw IPv4 packet can be copied
  through the loopback queue.  Packet delivery is intentionally raw and
  server-local for this slice; UDP/TCP endpoint delivery will consume this path
  in the next slices.
- Added native UDP endpoint state inside `net.server`: dynamic UDP socket
  objects, wildcard bind, loopback connect, ephemeral port allocation,
  localhost IPv4/IPv6 datagram delivery, per-datagram checksums, receive queues,
  poll readiness, and close cleanup.  Bootstrap now self-tests IPv4 and IPv6
  UDP delivery through `NET0` and boot markers assert `bootstrap: net udp ok`.
- Added native TCP localhost stream state inside `net.server`: dynamic stream
  sockets, wildcard bind, listen backlog, connect, accept queues, paired
  streams, ordered read/write queues, poll readiness, half-close EOF, close
  cleanup, and refused-connect reset/error behavior.  Bootstrap now self-tests
  IPv4 and IPv6 TCP streams in both directions and boot markers assert
  `bootstrap: net tcp ok`.
- Started Linux socket compatibility over `NET0`: the x86_64 Linux syscall
  bridge now forwards `bind`, `listen`, `accept`, `accept4`, and `shutdown` in
  addition to the previous socket/connect/send/recv path, and `linux.server`
  can create AF_INET/AF_INET6 TCP and UDP sockets backed by native NET0 socket
  IDs.  TCP sockets now participate in Linux `read`, `write`, `poll`, and
  `close`; UDP supports bind/connect/send/recv/close.  Remaining Linux socket
  work includes sockaddr return paths, `sendto` with an explicit destination,
  `getsockname`, `getpeername`, socket options, and dedicated userland tests.
- Added `nettest`, a static musl Linux-personality boot test that opens
  AF_INET sockets through the Linux server and verifies localhost UDP
  explicit-destination `sendto`/recv plus TCP
  bind/listen/connect/poll/accept/read/write.  The x86_64 bridge now grants
  `DUP` on Linux read/write/send/recv shared buffers where the Linux server
  forwards them to NET0.  Remaining Linux socket work includes sockaddr return
  paths, `getsockname`, `getpeername`, socket options, IPv6 userland coverage,
  and teardown/permission edge cases.
