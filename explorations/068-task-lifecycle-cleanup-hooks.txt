Add generic task lifecycle cleanup hooks for subsystem-owned per-task state.

Status:

- Pending.
- Schedule after the x86_64/riscv64 DRYness refactor exploration.

Problem:

- Some architecture frontend and kernel subsystem state is keyed by Bunix task
  ID but currently cleaned up only from the success path that subsystem owns.
- The riscv64 Linux frontend state added during exploration 039 is released on
  Linux `exit_group`, which is sufficient for the current smoke path but not a
  general task-lifecycle contract.
- As more per-task state moves out of fixed tables and into dynamic subsystem
  structures, cleanup should be driven by task death rather than by each syscall
  frontend remembering every possible exit path.

Tasks:

- [ ] Audit dynamically allocated per-task state in kernel, architecture
      frontend, scheduler, VM, IPC, and Linux-personality paths.
- [ ] Define a small task lifecycle notification API with clear lock-ordering
      rules.
- [ ] Wire task-death cleanup for riscv64 Linux frontend registration state.
- [ ] Move any similar ad hoc per-task cleanup to the shared lifecycle path.
- [ ] Add tests that kill or abandon Linux tasks without taking the ordinary
      `exit_group` path and verify stale per-task state is released.
- [ ] Document when subsystem state should live on `struct task` directly and
      when a lifecycle hook is the better boundary.
