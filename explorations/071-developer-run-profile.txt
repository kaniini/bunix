Create fast and faithful developer run profiles.

Parent exploration: `068-make-run-latency-baseline.txt`

Problem:

- `make run` currently mixes multiple goals: faithful Alpine boot, networking
  by default, OpenRC, login, broad server coverage, and an interactive shell.
- For everyday kernel/server development, a fast boot-to-shell profile is
  often more useful than the most complete environment.
- For integration debugging, the faithful profile must remain available and
  should be explicitly named so slow behavior is expected.
- Exploration 068 and 069 show that the current default profile reaches Linux
  init in roughly 5-7s but often does not reach login until roughly 34-40s.
  That makes normal `make run` feel slow even when the kernel/server bringup
  is already mostly done.

Goal:

- Provide clear run targets for fast iteration and full-system validation,
  with documented differences and stable timing expectations.
- Make the default interactive command serve the common developer loop unless
  the user explicitly requests the full Alpine/OpenRC/networking path.

Tasks:

- [x] Define the intended semantics for `make run`, `make run-fast`,
      `make run-alpine`, and networking-enabled run targets.
- [x] Decide whether default `make run` should prioritize fast interactive
      development or faithful Alpine/OpenRC/networking bringup, using the
      measured 34-40s login path as the deciding data point.
- [x] Add a fast profile that boots to an interactive shell with minimal
      blocking services, quiet logs, and clean poweroff behavior.
- [x] Keep a faithful Alpine networking profile that starts OpenRC/networking
      and validates DHCP, DNS, ping, and login behavior.
- [x] Ensure both profiles use the same kernel/server artifacts so performance
      differences come from policy, not hidden build skew.
- [x] Document the run targets in `README.md` with expected boot milestones and
      when to use each target.
- [x] Add lightweight smoke tests for the fast and faithful profiles.

Baseline from exploration 068:

- Warm `make run` host rebuild and staging overhead is negligible when outputs
  are current, but the current default Alpine networking profile reaches login
  at about 37-40s.
- The synthetic shell profile reaches login at about 8.6s.
- The default target should be reviewed with this data: if `make run` is meant
  for fast kernel/server iteration, it should use the fast profile; if it is
  meant to prove the full Alpine/OpenRC/networking environment, that slower
  behavior should move behind a name like `make run-alpine` or
  `make run-alpine-net`.

Inputs from exploration 069:

- Default output can be reduced to about 12.4 KB without fixing the login
  delay, so target design should not pretend quiet logs alone make `make run`
  fast.
- Any `make run-fast` profile must still use the same boot ordering semantics
  fixed in exploration 070; it must not rely on skipping enough services to
  hide a scheduler/readiness race.

Expected profile policy:

- `make run`: fast interactive development profile once 070 proves quiet
  launch ordering is reliable.
- `make run-alpine`: faithful Alpine/OpenRC login without external networking
  unless explicitly requested.
- `make run-alpine-net`: faithful Alpine/OpenRC/networking profile with DHCP,
  DNS, ping, and wget coverage.
- `make run-profile`: measurement harness that can target either fast or
  faithful policy and records comparable phase data.

Progress:

- `make run` now aliases to `make run-fast`, which boots the synthetic SquashFS
  profile through the normal `build/esp` EFI image without attaching a network
  device.
- `make run-alpine` boots the generated Alpine rootfs through `build/esp-alpine`
  without a network device.
- `make run-alpine-net` keeps the virtio-net Alpine profile with the real
  OpenRC networking service and QEMU user networking attached.
- `make run-profile` now defaults to the fast profile, while
  `make run-profile-fast`, `make run-profile-alpine`, and
  `make run-profile-alpine-net` select explicit policies.  The profile harness
  has a `RUN_PROFILE_NETWORK_CHECK` policy so synthetic and no-net Alpine
  profiles are not forced through a networking assertion.
- Added `test-run-fast`, `test-run-alpine`, and `test-run-alpine-net` smoke
  targets.  `test-run-alpine-net` reuses the existing OpenRC networking smoke
  that validates login, service status, DHCP/default IPv4 configuration, and
  external IPv4 pings.
- README documents the target split and current warm profile milestones.
- The high-level Alpine networking gates now observe the networking state
  produced by the boot runlevel.  They require `/run/openrc/started/networking`
  and `rc-service networking status`, but do not restart networking or rerun
  `udhcpc` by hand.
- Tightened the remaining external-network guest checks so the strict ping and
  DNS/wget gates also require `/run/openrc/started/networking` before passing.
  This keeps the faithful Alpine networking profile tied to the packaged
  OpenRC `/etc/init.d/networking` service instead of direct DHCP probes.
- The faithful Alpine networking profile now passes through the packaged
  Alpine OpenRC networking service.  `make test-boot-virtio-net-dns-wget`
  validates `/run/openrc/started/networking`, `rc-service networking status`,
  DHCP-provided resolver state, BusyBox `nslookup`, BusyBox `wget
  http://example.com`, and a non-empty fetched output file.  The fix was in
  the Linux/net compatibility path below the stock service: UDP receives now
  report source sockaddr data for musl resolver callers, and the net server has
  a minimal outbound IPv4 TCP client data path.

Needs:

- No remaining profile-policy work for the stock Alpine networking path.  The
  remaining OpenRC quality issues, such as the `bootmisc` cgroup/fork warning,
  belong to follow-up OpenRC/process-lifecycle explorations rather than a
  replacement networking script.
